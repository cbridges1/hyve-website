## Overview

Hyve validates workflows before execution to catch errors early and provide clear feedback. Validation checks YAML syntax, required fields, tool availability, secret presence, and more.

## Validation Phases

Hyve performs validation in multiple phases:

<Steps>
  <Step title="1. YAML Syntax">
    Checks that the workflow file is valid YAML

    ```
    [ERROR] Invalid YAML syntax in workflow 'deploy-app'
    [ERROR] Line 10: mapping values are not allowed here
    ```
  </Step>

  <Step title="2. Schema Validation">
    Verifies required fields and structure

    ```
    [ERROR] Workflow validation failed:
    - Missing required field: metadata.name
    - Missing required field: spec.jobs
    ```
  </Step>

  <Step title="3. Requirements Validation">
    Checks tools and secrets are available

    ```
    [INFO] Validating workflow requirements...
    [ERROR] Required tool 'kubectl' not found in PATH
    [ERROR] Required secret 'DOCKER_TOKEN' not found
    ```
  </Step>

  <Step title="4. Runtime Validation">
    Validates during execution (variable substitution, commands)

    ```
    [INFO] Executing workflow 'deploy-app'
    [ERROR] Command failed: kubectl apply -f manifests/
    ```
  </Step>
</Steps>

## YAML Syntax Validation

Hyve checks that workflow files are valid YAML:

### Valid YAML

```yaml
apiVersion: v1
kind: Workflow
metadata:
  name: deploy-app
spec:
  jobs:
    - name: deploy
      steps:
        - name: apply
          command: kubectl apply -f manifests/
```

### Invalid YAML Examples

<Tabs>
  <Tab title="Missing Colon">
    ```yaml
    # Invalid - missing colon after 'name'
    metadata:
      name deploy-app  # ❌ Missing colon

    # Valid
    metadata:
      name: deploy-app  # ✅ Correct
    ```
  </Tab>

  <Tab title="Wrong Indentation">
    ```yaml
    # Invalid - incorrect indentation
    spec:
      jobs:
      - name: deploy  # ❌ Wrong indentation
        steps:
          - name: apply

    # Valid
    spec:
      jobs:
        - name: deploy  # ✅ Correct indentation
          steps:
            - name: apply
    ```
  </Tab>

  <Tab title="Mixed Tabs/Spaces">
    ```yaml
    # Invalid - mixing tabs and spaces
    spec:
    →env:  # ❌ Tab character
      APP_NAME: my-app  # Spaces

    # Valid - use spaces consistently
    spec:
      env:  # ✅ Spaces only
        APP_NAME: my-app
    ```
  </Tab>
</Tabs>

## Schema Validation

Hyve validates the workflow structure and required fields:

### Required Fields

<ParamField path="apiVersion" type="string" required>
  Must be `v1`
</ParamField>

<ParamField path="kind" type="string" required>
  Must be `Workflow`
</ParamField>

<ParamField path="metadata.name" type="string" required>
  Unique workflow identifier
</ParamField>

<ParamField path="spec.jobs" type="array" required>
  At least one job must be defined
</ParamField>

<ParamField path="spec.jobs[].name" type="string" required>
  Each job must have a name
</ParamField>

<ParamField path="spec.jobs[].steps" type="array" required>
  Each job must have at least one step
</ParamField>

### Common Schema Errors

<AccordionGroup>
  <Accordion title="Missing Required Fields">
    ```yaml
    # Invalid - missing metadata.name
    apiVersion: v1
    kind: Workflow
    metadata:
      description: Deploy app
    # ❌ Missing: metadata.name

    # Valid
    apiVersion: v1
    kind: Workflow
    metadata:
      name: deploy-app
      description: Deploy app
    ```
  </Accordion>

  <Accordion title="Empty Jobs Array">
    ```yaml
    # Invalid - no jobs defined
    spec:
      jobs: []  # ❌ Empty array

    # Valid - at least one job
    spec:
      jobs:
        - name: deploy
          steps:
            - name: apply
              command: kubectl apply -f manifests/
    ```
  </Accordion>

  <Accordion title="Missing Step Command">
    ```yaml
    # Invalid - step has no command or script
    steps:
      - name: deploy  # ❌ No command or script

    # Valid - step has command
    steps:
      - name: deploy
        command: kubectl apply -f manifests/
    ```
  </Accordion>

  <Accordion title="Both Command and Script">
    ```yaml
    # Invalid - can't have both
    steps:
      - name: deploy
        command: kubectl apply -f manifests/  # ❌
        script: |  # ❌ Can't have both
          kubectl apply -f manifests/

    # Valid - use one or the other
    steps:
      - name: deploy
        command: kubectl apply -f manifests/
    ```
  </Accordion>
</AccordionGroup>

## Requirements Validation

Before executing, Hyve validates that requirements are met:

### Tool Validation

Checks that required tools are installed:

```yaml
requirements:
  tools:
    - name: kubectl
      version: "1.28"
    - name: helm
      version: "3.12"
```

**Validation checks:**
1. Tool exists in `PATH`
2. Version meets minimum requirement (if specified)

**Error messages:**

```
[ERROR] Requirements validation failed:
  - Required tool 'kubectl' not found in PATH
  - Tool 'helm' version mismatch: found 3.10.0, requires 3.12
```

### Secret Validation

Checks that required secrets are available:

```yaml
requirements:
  secrets:
    - name: DOCKER_TOKEN
      provider: docker
      required: true
    - name: GITHUB_TOKEN
      provider: github
      required: false
```

**Validation checks:**
1. Secret exists in environment or database
2. Required secrets must be present
3. Optional secrets show warnings if missing

**Error messages:**

```
[ERROR] Requirements validation failed:
  - Required secret 'DOCKER_TOKEN' not found
    Set via: hyve config set-token docker OR export DOCKER_TOKEN=your-secret

[WARN] Optional secret 'GITHUB_TOKEN' not found
```

## Manual Validation

Validate workflows without running them:

```bash
# Validate specific workflow
hyve workflow validate deploy-app

# Validate all workflows in repository
hyve workflow validate --all
```

**Output:**

```
[INFO] Validating workflow 'deploy-app'
[INFO] ✅ YAML syntax valid
[INFO] ✅ Schema validation passed
[INFO] ✅ Requirements validation passed
[INFO] Workflow 'deploy-app' is valid
```

## Dry Run

Preview workflow execution without actually running it:

```bash
# Dry run workflow
hyve workflow run deploy-app --dry-run
```

**Output:**

```
[INFO] DRY RUN: Workflow 'deploy-app'
[INFO] Would execute:

Job: build
  Step: docker-build
    Command: docker build -t my-app:latest .

Job: deploy
  Step: kubectl-apply
    Command: kubectl apply -f manifests/

[INFO] DRY RUN: No changes made
```

<Note>
  Dry run validates the workflow and shows what would execute, but doesn't run any commands.
</Note>

## Common Validation Errors

<AccordionGroup>
  <Accordion title="Invalid YAML syntax">
    **Error:**
    ```
    [ERROR] Invalid YAML syntax in workflow 'deploy-app'
    [ERROR] Line 10: mapping values are not allowed here
    ```

    **Solution:**
    - Check for missing colons
    - Verify indentation (use spaces, not tabs)
    - Ensure proper YAML structure
  </Accordion>

  <Accordion title="Missing required field">
    **Error:**
    ```
    [ERROR] Workflow validation failed:
    - Missing required field: metadata.name
    ```

    **Solution:**
    ```yaml
    # Add missing field
    metadata:
      name: my-workflow
    ```
  </Accordion>

  <Accordion title="Tool not found">
    **Error:**
    ```
    [ERROR] Required tool 'kubectl' not found in PATH
    ```

    **Solution:**
    ```bash
    # Install missing tool
    brew install kubectl

    # Verify installation
    kubectl version --client
    ```
  </Accordion>

  <Accordion title="Tool version mismatch">
    **Error:**
    ```
    [ERROR] Tool 'helm' version mismatch: found 3.10.0, requires 3.12
    ```

    **Solution:**
    ```bash
    # Update tool
    brew upgrade helm

    # Or adjust workflow requirement
    requirements:
      tools:
        - name: helm
          version: "3.10"  # Lower requirement
    ```
  </Accordion>

  <Accordion title="Secret not found">
    **Error:**
    ```
    [ERROR] Required secret 'DOCKER_TOKEN' not found
    ```

    **Solution:**
    ```bash
    # Set secret in database
    hyve config set-token docker

    # Or use environment variable
    export DOCKER_TOKEN=your-token
    ```
  </Accordion>

  <Accordion title="Invalid job dependency">
    **Error:**
    ```
    [ERROR] Job 'deploy' depends on non-existent job 'build'
    ```

    **Solution:**
    ```yaml
    # Ensure referenced job exists
    jobs:
      - name: build  # ✅ Add missing job
        steps:
          - name: build
            command: make build

      - name: deploy
        dependsOn: [build]  # Now valid
        steps:
          - name: deploy
            command: make deploy
    ```
  </Accordion>

  <Accordion title="Circular dependency">
    **Error:**
    ```
    [ERROR] Circular dependency detected: build → deploy → build
    ```

    **Solution:**
    ```yaml
    # Remove circular dependency
    jobs:
      - name: build
        steps:
          - name: build
            command: make build

      - name: deploy
        dependsOn: [build]  # ✅ One-way dependency
        steps:
          - name: deploy
            command: make deploy
    ```
  </Accordion>
</AccordionGroup>

## Best Practices

<AccordionGroup>
  <Accordion title="1. Validate Before Committing">
    ```bash
    # Validate workflow
    hyve workflow validate deploy-app

    # If valid, commit
    git add workflows/deploy-app.yaml
    git commit -m "Add deployment workflow"
    ```
  </Accordion>

  <Accordion title="2. Use Dry Run for Testing">
    ```bash
    # Test workflow without executing
    hyve workflow run deploy-app --dry-run

    # Review output, then run for real
    hyve workflow run deploy-app
    ```
  </Accordion>

  <Accordion title="3. Specify Tool Versions">
    ```yaml
    # Good - specific versions
    requirements:
      tools:
        - name: kubectl
          version: "1.28"
        - name: helm
          version: "3.12"

    # Less safe - no version check
    requirements:
      tools:
        - name: kubectl
        - name: helm
    ```
  </Accordion>

  <Accordion title="4. Test in Development First">
    ```bash
    # Switch to development
    hyve git use development

    # Test new workflow
    hyve workflow validate new-workflow
    hyve workflow run new-workflow --dry-run
    hyve workflow run new-workflow

    # If successful, use in production
    hyve git use production
    ```
  </Accordion>

  <Accordion title="5. Use Schema Validation Tools">
    ```bash
    # Use YAML linters
    yamllint workflows/deploy-app.yaml

    # Check with Hyve validator
    hyve workflow validate deploy-app
    ```
  </Accordion>
</AccordionGroup>

## Validation in CI/CD

Add validation to your CI/CD pipeline:

```yaml
# .github/workflows/validate-hyve.yml
name: Validate Hyve Workflows

on:
  pull_request:
    paths:
      - 'workflows/**.yaml'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Install Hyve
        run: |
          # Install Hyve
          go install github.com/your-org/hyve@latest

      - name: Validate workflows
        run: |
          hyve workflow validate --all
```

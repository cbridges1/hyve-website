## Overview

Hyve's CI/CD mode separates the **desired state** (what you want) from the **actual provisioning** (making it happen). Engineers declare infrastructure changes locally; a pipeline running in your CI/CD system performs the cloud operations. This keeps credentials off developer machines and provides a full audit trail through Git history.

<CardGroup cols={2}>
  <Card title="Local Mode" icon="laptop" color="#8b5cf6">
    Default. `hyve reconcile` provisions clusters directly from the machine that runs the command. Good for local development and simple setups.
  </Card>

  <Card title="CI/CD Mode" icon="arrows-rotate" color="#3b82f6">
    `hyve reconcile` validates and pushes desired state to Git, then exits. A pipeline calls `hyve reconcile --path .` to do the actual provisioning. Credentials never leave the pipeline.
  </Card>
</CardGroup>

## How It Works

<Steps>
  <Step title="Engineer declares a change">
    A team member adds or modifies a cluster definition and runs `hyve reconcile` locally.
  </Step>

  <Step title="Hyve validates and pushes state">
    In `cicd` mode, Hyve validates the YAML, commits the desired state to the Git repository, and exits without touching any cloud APIs.
  </Step>

  <Step title="Pipeline triggers">
    The push to the state repository triggers a CI/CD workflow (e.g. GitHub Actions).
  </Step>

  <Step title="Pipeline reconciles">
    The pipeline checks out the state repository and runs `hyve reconcile --path .`. Because `--path` is provided, Hyve always runs full local reconciliation regardless of the `hyve.yaml` mode setting.
  </Step>

  <Step title="Changes committed back">
    Hyve writes reconciliation results (cluster status, kubeconfigs, etc.) back to the repository and pushes them.
  </Step>
</Steps>

## Repository Setup

### Enable CI/CD Mode

Add a `hyve.yaml` file to the **root of your Hyve state repository**:

```yaml
# hyve.yaml
reconcile:
  mode: cicd
```

This single file tells Hyve to skip local provisioning when `hyve reconcile` is called without `--path`.

### Enable Strict Delete

By default, Hyve only manages clusters whose names begin with a recognized prefix (`hyve-`, `civo-deploy-`). Cloud clusters that exist but have no matching YAML definition are left untouched.

Enable **strict delete** to have Hyve delete _any_ cloud cluster that is no longer represented in your state repository:

```yaml
# hyve.yaml
reconcile:
  mode: cicd
  strictDelete: true
```

When `strictDelete: true` is set, every cloud cluster that cannot be matched to a YAML definition in your `clusters/` directory will be deleted during reconciliation.

<Warning>
  Strict delete removes clusters that have no YAML definition, regardless of their name. Enable this only after ensuring your state repository contains definitions for every cluster you want to keep.
</Warning>

### Recommended Repository Layout

```
hyve-state/                    # Your Hyve state repository
├── hyve.yaml                  # Reconcile mode: cicd
├── clusters/                  # Cluster definitions
│   ├── production.yaml
│   ├── staging.yaml
│   └── development.yaml
└── provider-configs/          # Provider-specific config
    ├── gcp.yaml
    ├── aws.yaml
    └── azure.yaml
```

### Example Cluster Definition

```yaml
# clusters/production.yaml
apiVersion: v1
kind: Cluster
metadata:
  name: production
spec:
  provider: gcp
  gcpProject: my-project
  region: us-central1
  nodes:
    - size: e2-standard-4
    - size: e2-standard-4
    - size: e2-standard-4
  clusterType: gke
```

## GitHub Actions Workflows

### Separate State Repository

Use this pattern when your Hyve state is kept in a dedicated repository, separate from your application repositories.

```
hyve-state/                    # Dedicated state repository
├── .github/
│   └── workflows/
│       └── reconcile.yml
├── hyve.yaml                  # reconcile.mode: cicd
├── clusters/
│   ├── production.yaml
│   └── staging.yaml
└── provider-configs/
    ├── aws.yaml
    ├── gcp.yaml
    ├── azure.yaml
    └── civo.yaml
```

**`.github/workflows/reconcile.yml`:**

```yaml
name: Hyve Reconcile

on:
  push:
    branches: [main]
    paths:
      - 'clusters/**'
      - 'provider-configs/**'
      - 'hyve.yaml'

jobs:
  reconcile:
    runs-on: ubuntu-latest
    permissions:
      contents: write   # Required to push reconciliation results back

    steps:
      - name: Checkout state repository
        uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Install Hyve
        run: |
          curl -sSL https://github.com/your-org/hyve/releases/latest/download/hyve-linux-amd64 \
            -o /usr/local/bin/hyve
          chmod +x /usr/local/bin/hyve

      - name: Reconcile clusters
        env:
          # Secret names match the ${...} references in your provider-configs/*.yaml files.
          # You choose the names — Hyve reads whatever variable each YAML field points to.

          # AWS (provider-configs/aws.yaml → accounts[*].access_key_id / secret_access_key)
          PROD_US_AWS_ACCESS_KEY_ID: ${{ secrets.PROD_US_AWS_ACCESS_KEY_ID }}
          PROD_US_AWS_SECRET_ACCESS_KEY: ${{ secrets.PROD_US_AWS_SECRET_ACCESS_KEY }}

          # GCP (provider-configs/gcp.yaml → projects[*].credentials_json)
          MY_PROJECT_GCP_CREDENTIALS_JSON: ${{ secrets.MY_PROJECT_GCP_CREDENTIALS_JSON }}

          # Azure (provider-configs/azure.yaml → subscriptions[*].tenant_id / client_id / client_secret)
          PROD_SUB_AZURE_TENANT_ID: ${{ secrets.PROD_SUB_AZURE_TENANT_ID }}
          PROD_SUB_AZURE_CLIENT_ID: ${{ secrets.PROD_SUB_AZURE_CLIENT_ID }}
          PROD_SUB_AZURE_CLIENT_SECRET: ${{ secrets.PROD_SUB_AZURE_CLIENT_SECRET }}

          # Civo (provider-configs/civo.yaml → organizations[*].token)
          MY_ORG_CIVO_TOKEN: ${{ secrets.MY_ORG_CIVO_TOKEN }}
        run: |
          hyve reconcile --path .
```

Credentials are resolved from the `${ENV_VAR}` references in your provider config files — the secret names are whatever you choose when writing those files. See [Provider Config Secret Mapping](#provider-config-secret-mapping) for the full reference.

### Scheduled Teardown

Use a scheduled workflow to automatically delete all clusters on a cron schedule — useful for ephemeral environments, cost-saving overnight teardowns, or end-of-sprint cleanup.

<Warning>
  This workflow **permanently deletes all clusters** managed by the state repository. Ensure `strictDelete: true` is set in `hyve.yaml` and that you intend to destroy all running clusters before scheduling this workflow.
</Warning>

Requires `strictDelete: true` in `hyve.yaml`:

```yaml
# hyve.yaml
reconcile:
  mode: cicd
  strictDelete: true
```

**`.github/workflows/scheduled-teardown.yml`:**

```yaml
name: Scheduled Cluster Teardown

on:
  schedule:
    - cron: '0 0 * * *'   # Every day at midnight UTC
  workflow_dispatch:        # Allow manual triggering

jobs:
  teardown:
    runs-on: ubuntu-latest
    permissions:
      contents: write

    steps:
      - name: Checkout state repository
        uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Install Hyve
        run: |
          curl -sSL https://github.com/your-org/hyve/releases/latest/download/hyve-linux-amd64 \
            -o /usr/local/bin/hyve
          chmod +x /usr/local/bin/hyve

      - name: Remove all cluster definitions
        run: find clusters/ -name '*.yaml' -o -name '*.yml' | xargs rm -f

      - name: Reconcile (tears down all clusters via strictDelete)
        env:
          # Same secrets as your regular reconcile workflow —
          # names match the ${...} references in provider-configs/*.yaml
          PROD_US_AWS_ACCESS_KEY_ID: ${{ secrets.PROD_US_AWS_ACCESS_KEY_ID }}
          PROD_US_AWS_SECRET_ACCESS_KEY: ${{ secrets.PROD_US_AWS_SECRET_ACCESS_KEY }}
          MY_PROJECT_GCP_CREDENTIALS_JSON: ${{ secrets.MY_PROJECT_GCP_CREDENTIALS_JSON }}
          PROD_SUB_AZURE_TENANT_ID: ${{ secrets.PROD_SUB_AZURE_TENANT_ID }}
          PROD_SUB_AZURE_CLIENT_ID: ${{ secrets.PROD_SUB_AZURE_CLIENT_ID }}
          PROD_SUB_AZURE_CLIENT_SECRET: ${{ secrets.PROD_SUB_AZURE_CLIENT_SECRET }}
          MY_ORG_CIVO_TOKEN: ${{ secrets.MY_ORG_CIVO_TOKEN }}
        run: |
          hyve reconcile --path .
```

Hyve finds zero cluster definitions in `clusters/`, discovers all existing clusters in your cloud provider, and deletes every cluster because none have a corresponding YAML definition (`strictDelete: true`). To limit teardown to specific providers or environments, keep the cluster definitions you want to preserve in `clusters/` and let `strictDelete` remove the rest.

## End-to-End Workflow

Here is the complete flow from developer change to provisioned cluster:

```bash
# 1. Developer: add a new cluster definition locally
hyve cluster add staging \
  --provider gcp \
  --project-name my-project \
  --region us-central1 \
  --nodes e2-standard-4,e2-standard-4

# 2. Developer: run reconcile (in cicd mode — pushes state, does NOT provision)
hyve reconcile
# Output:
# Reconcile mode: cicd
# Skipping local reconciliation — cluster provisioning will be handled by the CI/CD pipeline.
# Pushing desired state to repository...
# ✅ Desired state pushed to repository. The CI/CD pipeline will reconcile.

# 3. GitHub Actions triggers on push to main
# 4. Pipeline runs: hyve reconcile --path .
# Output:
# Using local repository path: /home/runner/work/hyve-state/hyve-state
# Cluster 'staging' created in GCP (us-central1)
# ✅ Changes committed and pushed to remote repository successfully
# Cluster reconciliation completed
```

## Provider Config Secret Mapping

Hyve resolves CI/CD credentials from your `provider-configs/*.yaml` files. Each credential field accepts either a **literal value** or an **environment variable reference** (`${VAR_NAME}`). At reconcile time, Hyve expands any `${...}` references using the process environment, so a single pipeline can serve multiple cloud accounts by simply making the right variables available.

You choose the secret names — Hyve has no naming requirements. The only rule is that the name inside `${...}` in your YAML must match the env var (or secret) you expose in the pipeline.

### AWS

```yaml
# provider-configs/aws.yaml
accounts:
  - name: prod-us
    account_id: "123456789012"
    access_key_id: ${PROD_US_AWS_ACCESS_KEY_ID}
    secret_access_key: ${PROD_US_AWS_SECRET_ACCESS_KEY}
    session_token: ${PROD_US_AWS_SESSION_TOKEN}   # optional
```

```yaml
# workflow env:
PROD_US_AWS_ACCESS_KEY_ID: ${{ secrets.PROD_US_AWS_ACCESS_KEY_ID }}
PROD_US_AWS_SECRET_ACCESS_KEY: ${{ secrets.PROD_US_AWS_SECRET_ACCESS_KEY }}
```

When `access_key_id` and `secret_access_key` are empty (or resolve to empty strings), Hyve falls back to the AWS SDK default credential chain (`~/.aws/credentials`, IAM instance roles, etc.).

### GCP

```yaml
# provider-configs/gcp.yaml
projects:
  - name: my-project
    project_id: my-gcp-project-id
    credentials_json: ${MY_PROJECT_GCP_CREDENTIALS_JSON}
```

```yaml
# workflow env:
MY_PROJECT_GCP_CREDENTIALS_JSON: ${{ secrets.MY_PROJECT_GCP_CREDENTIALS_JSON }}
```

The value should be the full contents of a GCP service account JSON key file. When empty, Hyve uses Application Default Credentials (`gcloud auth application-default login`).

### Azure

```yaml
# provider-configs/azure.yaml
subscriptions:
  - name: prod-sub
    subscription_id: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
    tenant_id: ${PROD_SUB_AZURE_TENANT_ID}
    client_id: ${PROD_SUB_AZURE_CLIENT_ID}
    client_secret: ${PROD_SUB_AZURE_CLIENT_SECRET}
    resource_groups:
      - name: prod-rg
        location: eastus
```

```yaml
# workflow env:
PROD_SUB_AZURE_TENANT_ID: ${{ secrets.PROD_SUB_AZURE_TENANT_ID }}
PROD_SUB_AZURE_CLIENT_ID: ${{ secrets.PROD_SUB_AZURE_CLIENT_ID }}
PROD_SUB_AZURE_CLIENT_SECRET: ${{ secrets.PROD_SUB_AZURE_CLIENT_SECRET }}
```

When credentials are empty, Hyve uses the Azure SDK default chain (`az login`, managed identity, etc.).

### Civo

```yaml
# provider-configs/civo.yaml
organizations:
  - name: my-org
    org_id: your-org-id
    token: ${MY_ORG_CIVO_TOKEN}   # name this whatever you like
```

```yaml
# workflow env:
MY_ORG_CIVO_TOKEN: ${{ secrets.MY_ORG_CIVO_TOKEN }}
```

### Credential Resolution Order

For each provider account, Hyve resolves credentials in this order:

1. **Provider config YAML** — `${ENV_VAR}` references are expanded; literal values are used directly
2. **Provider SDK default chain** — AWS credential chain, `gcloud` ADC, `az login` (used when YAML fields are empty)

<Info>
  GCP project IDs and Azure subscription IDs are always resolved from the YAML `project_id` / `subscription_id` fields — they are never secrets.
</Info>

## Required GitHub Secrets

Set these in your repository's **Settings → Secrets and variables → Actions**. The names must match whatever you put in the `${...}` references in your provider config YAML files.

| Secret | Description |
|--------|-------------|
| `GITHUB_TOKEN` | Automatically provided by GitHub Actions; needs `contents: write` permission |
| Credential secrets | Named however you like in `provider-configs/*.yaml` |

<Warning>
  Never commit literal API tokens or cloud credentials to your repository. Use `${ENV_VAR}` references in your YAML and store the values as GitHub Actions secrets or use OIDC-based authentication.
</Warning>

## Security Considerations

<AccordionGroup>
  <Accordion title="Use OIDC for AWS and GCP">
    Instead of storing long-lived AWS/GCP credentials as secrets, use GitHub Actions' OIDC provider with AWS IAM roles or GCP Workload Identity Federation. This eliminates the need to rotate secrets.

    ```yaml
    permissions:
    id-token: write
    contents: read
    ```
  </Accordion>

  <Accordion title="Limit repository permissions">
    Grant the `contents: write` permission only to the reconcile workflow, not globally. Use branch protection rules to require PR reviews before merging cluster changes to `main`.
  </Accordion>

  <Accordion title="Audit trail through Git history">
    Because all state changes flow through Git, every cluster change has an associated commit, author, and timestamp. Use `git log clusters/` to see the full history.
  </Accordion>

  <Accordion title="Review changes before they apply">
    Use pull requests for cluster changes. Engineers open a PR modifying `clusters/*.yaml`; reviewers approve; the merge to `main` triggers the pipeline. This gives you a review gate before anything is provisioned.
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Pipeline fails: Civo API token not found">
    Check that `provider-configs/civo.yaml` has a `token` field pointing to an env var reference, and that the corresponding secret is available in the workflow:

    ```yaml
    # provider-configs/civo.yaml
    organizations:
    - name: my-org
    token: ${MY_ORG_CIVO_TOKEN}
    ```

    ```yaml
    # workflow env:
    MY_ORG_CIVO_TOKEN: ${{ secrets.MY_ORG_CIVO_TOKEN }}
    ```
  </Accordion>

  <Accordion title="Pipeline fails: permission denied pushing to repository">
    Ensure the workflow has `contents: write` permission:

    ```yaml
    permissions:
    contents: write
    ```

    Or use a `token` with write access in the `checkout` step.
  </Accordion>

  <Accordion title="Local hyve reconcile provisions instead of pushing">
    Check that `hyve.yaml` is committed to the root of your state repository with `reconcile.mode: cicd`. If the file is absent, Hyve defaults to `local` mode.
  </Accordion>

  <Accordion title="hyve reconcile --path . fails: no clusters found">
    Verify that cluster YAML files exist under a `clusters/` directory at the path you provided. The `--path` flag should point to the root of the repository, not the `clusters/` subdirectory.
  </Accordion>
</AccordionGroup>

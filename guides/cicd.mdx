## Overview

Hyve's CI/CD mode separates the **desired state** (what you want) from the **actual provisioning** (making it happen). Engineers declare infrastructure changes locally; a pipeline running in your CI/CD system performs the cloud operations. This keeps credentials off developer machines and provides a full audit trail through Git history.

<CardGroup cols={2}>
  <Card title="Local Mode" icon="laptop" color="#8b5cf6">
    Default. `hyve reconcile` provisions clusters directly from the machine that runs the command. Good for local development and simple setups.
  </Card>

  <Card title="CI/CD Mode" icon="arrows-rotate" color="#3b82f6">
    `hyve reconcile` validates and pushes desired state to Git, then exits. A pipeline calls `hyve reconcile --path .` to do the actual provisioning. Credentials never leave the pipeline.
  </Card>
</CardGroup>

## How It Works

<Steps>
  <Step title="Engineer declares a change">
    A team member adds or modifies a cluster definition and runs `hyve reconcile` locally.
  </Step>

  <Step title="Hyve validates and pushes state">
    In `cicd` mode, Hyve validates the YAML, commits the desired state to the Git repository, and exits without touching any cloud APIs.
  </Step>

  <Step title="Pipeline triggers">
    The push to the state repository triggers a CI/CD workflow (e.g. GitHub Actions).
  </Step>

  <Step title="Pipeline reconciles">
    The pipeline checks out the state repository and runs `hyve reconcile --path .`. Because `--path` is provided, Hyve always runs full local reconciliation regardless of the `hyve.yaml` mode setting.
  </Step>

  <Step title="Changes committed back">
    Hyve writes reconciliation results (cluster status, kubeconfigs, etc.) back to the repository and pushes them.
  </Step>
</Steps>

## Repository Setup

### Enable CI/CD Mode

Add a `hyve.yaml` file to the **root of your Hyve state repository**:

```yaml
# hyve.yaml
reconcile:
  mode: cicd
```

This single file tells Hyve to skip local provisioning when `hyve reconcile` is called without `--path`.

### Enable Strict Delete

By default, Hyve only manages clusters whose names begin with a recognized prefix (`hyve-`, `civo-deploy-`). Cloud clusters that exist but have no matching YAML definition are left untouched.

Enable **strict delete** to have Hyve delete _any_ cloud cluster that is no longer represented in your state repository:

```yaml
# hyve.yaml
reconcile:
  mode: cicd
  strictDelete: true
```

When `strictDelete: true` is set, every cloud cluster that cannot be matched to a YAML definition in your `clusters/` directory will be deleted during reconciliation.

<Warning>
  Strict delete removes clusters that have no YAML definition, regardless of their name. Enable this only after ensuring your state repository contains definitions for every cluster you want to keep.
</Warning>

### Recommended Repository Layout

```
hyve-state/                    # Your Hyve state repository
тФЬтФАтФА hyve.yaml                  # Reconcile mode: cicd
тФЬтФАтФА clusters/                  # Cluster definitions
тФВ   тФЬтФАтФА production.yaml
тФВ   тФЬтФАтФА staging.yaml
тФВ   тФФтФАтФА development.yaml
тФФтФАтФА provider-configs/          # Provider-specific config
    тФЬтФАтФА gcp.yaml
    тФЬтФАтФА aws.yaml
    тФФтФАтФА azure.yaml
```

### Example Cluster Definition

```yaml
# clusters/production.yaml

  name: production

  provider: civo
  region: NYC1
  nodes:
    - size: g4s.kube.large
    - size: g4s.kube.large
    - size: g4s.kube.large

```

## GitHub Actions Workflows

### Option 1: Single Repository (App + State)

Use this pattern when your application code and Hyve state live in the same repository.

```
my-app/
тФЬтФАтФА .github/
тФВ   тФФтФАтФА workflows/
тФВ       тФФтФАтФА hyve-reconcile.yml
тФЬтФАтФА hyve.yaml                  # reconcile.mode: cicd
тФЬтФАтФА clusters/
тФВ   тФФтФАтФА production.yaml
тФФтФАтФА src/                       # Application code
```

**`.github/workflows/hyve-reconcile.yml`:**

```yaml
name: Hyve Reconcile

on:
  push:
    branches: [main]
    paths:
      - 'clusters/**'
      - 'hyve.yaml'

jobs:
  reconcile:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Install Hyve
        run: |
          curl -sSL https://github.com/your-org/hyve/releases/latest/download/hyve-linux-amd64 \
            -o /usr/local/bin/hyve
          chmod +x /usr/local/bin/hyve

      - name: Configure Git identity
        run: |
          git config --global user.name "hyve-bot"
          git config --global user.email "hyve-bot@users.noreply.github.com"

      - name: Reconcile clusters
        env:
          CIVO_TOKEN: ${{ secrets.CIVO_TOKEN }}
          HYVE_GIT_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          hyve reconcile --path .
```

### Option 2: Separate State Repository

Use this pattern when your Hyve state is kept in a dedicated repository, separate from your application repositories.

```
hyve-state/                    # Dedicated state repository
тФЬтФАтФА .github/
тФВ   тФФтФАтФА workflows/
тФВ       тФФтФАтФА reconcile.yml
тФЬтФАтФА hyve.yaml                  # reconcile.mode: cicd
тФФтФАтФА clusters/
    тФЬтФАтФА production.yaml
    тФФтФАтФА staging.yaml
```

**`.github/workflows/reconcile.yml`:**

```yaml
name: Hyve Reconcile

on:
  push:
    branches: [main]
    paths:
      - 'clusters/**'
      - 'provider-configs/**'
      - 'hyve.yaml'

jobs:
  reconcile:
    runs-on: ubuntu-latest
    permissions:
      contents: write   # Required to push reconciliation results back

    steps:
      - name: Checkout state repository
        uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Install Hyve
        run: |
          curl -sSL https://github.com/your-org/hyve/releases/latest/download/hyve-linux-amd64 \
            -o /usr/local/bin/hyve
          chmod +x /usr/local/bin/hyve

      - name: Configure Git identity
        run: |
          git config --global user.name "hyve-bot"
          git config --global user.email "hyve-bot@users.noreply.github.com"

      - name: Reconcile clusters
        env:
          CIVO_TOKEN: ${{ secrets.CIVO_TOKEN }}
          HYVE_GIT_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          hyve reconcile --path .
```

### Option 3: Multi-Cloud with Multiple Providers

This workflow handles multiple cloud providers in the same reconcile run.

```yaml
name: Hyve Reconcile (Multi-Cloud)

on:
  push:
    branches: [main]
    paths:
      - 'clusters/**'

jobs:
  reconcile:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      id-token: write   # Required for GCP Workload Identity / AWS OIDC

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/hyve-reconciler
          aws-region: us-east-1

      - name: Configure GCP credentials
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: projects/123/locations/global/workloadIdentityPools/hyve/providers/github
          service_account: hyve-reconciler@my-project.iam.gserviceaccount.com

      - name: Install Hyve
        run: |
          curl -sSL https://github.com/your-org/hyve/releases/latest/download/hyve-linux-amd64 \
            -o /usr/local/bin/hyve
          chmod +x /usr/local/bin/hyve

      - name: Configure Git identity
        run: |
          git config --global user.name "hyve-bot"
          git config --global user.email "hyve-bot@users.noreply.github.com"

      - name: Reconcile clusters
        env:
          CIVO_TOKEN: ${{ secrets.CIVO_TOKEN }}
          HYVE_GIT_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          hyve reconcile --path .
```

## End-to-End Workflow

Here is the complete flow from developer change to provisioned cluster:

```bash
# 1. Developer: add a new cluster definition locally
hyve cluster add staging \
  --provider civo \
  --region PHX1 \
  --nodes g4s.kube.medium,g4s.kube.medium

# 2. Developer: run reconcile (in cicd mode тАФ pushes state, does NOT provision)
hyve reconcile
# Output:
# Reconcile mode: cicd
# Skipping local reconciliation тАФ cluster provisioning will be handled by the CI/CD pipeline.
# Pushing desired state to repository...
# тЬЕ Desired state pushed to repository. The CI/CD pipeline will reconcile.

# 3. GitHub Actions triggers on push to main
# 4. Pipeline runs: hyve reconcile --path .
# Output:
# Using local repository path: /home/runner/work/hyve-state/hyve-state
# Cluster 'staging' created in Civo (NYC1)
# тЬЕ Changes committed and pushed to remote repository successfully
# Cluster reconciliation completed
```

## Named Account Secrets

Hyve supports per-account environment variables so a single pipeline can reconcile clusters across multiple cloud accounts, GCP projects, or Azure subscriptions. The naming pattern is:

```
{ACCOUNT_NAME}_{PROVIDER}_{CREDENTIAL}
```

The account name comes from the alias set in your cluster definition (e.g. `awsAccount: main-account`). Hyphens become underscores and everything is uppercased.

<Info>
  **GCP project IDs and Azure subscription IDs are never required as secrets.** Hyve resolves them automatically from the `provider-configs/gcp.yaml` and `provider-configs/azure.yaml` files in your state repository using the name alias. You only need to supply credentials (keys, tokens, service principal details).
</Info>

### AWS

| Cluster spec field | `awsAccount: main-account` |
|--------------------|---------------------------|
| Access key ID | `MAIN_ACCOUNT_AWS_ACCESS_KEY_ID` |
| Secret access key | `MAIN_ACCOUNT_AWS_SECRET_ACCESS_KEY` |
| Session token (optional) | `MAIN_ACCOUNT_AWS_SESSION_TOKEN` |

### GCP

| Cluster spec field | `gcpProject: my-project` |
|--------------------|--------------------------|
| Service account JSON (inline) | `MY_PROJECT_GCP_CREDENTIALS_JSON` |

The GCP project ID is resolved from `provider-configs/gcp.yaml` using the `gcpProject` alias тАФ no project ID secret needed.

### Azure

| Cluster spec field | `azureSubscription: prod-sub` |
|--------------------|-----------------------------|
| Tenant ID | `PROD_SUB_AZURE_TENANT_ID` |
| Client ID | `PROD_SUB_AZURE_CLIENT_ID` |
| Client secret | `PROD_SUB_AZURE_CLIENT_SECRET` |
| Resource group | `PROD_SUB_AZURE_RESOURCE_GROUP` |

The Azure subscription ID is resolved from `provider-configs/azure.yaml` using the `azureSubscription` alias тАФ no subscription ID secret needed.

### Civo

| Cluster spec field | `civoOrganization: my-org` |
|--------------------|---------------------------|
| API token | `MY_ORG_CIVO_TOKEN` |

### Priority Order

For each credential, Hyve checks in this order:

1. **Provider YAML config** тАФ GCP project ID and Azure subscription ID are always resolved from `provider-configs/*.yaml` by name alias
2. **Named account env var** тАФ `{ACCOUNT_NAME}_{PROVIDER}_{CREDENTIAL}` (checked before generic env vars)
3. **Standard env var** тАФ `AWS_ACCESS_KEY_ID`, `CIVO_TOKEN`, `AZURE_SUBSCRIPTION_ID`, etc.
4. **Provider SDK default chain** тАФ AWS credential chain (`~/.aws/credentials`, IAM roles), `az login`, GCP ADC

<Info>
  If no named env vars are set, the provider falls back to the standard credential chain, so existing local workflows are unaffected.
</Info>

### Multi-Account Example

**Provider config (committed to state repository):**

```yaml
# provider-configs/aws.yaml
accounts:
  - name: prod-us
    account_id: "111122223333"
  - name: prod-eu
    account_id: "444455556666"
```

**Cluster definitions:**

```yaml
# clusters/prod-us.yaml
spec:
  provider: aws
  awsAccount: prod-us   # тЖТ PROD_US_AWS_ACCESS_KEY_ID, etc.
  region: us-east-1
```

```yaml
# clusters/prod-eu.yaml
spec:
  provider: aws
  awsAccount: prod-eu   # тЖТ PROD_EU_AWS_ACCESS_KEY_ID, etc.
  region: eu-west-1
```

**GitHub Actions secrets (credentials only тАФ no account IDs needed):**

```
PROD_US_AWS_ACCESS_KEY_ID     = AKIA...
PROD_US_AWS_SECRET_ACCESS_KEY = ...
PROD_EU_AWS_ACCESS_KEY_ID     = AKIA...
PROD_EU_AWS_SECRET_ACCESS_KEY = ...
```

Hyve resolves the correct credentials for each cluster automatically during `hyve reconcile --path .`.

## Required GitHub Secrets

Set these secrets in your repository's **Settings тЖТ Secrets and variables тЖТ Actions**:

| Secret | Description |
|--------|-------------|
| `CIVO_TOKEN` | Fallback Civo API token (used when no named account token is found) |
| `GITHUB_TOKEN` | Automatically provided by GitHub Actions; needs `contents: write` permission |
| Named account secrets | Per-account credentials following the pattern above |

<Warning>
  Never commit API tokens or cloud credentials to your repository. Use GitHub Actions secrets or OIDC-based authentication for cloud providers.
</Warning>

## Security Considerations

<AccordionGroup>
  <Accordion title="Use OIDC for AWS and GCP">
    Instead of storing long-lived AWS/GCP credentials as secrets, use GitHub Actions' OIDC provider with AWS IAM roles or GCP Workload Identity Federation. This eliminates the need to rotate secrets.

    ```yaml
    permissions:
      id-token: write
      contents: read
    ```
  </Accordion>

  <Accordion title="Limit repository permissions">
    Grant the `contents: write` permission only to the reconcile workflow, not globally. Use branch protection rules to require PR reviews before merging cluster changes to `main`.
  </Accordion>

  <Accordion title="Audit trail through Git history">
    Because all state changes flow through Git, every cluster change has an associated commit, author, and timestamp. Use `git log clusters/` to see the full history.
  </Accordion>

  <Accordion title="Review changes before they apply">
    Use pull requests for cluster changes. Engineers open a PR modifying `clusters/*.yaml`; reviewers approve; the merge to `main` triggers the pipeline. This gives you a review gate before anything is provisioned.
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Pipeline fails: CIVO_TOKEN not found">
    Ensure the `CIVO_TOKEN` secret is set in your repository secrets and that the workflow references it correctly:

    ```yaml
    env:
      CIVO_TOKEN: ${{ secrets.CIVO_TOKEN }}
    ```
  </Accordion>

  <Accordion title="Pipeline fails: permission denied pushing to repository">
    Ensure the workflow has `contents: write` permission:

    ```yaml
    permissions:
      contents: write
    ```

    Or use a `token` with write access in the `checkout` step.
  </Accordion>

  <Accordion title="Local hyve reconcile provisions instead of pushing">
    Check that `hyve.yaml` is committed to the root of your state repository with `reconcile.mode: cicd`. If the file is absent, Hyve defaults to `local` mode.
  </Accordion>

  <Accordion title="hyve reconcile --path . fails: no clusters found">
    Verify that cluster YAML files exist under a `clusters/` directory at the path you provided. The `--path` flag should point to the root of the repository, not the `clusters/` subdirectory.
  </Accordion>
</AccordionGroup>

## Overview

Hyve's CI/CD mode separates the **desired state** (what you want) from the **actual provisioning** (making it happen). Engineers declare infrastructure changes locally; a pipeline running in your CI/CD system performs the cloud operations. This keeps credentials off developer machines and provides a full audit trail through Git history.

<CardGroup cols={2}>
  <Card title="Local Mode" icon="laptop" color="#8b5cf6">
    Default. `hyve reconcile` provisions clusters directly from the machine that runs the command. Good for local development and simple setups.
  </Card>

  <Card title="CI/CD Mode" icon="arrows-rotate" color="#3b82f6">
    `hyve reconcile` validates and pushes desired state to Git, then exits. A pipeline calls `hyve reconcile --path .` to do the actual provisioning. Credentials never leave the pipeline.
  </Card>
</CardGroup>

## How It Works

<Steps>
  <Step title="Engineer declares a change">
    A team member adds or modifies a cluster definition and runs `hyve reconcile` locally.
  </Step>

  <Step title="Hyve validates and pushes state">
    In `cicd` mode, Hyve validates the YAML, commits the desired state to the Git repository, and exits without touching any cloud APIs.
  </Step>

  <Step title="Pipeline triggers">
    The push to the state repository triggers a CI/CD workflow (e.g. GitHub Actions).
  </Step>

  <Step title="Pipeline reconciles">
    The pipeline checks out the state repository and runs `hyve reconcile --path .`. Because `--path` is provided, Hyve always runs full local reconciliation regardless of the `hyve.yaml` mode setting.
  </Step>

  <Step title="Changes committed back">
    Hyve writes reconciliation results (cluster status, kubeconfigs, etc.) back to the repository and pushes them.
  </Step>
</Steps>

## Repository Setup

### Enable CI/CD Mode

Add a `hyve.yaml` file to the **root of your Hyve state repository**:

```yaml
# hyve.yaml
reconcile:
  mode: cicd
```

This single file tells Hyve to skip local provisioning when `hyve reconcile` is called without `--path`.

### Enable Strict Delete

By default, Hyve only manages clusters whose names begin with a recognized prefix (`hyve-`, `civo-deploy-`). Cloud clusters that exist but have no matching YAML definition are left untouched.

Enable **strict delete** to have Hyve delete _any_ cloud cluster that is no longer represented in your state repository:

```yaml
# hyve.yaml
reconcile:
  mode: cicd
  strictDelete: true
```

When `strictDelete: true` is set, every cloud cluster that cannot be matched to a YAML definition in your `clusters/` directory will be deleted during reconciliation.

<Warning>
  Strict delete removes clusters that have no YAML definition, regardless of their name. Enable this only after ensuring your state repository contains definitions for every cluster you want to keep.
</Warning>

### Recommended Repository Layout

```
hyve-state/                    # Your Hyve state repository
├── hyve.yaml                  # Reconcile mode: cicd
├── clusters/                  # Cluster definitions
│   ├── production.yaml
│   ├── staging.yaml
│   └── development.yaml
└── provider-configs/          # Provider-specific config
    ├── gcp.yaml
    ├── aws.yaml
    └── azure.yaml
```

## GitHub Actions Workflows

### Separate State Repository

Use this pattern when your Hyve state is kept in a dedicated repository, separate from your application repositories.

```
hyve-state/                    # Dedicated state repository
├── .github/
│   └── workflows/
│       └── reconcile.yml
├── hyve.yaml                  # reconcile.mode: cicd
├── clusters/
│   ├── production.yaml
│   └── staging.yaml
└── provider-configs/
    ├── aws.yaml
    ├── gcp.yaml
    ├── azure.yaml
    └── civo.yaml
```

**`.github/workflows/reconcile.yml`:**

```yaml
name: Hyve Reconcile

on:
  push:
    branches: [main]
    paths:
      - 'clusters/**'
      - 'provider-configs/**'
      - 'hyve.yaml'

  schedule:
    - cron: '0 20 * * *'   # Every day at 8:00 PM UTC

  workflow_dispatch:

jobs:
  reconcile:
    runs-on: ubuntu-latest
    permissions:
      contents: write   # Required to push reconciliation results back

    steps:
      - name: Checkout state repository
        uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Configure git identity
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"

      - name: Install Hyve
        run: |
          curl -sSL https://github.com/cbridges1/hyve/releases/latest/download/hyve-linux-amd64 \
            -o /usr/local/bin/hyve
          chmod +x /usr/local/bin/hyve

      - name: Reconcile clusters
        env:
          # Secret names match the ${...} references in your provider-configs/**/*.yaml files.
          # You choose the names — Hyve reads whatever variable each YAML field points to.

          # AWS (provider-configs/aws.yaml → access_key_id / secret_access_key per account)
          PROD_US_AWS_ACCESS_KEY_ID: ${{ secrets.PROD_US_AWS_ACCESS_KEY_ID }}
          PROD_US_AWS_SECRET_ACCESS_KEY: ${{ secrets.PROD_US_AWS_SECRET_ACCESS_KEY }}

          # GCP (provider-configs/gcp.yaml → credentials_json per project)
          MY_PROJECT_GCP_CREDENTIALS_JSON: ${{ secrets.MY_PROJECT_GCP_CREDENTIALS_JSON }}

          # Azure (provider-configs/azure.yaml → tenant_id / client_id / client_secret per subscription)
          PROD_SUB_AZURE_TENANT_ID: ${{ secrets.PROD_SUB_AZURE_TENANT_ID }}
          PROD_SUB_AZURE_CLIENT_ID: ${{ secrets.PROD_SUB_AZURE_CLIENT_ID }}
          PROD_SUB_AZURE_CLIENT_SECRET: ${{ secrets.PROD_SUB_AZURE_CLIENT_SECRET }}

          # Civo (provider-configs/civo.yaml → token per organization)
          MY_ORG_CIVO_TOKEN: ${{ secrets.MY_ORG_CIVO_TOKEN }}
        run: hyve reconcile --path .
```

Credentials are resolved from the `${ENV_VAR}` references in your provider config files — the secret names are whatever you choose when writing those files. See [Provider Config Secret Mapping](#provider-config-secret-mapping) for the full reference.

### Scheduled Reconcile

Add a `schedule` trigger directly to your existing reconcile workflow. This is the backbone for all time-based cluster automation — expiry-based deletion (`expiresAt`), normal drift correction, and paused-cluster awareness all happen through the same reconcile path.

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'clusters/**'
      - 'provider-configs/**'
      - 'hyve.yaml'

  schedule:
    - cron: '0 20 * * *'   # Every day at 8:00 PM UTC

  workflow_dispatch:
```

Adjust the cron expression to match your preferred cadence. Common schedules:

| Cron | Description |
|------|-------------|
| `0 20 * * *` | Daily at 8:00 PM UTC |
| `0 * * * *` | Hourly — tightest expiry window |
| `0 0 * * *` | Daily at midnight UTC |
| `0 8,20 * * *` | Twice daily at 8 AM and 8 PM UTC |

<Info>
  The git identity and `contents: write` permission are only exercised when a cluster is deleted (via `spec.delete: true` or an expired `spec.expiresAt`). They have no effect on create, update, or paused-cluster runs.
</Info>

### Full Teardown

To destroy **every** cluster in the state repository on a schedule (e.g. overnight cost savings), use `strictDelete: true` and remove all cluster definitions before reconciling:

<Warning>
  This workflow permanently deletes all clusters managed by the state repository.
</Warning>

```yaml
# hyve.yaml — enable strict delete
reconcile:
  mode: cicd
  strictDelete: true
```

```yaml
# .github/workflows/nightly-teardown.yml
name: Nightly Teardown

on:
  schedule:
    - cron: '0 0 * * *'   # Midnight UTC
  workflow_dispatch:

jobs:
  teardown:
    runs-on: ubuntu-latest
    permissions:
      contents: write

    steps:
      - uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Configure git identity
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"

      - name: Install Hyve
        run: |
          curl -sSL https://github.com/cbridges1/hyve/releases/latest/download/hyve-linux-amd64 \
            -o /usr/local/bin/hyve
          chmod +x /usr/local/bin/hyve

      - name: Remove all cluster definitions
        run: find clusters/ \( -name '*.yaml' -o -name '*.yml' \) -delete

      - name: Reconcile (strict-delete removes all cloud clusters)
        env:
          PROD_US_AWS_ACCESS_KEY_ID: ${{ secrets.PROD_US_AWS_ACCESS_KEY_ID }}
          PROD_US_AWS_SECRET_ACCESS_KEY: ${{ secrets.PROD_US_AWS_SECRET_ACCESS_KEY }}
          MY_PROJECT_GCP_CREDENTIALS_JSON: ${{ secrets.MY_PROJECT_GCP_CREDENTIALS_JSON }}
          MY_ORG_CIVO_TOKEN: ${{ secrets.MY_ORG_CIVO_TOKEN }}
        run: hyve reconcile --path .
```

## Deleting Clusters with onDestroy Workflows

Deleting the YAML file for a cluster directly means the reconciler never sees the cluster definition — and therefore cannot run any `onDestroy` workflows before the cluster is removed. To safely delete a cluster and have its lifecycle workflows execute, use the **mark-for-deletion** pattern instead.

### How It Works

Set `spec.delete: true` in the cluster YAML **without removing the file**:

```yaml
# clusters/staging.yaml
apiVersion: v1
kind: Cluster
metadata:
  name: staging
spec:
  delete: true          # <-- marks this cluster for deletion
  provider: aws
  awsAccount: prod-us
  region: us-east-1
  clusterType: eks
  workflows:
    onDestroy:
      - cleanup-namespaces
      - drain-nodes
```

When the reconciler processes this file it will:

<Steps>
  <Step title="Detect the deletion marker">
    `DetermineAction` returns `ActionDelete` as soon as it sees `spec.delete: true` — no cloud API call is needed to make this decision.
  </Step>

  <Step title="Run onDestroy workflows">
    All workflows listed under `spec.workflows.onDestroy` are executed against the cluster before any cloud resources are touched.
  </Step>

  <Step title="Delete the cluster">
    The cluster is removed from the cloud provider.
  </Step>

  <Step title="Delete the cluster">
    The cluster is removed from the cloud provider.
  </Step>

  <Step title="Remove the YAML file and push">
    The reconciler removes the cluster definition file from the repository, commits the change, and pushes it back to the remote. The deletion marker is never left behind.
  </Step>
</Steps>

<Warning>
  Do not delete the YAML file directly if you need `onDestroy` workflows to run. The reconciler must be able to read the file — including the workflow list — before it can execute lifecycle hooks.
</Warning>

### Required GitHub Actions Configuration

Because the reconciler removes the cluster file and pushes back to the repository after deletion, the GitHub Actions workflow needs write access to the repository and a valid git committer identity.

#### 1. Contents write permission

```yaml
jobs:
  reconcile:
    runs-on: ubuntu-latest
    permissions:
      contents: write
```

#### 2. Git committer identity

GitHub Actions runners do not configure a git identity by default. Add this step before running Hyve:

```yaml
- name: Configure git identity
  run: |
    git config user.name "github-actions[bot]"
    git config user.email "github-actions[bot]@users.noreply.github.com"
```

#### 3. Branch protection rules

If your default branch requires pull request reviews or passing checks before merging, the reconciler's direct push will be rejected.

Choose one of:

- **Exempt the Actions bot** — add `github-actions[bot]` to the bypass list in your branch protection settings.
- **Use a Personal Access Token (PAT)** — pass it to the checkout step so the push is made as a user with bypass permissions.

  ```yaml
  - uses: actions/checkout@v4
    with:
      token: ${{ secrets.HYVE_PAT }}
  ```

#### Complete example

```yaml
name: Hyve Reconcile

on:
  push:
    branches: [main]
    paths:
      - 'clusters/**'
      - 'provider-configs/**'
      - 'hyve.yaml'

  schedule:
    - cron: '0 20 * * *'   # Every day at 8:00 PM UTC

  workflow_dispatch:

jobs:
  reconcile:
    runs-on: ubuntu-latest
    permissions:
      contents: write

    steps:
      - name: Checkout state repository
        uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Configure git identity
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"

      - name: Install Hyve
        run: |
          curl -sSL https://github.com/cbridges1/hyve/releases/latest/download/hyve-linux-amd64 \
            -o /usr/local/bin/hyve
          chmod +x /usr/local/bin/hyve

      - name: Reconcile clusters
        env:
          PROD_US_AWS_ACCESS_KEY_ID: ${{ secrets.PROD_US_AWS_ACCESS_KEY_ID }}
          PROD_US_AWS_SECRET_ACCESS_KEY: ${{ secrets.PROD_US_AWS_SECRET_ACCESS_KEY }}
        run: hyve reconcile --path .
```

<Info>
  The `Configure git identity` step and `contents: write` permission are only exercised when a cluster marked `spec.delete: true` is processed. They have no effect on normal create or update reconciliations.
</Info>

## Pausing Reconciliation

Set `spec.pause: true` to keep a cluster's definition in the repository while telling the reconciler to skip it entirely. The cluster continues running in the cloud — Hyve simply does not compare desired state or make any changes until `pause` is removed.

```yaml
# clusters/staging.yaml
apiVersion: v1
kind: Cluster
metadata:
  name: staging
spec:
  pause: true       # reconciler skips this cluster on every run
  provider: civo
  region: NYC1
  nodes: [g4s.kube.medium]
```

**Common use cases:**

- **Maintenance windows** — prevent Hyve from touching a cluster while you perform manual operations
- **Cost investigation** — keep the definition versioned while you decide whether to delete or resize
- **Temporary divergence** — allow a cluster to drift from its YAML definition without triggering an update

To resume normal reconciliation, remove the `pause: true` line (or set it to `false`) and commit.

<Warning>
  A paused cluster still counts toward the active cluster set, so orphan cleanup (`strictDelete`) will not delete it. Pause does not affect cloud resources — it only affects Hyve's behavior.
</Warning>

## Scheduled Cluster Expiry

Set `spec.expiresAt` to an RFC 3339 timestamp to have the reconciler automatically delete a cluster when that time passes. When the scheduled reconcile runs after the expiry time, the reconciler treats the cluster exactly as if `spec.delete: true` had been set — running `onDestroy` workflows, deleting from the cloud provider, and removing the YAML file.

```yaml
# clusters/dev-sprint-47.yaml
apiVersion: v1
kind: Cluster
metadata:
  name: dev-sprint-47
spec:
  provider: civo
  region: PHX1
  nodes: [g4s.kube.small]
  expiresAt: "2026-05-01T00:00:00Z"   # deleted automatically after this time
  workflows:
    onDestroy:
      - notify-team
      - cleanup-namespaces
```

The expiry is checked on every reconcile run. With an hourly scheduled reconcile, clusters are cleaned up within ~1 hour of their deadline.

**Combining pause and expiresAt:**

A paused cluster is always skipped — the expiry check is never reached. If you pause a cluster that has an `expiresAt` set, it will not be deleted until `pause` is removed.

### Scheduled Reconcile Workflow

No special pipeline logic is needed. The `schedule` trigger on your existing reconcile workflow handles drift correction, expirations, and deletions in a single run — no separate workflow file required.

The expiry check runs on every reconcile. With a daily schedule at 8 PM, expired clusters are cleaned up within 24 hours of their deadline. Use an hourly schedule (`0 * * * *`) if you need tighter expiry windows.

### Ephemeral Environment Pattern

A common pattern for feature-branch or sprint clusters:

```yaml
# clusters/feature-login-redesign.yaml
apiVersion: v1
kind: Cluster
metadata:
  name: feature-login-redesign
spec:
  provider: aws
  awsAccount: dev
  region: us-east-1
  clusterType: eks
  expiresAt: "2026-04-25T18:00:00Z"   # end of sprint
  workflows:
    onCreate:
      - deploy-feature-branch
    onDestroy:
      - export-test-results
      - notify-slack
```

When the sprint ends, the cluster tears itself down automatically — no manual cleanup required.

## End-to-End Workflow

Here is the complete flow from developer change to provisioned cluster:

```bash
# 1. Developer: add a new cluster definition locally
hyve cluster add staging \
  --provider gcp \
  --project-name my-project \
  --region us-central1 \
  --nodes e2-standard-4,e2-standard-4

# 2. Developer: run reconcile (in cicd mode — pushes state, does NOT provision)
hyve reconcile
# Output:
# Reconcile mode: cicd
# Skipping local reconciliation — cluster provisioning will be handled by the CI/CD pipeline.
# Pushing desired state to repository...
# ✅ Desired state pushed to repository. The CI/CD pipeline will reconcile.

# 3. GitHub Actions triggers on push to main
# 4. Pipeline runs: hyve reconcile --path .
# Output:
# Using local repository path: /home/runner/work/hyve-state/hyve-state
# Cluster 'staging' created in GCP (us-central1)
# ✅ Changes committed and pushed to remote repository successfully
# Cluster reconciliation completed
```

## Provider Config Secret Mapping

Hyve resolves CI/CD credentials from your `provider-configs/**/*.yaml` files. Each credential field accepts either a **literal value** or an **environment variable reference** (`${VAR_NAME}`). At reconcile time, Hyve expands any `${...}` references using the process environment, so a single pipeline can serve multiple cloud accounts by simply making the right variables available.

You choose the secret names — Hyve has no naming requirements. The only rule is that the name inside `${...}` in your YAML must match the env var (or secret) you expose in the pipeline.

### AWS

```yaml
# provider-configs/aws.yaml
accounts:
  - name: prod-us
    account_id: "123456789012"
    access_key_id: ${PROD_US_AWS_ACCESS_KEY_ID}
    secret_access_key: ${PROD_US_AWS_SECRET_ACCESS_KEY}
    session_token: ${PROD_US_AWS_SESSION_TOKEN}   # optional
```

```yaml
# workflow env:
PROD_US_AWS_ACCESS_KEY_ID: ${{ secrets.PROD_US_AWS_ACCESS_KEY_ID }}
PROD_US_AWS_SECRET_ACCESS_KEY: ${{ secrets.PROD_US_AWS_SECRET_ACCESS_KEY }}
```

When `access_key_id` and `secret_access_key` are empty (or resolve to empty strings), Hyve falls back to the AWS SDK default credential chain (`~/.aws/credentials`, IAM instance roles, etc.).

### GCP

```yaml
# provider-configs/gcp.yaml
projects:
  - name: my-project
    project_id: my-gcp-project-id
    credentials_json: ${MY_PROJECT_GCP_CREDENTIALS_JSON}
```

```yaml
# workflow env:
MY_PROJECT_GCP_CREDENTIALS_JSON: ${{ secrets.MY_PROJECT_GCP_CREDENTIALS_JSON }}
```

The value should be the full contents of a GCP service account JSON key file. When empty, Hyve uses Application Default Credentials (`gcloud auth application-default login`).

### Azure

```yaml
# provider-configs/azure.yaml
subscriptions:
  - name: prod-sub
    subscription_id: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
    tenant_id: ${PROD_SUB_AZURE_TENANT_ID}
    client_id: ${PROD_SUB_AZURE_CLIENT_ID}
    client_secret: ${PROD_SUB_AZURE_CLIENT_SECRET}
```

```yaml
# workflow env:
PROD_SUB_AZURE_TENANT_ID: ${{ secrets.PROD_SUB_AZURE_TENANT_ID }}
PROD_SUB_AZURE_CLIENT_ID: ${{ secrets.PROD_SUB_AZURE_CLIENT_ID }}
PROD_SUB_AZURE_CLIENT_SECRET: ${{ secrets.PROD_SUB_AZURE_CLIENT_SECRET }}
```

When credentials are empty, Hyve uses the Azure SDK default chain (`az login`, managed identity, etc.).

### Civo

```yaml
# provider-configs/civo.yaml
organizations:
  - name: my-org
    org_id: cbridges1-id
    token: ${MY_ORG_CIVO_TOKEN}   # name this whatever you like
```

```yaml
# workflow env:
MY_ORG_CIVO_TOKEN: ${{ secrets.MY_ORG_CIVO_TOKEN }}
```

### Credential Resolution Order

For each provider account, Hyve resolves credentials in this order:

1. **Provider config YAML** — `${ENV_VAR}` references are expanded; literal values are used directly
2. **Provider SDK default chain** — AWS credential chain, `gcloud` ADC, `az login` (used when YAML fields are empty)

<Info>
  GCP project IDs and Azure subscription IDs are always resolved from the YAML `project_id` / `subscription_id` fields — they are never secrets.
</Info>

## Required GitHub Secrets

Set these in your repository's **Settings → Secrets and variables → Actions**. The names must match whatever you put in the `${...}` references in your provider config YAML files.

| Secret | Description |
|--------|-------------|
| `GITHUB_TOKEN` | Automatically provided by GitHub Actions; needs `contents: write` permission |
| Credential secrets | Named however you like in `provider-configs/*.yaml` |

<Warning>
  Never commit literal API tokens or cloud credentials to your repository. Use `${ENV_VAR}` references in your YAML and store the values as GitHub Actions secrets or use OIDC-based authentication.
</Warning>

## Security Considerations

<AccordionGroup>
  <Accordion title="Use OIDC for AWS and GCP">
    Instead of storing long-lived AWS/GCP credentials as secrets, use GitHub Actions' OIDC provider with AWS IAM roles or GCP Workload Identity Federation. This eliminates the need to rotate secrets.

    ```yaml
    permissions:
    id-token: write
    contents: read
    ```
  </Accordion>

  <Accordion title="Limit repository permissions">
    Grant the `contents: write` permission only to the reconcile workflow, not globally. Use branch protection rules to require PR reviews before merging cluster changes to `main`.
  </Accordion>

  <Accordion title="Audit trail through Git history">
    Because all state changes flow through Git, every cluster change has an associated commit, author, and timestamp. Use `git log clusters/` to see the full history.
  </Accordion>

  <Accordion title="Review changes before they apply">
    Use pull requests for cluster changes. Engineers open a PR modifying `clusters/*.yaml`; reviewers approve; the merge to `main` triggers the pipeline. This gives you a review gate before anything is provisioned.
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Pipeline fails: Civo API token not found">
    Check that `provider-configs/civo.yaml` has a `token` field pointing to an env var reference, and that the corresponding secret is available in the workflow:

    ```yaml
    # provider-configs/civo.yaml
    organizations:
      - name: my-org
        org_id: cbridges1-id
        token: ${MY_ORG_CIVO_TOKEN}
    ```

    ```yaml
    # workflow env:
    MY_ORG_CIVO_TOKEN: ${{ secrets.MY_ORG_CIVO_TOKEN }}
    ```
  </Accordion>

  <Accordion title="Pipeline fails: permission denied pushing to repository">
    Ensure the workflow has `contents: write` permission:

    ```yaml
    permissions:
    contents: write
    ```

    Or use a `token` with write access in the `checkout` step.
  </Accordion>

  <Accordion title="Local hyve reconcile provisions instead of pushing">
    Check that `hyve.yaml` is committed to the root of your state repository with `reconcile.mode: cicd`. If the file is absent, Hyve defaults to `local` mode.
  </Accordion>

  <Accordion title="onDestroy workflows did not run before cluster was deleted">
    This happens when the cluster YAML file was deleted directly from the repository instead of using the `spec.delete: true` marker. The reconciler never saw the file, so it had no workflow list to execute — the cluster appeared as an unmanaged orphan and was removed without running lifecycle hooks.

    **Prevention:** Always set `spec.delete: true` in the YAML file and let the reconciler handle the deletion. Do not `git rm` or manually delete cluster files when `onDestroy` workflows need to run.
  </Accordion>

  <Accordion title="Pipeline fails: permission denied pushing cluster file removal">
    The reconciler needs write access to commit the cluster file deletion after `spec.delete: true` is processed.

    Ensure the workflow has `contents: write` permission and a git identity is configured:

    ```yaml
    permissions:
      contents: write

    steps:
      - name: Configure git identity
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
    ```

    If branch protection rules block direct pushes, see [Branch protection rules](#3-branch-protection-rules).
  </Accordion>

  <Accordion title="hyve reconcile --path . fails: no clusters found">
    Verify that cluster YAML files exist under a `clusters/` directory at the path you provided. The `--path` flag should point to the root of the repository, not the `clusters/` subdirectory.
  </Accordion>

  <Accordion title="Cluster with expiresAt was not deleted on schedule">
    The expiry check runs on every reconcile. Verify:

    1. The scheduled reconcile workflow is running — check the Actions tab in GitHub.
    2. The `expiresAt` value is a valid RFC 3339 timestamp: `"2026-05-01T00:00:00Z"`. An invalid format logs a warning and skips the expiry check.
    3. The cluster does not also have `pause: true` set — a paused cluster is skipped entirely before the expiry check is reached.
    4. The workflow has `contents: write` permission and a git identity configured, since the reconciler needs to commit the YAML file removal after deletion.
  </Accordion>

  <Accordion title="Paused cluster is being modified by the reconciler">
    If a cluster with `pause: true` is still being reconciled, verify that the change has been committed and pushed to the branch the pipeline checks out. The reconciler reads the YAML files from the checked-out repository — a local change that has not been pushed will not take effect in the pipeline.
  </Accordion>
</AccordionGroup>

## Overview

Hyve uses Git repositories to store all cluster state, workflows, and templates. This guide covers repository management, branching, synchronization, and multi-environment workflows.

## Repository Basics

### Adding Repositories

Add a Git repository to store your infrastructure state:

```bash
# Add a repository
hyve git add production --repo-url https://github.com/company/hyve-prod.git

# Add with custom local path
hyve git add staging \
  --repo-url https://github.com/company/hyve-staging.git \
  --local-path ~/hyve/staging
```

<Note>
  The repository is cloned to `~/.hyve/repositories/<name>/` by default
</Note>

### Listing Repositories

```bash
hyve git list
```

**Output:**
```
📁 Configured repositories (3):

  production (active)
    URL: https://github.com/company/hyve-prod.git
    Path: /Users/username/.hyve/repositories/production
    Branch: main

  staging
    URL: https://github.com/company/hyve-staging.git
    Path: /Users/username/.hyve/repositories/staging
    Branch: main

  development
    URL: https://github.com/company/hyve-dev.git
    Path: /Users/username/.hyve/repositories/development
    Branch: develop
```

### Switching Repositories

```bash
# Switch to a different repository
hyve git use staging

# Check current repository
hyve git status
```

### Removing Repositories

```bash
# Remove a repository
hyve git remove development

# Reset all repositories (clean slate)
hyve git reset
```

<Warning>
  Removing a repository deletes the local clone and configuration. The remote repository is not affected.
</Warning>

## Branch Management

```bash
# List all branches
hyve git branch list
```

**Output:**
```

  feature/add-staging-cluster (abc1234)
* main (def5678)
  feature/update-workflows (ghi9012)
```

<Tabs>
  <Tab title="Basic">

    ```bash
    hyve git branch create feature/new-cluster
    ```

  </Tab>

  <Tab title="Create and Switch">

    ```bash
    hyve git branch create feature/staging --switch
    ```

### Switching Branches

```bash
# Switch to existing branch
hyve git branch switch develop

# Switch and pull latest changes
hyve git branch switch main --pull
```

### Deleting Branches

```bash
# Delete a branch
hyve git branch delete feature/old-config

# Force delete
hyve git branch delete feature/experimental --force
```

## Synchronization

### Pull Changes

Pull latest changes from the remote:

```bash
hyve git pull
```

### Push Changes

Push local changes to the remote:

```bash
# Push with custom message
hyve git push "Updated cluster configuration"

# Push with default message (auto-generated from changes)
hyve git push
```

<Note>
  The push command automatically stages all changes, commits them, and pushes to remote
</Note>

### Sync (Pull + Push)

Synchronize both ways:

```bash
# Sync with custom message
hyve git sync "Synced infrastructure changes"

# Sync with prompts
hyve git sync
```

**Workflow:**
1. Pulls latest changes from remote
2. Shows local changes detected
3. Commits local changes with your message
4. Pushes to remote

## Authentication

### Global Credentials

Store encrypted credentials for all repositories:

```bash
# Store credentials
hyve git credentials --username your-username --password your_token

# Update password only
hyve git credentials --password new_token

# Update username only
hyve git credentials --username new_username

# View current credentials
hyve git credentials

# Clear credentials
hyve git credentials --clear
```

<Info>
  Credentials are encrypted using AES-GCM and stored in `~/.hyve/credentials.db`
</Info>

### Environment Variable Fallback

```bash
# Set environment variable
export HYVE_GIT_TOKEN=your_personal_access_token

# Add to shell profile
echo 'export HYVE_GIT_TOKEN=your_token' >> ~/.bashrc
```

### Credential Priority

1. **Global credentials** (encrypted database)
2. **Environment variable** (`HYVE_GIT_TOKEN`)
3. **Per-repository username** (if configured)

## Multi-Environment Workflows

### Scenario: Dev, Staging, Production

<Steps>
  <Step title="Set up repositories">
    ```bash
    # Development
    hyve git add development \
      --repo-url https://github.com/company/hyve-dev.git

    # Staging
    hyve git add staging \
      --repo-url https://github.com/company/hyve-staging.git

    # Production
    hyve git add production \
      --repo-url https://github.com/company/hyve-prod.git
    ```
  </Step>

  <Step title="Configure each environment">
    ```bash
    # Development clusters
    hyve git use development
    hyve cluster add dev-app --region PHX1 --nodes g4s.kube.small

    # Staging clusters
    hyve git use staging
    hyve cluster add staging-app --region NYC1 --nodes g4s.kube.medium

    # Production clusters
    hyve git use production
    hyve cluster add prod-app --region NYC1 --nodes g4s.kube.large
    ```
  </Step>

  <Step title="Manage independently">
    Each repository maintains its own:
    - Cluster definitions
    - Workflows
    - Templates
    - Kubeconfigs (encrypted per repository)
  </Step>
</Steps>

### Branch-Based Environments

Use branches within a single repository:

```bash
# Main repository
hyve git add infrastructure --repo-url https://github.com/company/infra.git

# Development branch
hyve git branch create develop --switch
hyve cluster add dev-cluster --region PHX1 --nodes g4s.kube.small

# Staging branch
hyve git branch create staging --switch
hyve cluster add staging-cluster --region NYC1 --nodes g4s.kube.medium

# Production in main
hyve git branch switch main
hyve cluster add prod-cluster --region NYC1 --nodes g4s.kube.large
```

## Repository Structure

Each repository follows this structure:

```
repository/
├── clusters/           # Cluster definitions
│   ├── production.yaml
│   ├── staging.yaml
│   └── dev-cluster.yaml
├── workflows/          # Workflow definitions
│   ├── deploy-app.yaml
│   ├── setup-monitoring.yaml
│   └── backup-db.yaml
├── templates/          # Cluster templates
│   ├── prod-template.yaml
│   ├── staging-template.yaml
│   └── dev-template.yaml
└── .gitignore         # Ignore temporary files
```

## Best Practices

<AccordionGroup>
  <Accordion title="1. Use Descriptive Repository Names">
    ```bash
    # Good
    hyve git add production-us-east
    hyve git add production-eu-west
    hyve git add staging-shared

    # Avoid
    hyve git add repo1
    hyve git add my-clusters
    ```
  </Accordion>

  <Accordion title="2. Regular Synchronization">
    ```bash
    # At start of day
    hyve git pull

    # Before making changes
    hyve git pull

    # After making changes
    hyve git push "Your changes description"

    # Or use sync for both
    hyve git sync "Daily sync"
    ```
  </Accordion>

  <Accordion title="3. Branch for Experimental Changes">
    ```bash
    # Create feature branch
    hyve git branch create feature/test-new-cluster --switch

    # Make changes
    hyve cluster add test-cluster --region PHX1 --nodes g4s.kube.small

    # Test and verify
    hyve reconcile

    # Merge to main when ready
    hyve git branch switch main
    # Manually merge via Git or pull request
    ```
  </Accordion>

  <Accordion title="4. Separate Repositories for Environments">
    Use different repositories for different environments to:
    - Isolate configurations
    - Control access permissions
    - Prevent accidental changes
    - Enable different Git workflows per environment
  </Accordion>

  <Accordion title="5. Commit Messages">
    ```bash
    # Descriptive messages
    hyve git push "Add production cluster for US region"
    hyve git push "Update workflow to include monitoring setup"
    hyve git push "Scale production cluster to 5 nodes"

    # Avoid vague messages
    hyve git push "Update"
    hyve git push "Changes"
    ```
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Authentication Failed">
    **Problem:** Git operations fail with authentication error

    **Solutions:**
    ```bash
    # Check credentials
    hyve git credentials

    # Update credentials
    hyve git credentials --username user --password new_token

    # Use environment variable
    export HYVE_GIT_TOKEN=your_token

    # For private repos, ensure you have access
    # Check SSH keys if using SSH URLs
    ```
  </Accordion>

  <Accordion title="Merge Conflicts">
    **Problem:** Git pull fails due to conflicts

    **Solutions:**
    ```bash
    # Navigate to repository
    cd ~/.hyve/repositories/production

    # Check status
    git status

    # Resolve conflicts manually
    # Edit conflicting files

    # Commit resolution
    git add .
    git commit -m "Resolve merge conflicts"

    # Push resolved changes
    hyve git push "Resolved conflicts"
    ```
  </Accordion>

  <Accordion title="Repository Not Found">
    **Problem:** No Git repository configured error

    **Solution:**
    ```bash
    # Add a repository
    hyve git add production --repo-url https://github.com/company/hyve-prod.git

    # Verify it's added
    hyve git list

    # Set as active if needed
    hyve git use production
    ```
  </Accordion>

  <Accordion title="Local Changes Conflict">
    **Problem:** Can't switch repositories/branches with uncommitted changes

    **Solutions:**
    ```bash
    # Option 1: Commit changes
    hyve git push "WIP: In progress work"

    # Option 2: Stash changes manually
    cd ~/.hyve/repositories/production
    git stash
    # Switch repositories/branches
    # Later: git stash pop

    # Option 3: Discard changes (careful!)
    cd ~/.hyve/repositories/production
    git reset --hard HEAD
    ```
  </Accordion>
</AccordionGroup>

## Advanced Workflows

### Team Collaboration

```bash
# Team member 1
hyve git pull                           # Get latest
hyve cluster add new-cluster            # Make changes
hyve git push "Add new cluster"         # Push changes

# Team member 2
hyve git pull                           # Pull changes from team member 1
# See new cluster automatically
hyve reconcile                          # Apply changes
```

### Multi-Region Setup

```bash
# US Region
hyve git add us-infrastructure \
  --repo-url https://github.com/company/hyve-us.git

hyve cluster add us-cluster --region NYC1 --nodes g4s.kube.large

# EU Region
hyve git add eu-infrastructure \
  --repo-url https://github.com/company/hyve-eu.git

hyve git use eu-infrastructure
hyve cluster add eu-cluster --region FRA1 --nodes g4s.kube.large
```

### Promotion Workflow

```bash
# Develop in dev repository
hyve git use development
hyve workflow create deploy-v2 --template deployment-pipeline

# Test in staging
# Copy workflow file to staging repo
cp ~/.hyve/repositories/development/workflows/deploy-v2.yaml \
   ~/.hyve/repositories/staging/workflows/

hyve git use staging
hyve git push "Add deploy-v2 workflow"
hyve workflow run deploy-v2 --cluster staging-cluster

# Promote to production after testing
cp ~/.hyve/repositories/staging/workflows/deploy-v2.yaml \
   ~/.hyve/repositories/production/workflows/

hyve git use production
hyve git push "Promote deploy-v2 workflow to production"
```

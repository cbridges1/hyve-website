## Overview

Hyve enables declarative Kubernetes cluster management through Git-based YAML definitions. This guide covers creating, modifying, scaling, deleting, and managing clusters across regions.

## Cluster Basics

### Adding Clusters

Create a new cluster definition and provision it to your cloud provider:

<Tabs>
  <Tab title="Basic">
    ```bash
    # Create a basic cluster
    hyve cluster add my-cluster --provider civo --region PHX1 --nodes g4s.kube.small
    ```
  </Tab>

  <Tab title="Production">
    ```bash
    # Production cluster with multiple nodes
    hyve cluster add prod-app \
      --provider civo \
      --region NYC1 \
      --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large \
      --cluster-type k3s
    ```
  </Tab>

  <Tab title="Multi-Region">
    ```bash
    # US East cluster
    hyve cluster add us-east-api --provider civo --region NYC1 --nodes g4s.kube.medium

    # EU cluster
    hyve cluster add eu-api --provider civo --region FRA1 --nodes g4s.kube.medium

    # Asia-Pacific cluster
    hyve cluster add apac-api --provider civo --region LON1 --nodes g4s.kube.medium
    ```
  </Tab>
</Tabs>

<Note>
  Cluster definitions are stored as YAML files in `~/.hyve/repositories/<repo>/clusters/`
</Note>

### Cluster Definition Structure

When you add a cluster, Hyve creates a YAML definition:

```yaml
apiVersion: v1
kind: Cluster
metadata:
  name: production
  region: PHX1
spec:
  provider: civo
  nodes:
    - g4s.kube.large
    - g4s.kube.large
  clusterType: k3s
  ingress:
    enabled: true
    loadBalancer: true
```

## Scaling Clusters

### Scaling Up

Add more nodes to an existing cluster:

```bash
# Scale from 2 to 3 nodes
hyve cluster modify production \
  --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large
```

### Scaling Down

Reduce the number of nodes:

```bash
# Scale from 3 to 2 nodes
hyve cluster modify production \
  --nodes g4s.kube.large,g4s.kube.large
```

<Warning>
  Scaling down removes nodes immediately. Ensure workloads are properly distributed before scaling down.
</Warning>

### Vertical Scaling

Change node sizes:

```bash
# Upgrade from medium to large nodes
hyve cluster modify staging-app \
  --nodes g4s.kube.large,g4s.kube.large
```

## Deleting Clusters

### Standard Delete

Remove the cluster YAML and reconcile — the pipeline (or local reconcile) handles cloud deletion:

```bash
hyve cluster delete my-cluster
```

**This command:**
1. Removes the cluster configuration YAML file
2. Commits and pushes the change to the state repository
3. Runs reconciliation

In CI/CD mode with `strictDelete: true`, the push triggers the pipeline which deletes the cloud cluster. In local mode with `strictDelete: true`, the local reconcile deletes the cloud cluster immediately.

### Delete Options

<Tabs>
  <Tab title="Force (immediate cloud delete)">
    ```bash
    # Delete from cloud immediately, bypassing CI/CD
    hyve cluster delete my-cluster --force
    ```
    Searches for the cluster, removes ingress controllers, deletes from cloud, then removes the YAML and reconciles.
  </Tab>

  <Tab title="Force Cloud Delete (no config)">
    ```bash
    # Delete from cloud even without a config file
    hyve cluster delete my-cluster --force --force-cloud
    ```
  </Tab>

  <Tab title="Force Delete by Name">
    ```bash
    # Search and delete cluster by name across regions
    hyve cluster force-delete my-cluster

    # Force delete in specific region
    hyve cluster force-delete my-cluster --region NYC1
    ```
  </Tab>
</Tabs>

<Warning>
  Cluster deletion is permanent and cannot be undone. Always backup critical data before deleting clusters.
</Warning>

## Cluster Types

### K3s Clusters (Default)

Lightweight Kubernetes distribution:

```bash
hyve cluster add k3s-cluster \
  --provider civo \
  --region PHX1 \
  --nodes g4s.kube.medium \
  --cluster-type k3s
```

**Benefits:**
- Fast startup time
- Lower resource overhead
- Perfect for edge/IoT workloads
- Full Kubernetes compatibility

### Talos Clusters

Immutable Kubernetes OS:

```bash
hyve cluster add talos-cluster \
  --provider civo \
  --region NYC1 \
  --nodes g4s.kube.large \
  --cluster-type talos
```

**Benefits:**
- Enhanced security
- Immutable infrastructure
- API-driven management
- Minimal attack surface

## Ingress Configuration

### Enable Ingress

Clusters are created with ingress enabled by default:

```yaml
spec:
  ingress:
    enabled: true
    loadBalancer: true
```

### Ingress Features

- **Automatic Load Balancer**: External access to services
- **TLS/SSL Support**: Secure HTTPS connections
- **Path-Based Routing**: Route traffic by URL path
- **Host-Based Routing**: Multiple domains on one cluster

Deploy across regions for high availability:

```bash

  --provider civo \
  --region NYC1 \

  --provider civo \
  --region PHX1 \
```

## Best Practices

<AccordionGroup>
  <Accordion title="1. Use Descriptive Cluster Names">
    ```bash
    # Good - clear environment and purpose
    hyve cluster add prod-api-us-east --provider civo --region NYC1 --nodes g4s.kube.large
    hyve cluster add staging-web-app --provider civo --region PHX1 --nodes g4s.kube.medium

    # Avoid - unclear naming
    hyve cluster add cluster1 --provider civo --region NYC1 --nodes g4s.kube.small
    hyve cluster add test --provider civo --region PHX1 --nodes g4s.kube.small
    ```
  </Accordion>

  <Accordion title="2. High Availability for Production">
    ```bash
    # Always use multiple nodes for production
    hyve cluster add prod-cluster \
      --provider civo \
      --region NYC1 \
      --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large

    # Consider multi-region for critical workloads
    hyve cluster add prod-primary --provider civo --region NYC1 --nodes g4s.kube.large,g4s.kube.large
    hyve cluster add prod-failover --provider civo --region FRA1 --nodes g4s.kube.large,g4s.kube.large
    ```
  </Accordion>

  <Accordion title="3. Right-Size Your Clusters">
    ```bash
    # Development - small and cost-effective
    hyve cluster add dev --provider civo --region PHX1 --nodes g4s.kube.small

    # Staging - similar to production
    hyve cluster add staging --provider civo --region NYC1 --nodes g4s.kube.medium,g4s.kube.medium

    # Production - appropriate for load
    hyve cluster add prod --provider civo --region NYC1 --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large
    ```
  </Accordion>

  <Accordion title="4. Use GitOps for Changes">
    ```bash
    # Make changes through Git
    hyve cluster modify prod-cluster --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large
    hyve git push "Scale production cluster to 3 nodes"

    # Changes are tracked in Git history
    # Easy to rollback if needed
    ```
  </Accordion>

  <Accordion title="5. Regular Reconciliation">
    ```bash
    # Sync cluster state regularly
    hyve reconcile

    # After any manual changes
    hyve cluster modify staging-app --nodes g4s.kube.medium,g4s.kube.medium
    hyve reconcile
    ```
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Cluster Creation Fails">
    **Problem:** Cluster creation returns an error

    **Solutions:**
    ```bash
    # 1. Verify API token is set
    hyve config list-tokens

    # 2. Check region is valid
    hyve cluster add test-cluster --provider civo --region PHX1 --nodes g4s.kube.small

    # 3. Verify node size is available
    # Use only valid node sizes: xsmall, small, medium, large, xlarge

    # 4. Check provider account limits
    # Log into Civo dashboard to verify quota
    ```
  </Accordion>

  <Accordion title="Cannot Delete Cluster">
    **Problem:** Cluster deletion fails

    **Solutions:**
    ```bash
    # Force delete from cloud provider
    hyve cluster force-delete my-cluster

    # Search specific region
    hyve cluster force-delete my-cluster --region NYC1

    # Manually delete via Civo dashboard as last resort
    ```
  </Accordion>

  <Accordion title="Cluster Not Found After Creation">
    **Problem:** Created cluster doesn't appear in provider

    **Solutions:**
    ```bash
    # Check reconciliation status
    hyve reconcile

    # Sync kubeconfigs
    hyve kubeconfig sync

    # List clusters in repository
    hyve cluster list

    # Verify Git sync
    hyve git pull
    hyve git status
    ```
  </Accordion>

  <Accordion title="Modification Not Applied">
    **Problem:** Cluster changes don't take effect

    **Solutions:**
    ```bash
    # 1. Verify YAML was updated
    cat ~/.hyve/repositories/production/clusters/my-cluster.yaml

    # 2. Run reconciliation
    hyve reconcile

    # 3. Check for Git sync issues
    hyve git pull
    hyve git status

    # 4. Verify provider API access
    hyve config get-token civo
    ```
  </Accordion>

  <Accordion title="Region Errors">
    **Problem:** Invalid region specified

    **Solution:** Check your provider's documentation for valid region codes (see [Available Regions](#available-regions)), then update the cluster:
    ```bash
    hyve cluster modify my-cluster --region <valid-region>
    hyve reconcile
    ```
  </Accordion>
</AccordionGroup>

## Advanced Workflows

### Blue-Green Deployments

Deploy new version alongside existing:

```bash
# Create green cluster with new version
hyve cluster add prod-green \
  --provider civo \
  --region NYC1 \
  --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large

# Test green cluster
hyve workflow run test-suite --cluster prod-green

# Switch traffic (update DNS/load balancer)
# ...

# Delete blue cluster when green is stable
hyve cluster delete prod-blue
```

### Cluster Migration

Move workloads between regions:

<Steps>
  <Step title="Create new cluster">
    ```bash
    hyve cluster add prod-new-region \
      --provider civo \
      --region FRA1 \
      --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large
    ```
  </Step>

  <Step title="Sync kubeconfig">
    ```bash
    hyve kubeconfig sync
    ```
  </Step>

  <Step title="Deploy to new cluster">
    ```bash
    hyve workflow run deploy-app --cluster prod-new-region
    ```
  </Step>

  <Step title="Verify and switch traffic">
    ```bash
    # Test new cluster
    eval $(hyve use prod-new-region)
    kubectl get pods

    # Update DNS/LB to new cluster
    ```
  </Step>

  <Step title="Decommission old cluster">
    ```bash
    hyve cluster delete prod-old-region
    ```
  </Step>
</Steps>

## Overview

Hyve provides secure, encrypted kubeconfig storage and management for seamless cluster access. This guide covers syncing, retrieving, merging, and using kubeconfigs across your clusters.

## Kubeconfig Basics

### What is a Kubeconfig?

A kubeconfig file contains:
- Cluster connection details (API server URL, CA certificate)
- User authentication credentials
- Context definitions (cluster + user + namespace)

Hyve stores kubeconfigs encrypted in `~/.hyve/kubeconfigs.db` per repository.

### Storage Architecture

```
~/.hyve/
├── kubeconfigs.db          # Encrypted kubeconfig storage (AES-GCM)
├── temp/                   # Temporary kubeconfig files
│   └── kubeconfig-repo-cluster
└── repositories/           # Git repositories
    └── production/
        └── clusters/       # Cluster definitions
```

<Info>
  Kubeconfigs are encrypted using AES-GCM encryption with repository-scoped keys
</Info>

## Syncing Kubeconfigs

### Sync All Clusters

Retrieve and store kubeconfigs from all active clusters:

```bash
hyve kubeconfig sync
```

**This command:**
1. Loads all cluster definitions from current repository
2. Connects to cloud provider for each cluster
3. Retrieves kubeconfig for active clusters
4. Encrypts and stores in local database
5. Updates existing kubeconfigs if changed

**Output:**
```
📁 Syncing kubeconfigs for repository 'production'
🔍 Found 3 cluster definitions
📥 Retrieving kubeconfig for prod-app... ✅
📥 Retrieving kubeconfig for staging-app... ✅
📥 Retrieving kubeconfig for dev-cluster... ✅
✅ Kubeconfig sync completed
```

## Quick Cluster Access

### Using `hyve use` (Recommended)

The fastest way to switch cluster context:

```bash
# Set kubeconfig for current terminal session
eval $(hyve use my-cluster)
```

**This command:**
1. Creates temporary kubeconfig in `~/.hyve/temp/`
2. Outputs `export KUBECONFIG=...` command
3. Evaluating sets `KUBECONFIG` environment variable

**Example workflow:**
```bash
# Switch to production cluster
eval $(hyve use prod-app)
kubectl get nodes

# Switch to staging cluster
eval $(hyve use staging-app)
kubectl get pods

# Revert to default kubeconfig
unset KUBECONFIG
```

<Note>
  The `eval $(hyve use <cluster>)` pattern is the recommended approach for quick cluster switching
</Note>

### Manual Use Command

For more control and information:

```bash
hyve kubeconfig use my-cluster
```

**Output:**
```
✅ Temporary kubeconfig created for cluster 'my-cluster' (repository: production)
📁 Temporary file: /Users/username/.hyve/temp/kubeconfig-production-my-cluster

🔧 To use this kubeconfig in your current terminal session, run:
   export KUBECONFIG='/Users/username/.hyve/temp/kubeconfig-production-my-cluster'

💡 This will only affect your current terminal session.
💡 To revert, use: unset KUBECONFIG

🧪 Test your connection:
   export KUBECONFIG='/Users/username/.hyve/temp/kubeconfig-production-my-cluster' && kubectl get nodes

⚡ For automatic setup, use:
   eval $(hyve kubeconfig use my-cluster --eval)
```

## Merging Kubeconfigs

### Merge into ~/.kube/config

Add cluster context to your main kubeconfig:

```bash
hyve kubeconfig merge my-cluster
```

**This command:**
1. Reads cluster kubeconfig from Hyve storage
2. Creates backup of existing `~/.kube/config`
3. Merges cluster context into main config
4. Preserves existing contexts

**Output:**
```
🔀 Merging cluster 'my-cluster' into /Users/username/.kube/config
📦 Backup created at /Users/username/.kube/config.backup
✅ Successfully merged cluster 'my-cluster' into /Users/username/.kube/config

💡 Next steps:
   kubectl config use-context my-cluster
   kubectl get nodes
```

### Verify Merged Contexts

```bash
# List all contexts
kubectl config get-contexts

# Switch to merged context
kubectl config use-context my-cluster

# Test connection
kubectl get nodes
```

## Removing Kubeconfigs

### Remove from ~/.kube/config

Remove cluster context from main kubeconfig:

```bash
hyve kubeconfig remove my-cluster
```

**Output:**
```
📦 Backup created at /Users/username/.kube/config.backup
🗑️  Removing cluster 'my-cluster' from /Users/username/.kube/config
✅ Successfully removed cluster 'my-cluster' from /Users/username/.kube/config

💡 View remaining contexts:
   kubectl config get-contexts
```

<Warning>
  This removes the cluster from `~/.kube/config` only. The kubeconfig remains stored in Hyve's database.
</Warning>

## Multi-Cluster Workflows

### Quick Cluster Switching

```bash
# Check production
eval $(hyve use prod-app)
kubectl get pods -A

# Check staging
eval $(hyve use staging-app)
kubectl get pods -A

# Check development
eval $(hyve use dev-cluster)
kubectl get pods -A

# Revert to default
unset KUBECONFIG
```

### Parallel Operations

```bash
# Terminal 1 - Production
eval $(hyve use prod-app)
kubectl get nodes
kubectl logs -f deployment/my-app

# Terminal 2 - Staging
eval $(hyve use staging-app)
kubectl get nodes
kubectl logs -f deployment/my-app

# Terminal 3 - Development
eval $(hyve use dev-cluster)
kubectl get nodes
kubectl logs -f deployment/my-app
```

### Cross-Cluster Operations

```bash
# Export from production
eval $(hyve use prod-app)
kubectl get configmap my-config -o yaml > config.yaml

# Import to staging
eval $(hyve use staging-app)
kubectl apply -f config.yaml

# Verify
kubectl get configmap my-config
```

## Repository-Scoped Kubeconfigs

### How It Works

Kubeconfigs are stored per repository:

```bash
# Production repository clusters
hyve git use production
hyve kubeconfig sync
eval $(hyve use prod-cluster)

# Staging repository clusters
hyve git use staging
hyve kubeconfig sync
eval $(hyve use staging-cluster)

# Development repository clusters
hyve git use development
hyve kubeconfig sync
eval $(hyve use dev-cluster)
```

### Benefits

- **Isolation**: Production and development kubeconfigs separated
- **Security**: Different repositories can have different access controls
- **Organization**: Clusters grouped by environment or team
- **Simplicity**: Same cluster names in different repositories

## Encryption and Security

### Encryption Details

Hyve uses AES-GCM encryption for kubeconfig storage:

- **Algorithm**: AES-256-GCM
- **Key Derivation**: Repository-scoped keys
- **Storage**: SQLite database with encrypted values
- **In-Transit**: Never transmitted unencrypted

### Security Best Practices

<AccordionGroup>
  <Accordion title="1. Regular Credential Rotation">
    ```bash
    # Periodically sync to get new credentials
    hyve kubeconfig sync

    # Clusters rotate credentials automatically
    # Sync ensures you have current tokens
    ```
  </Accordion>

  <Accordion title="2. Limit Kubeconfig Sharing">
    ```bash
    # Don't share kubeconfig files
    # Instead, grant cluster access through cloud provider

    # Users should sync their own kubeconfigs
    hyve kubeconfig sync
    ```
  </Accordion>

  <Accordion title="3. Use Temporary Files">
    ```bash
    # Temporary kubeconfigs are created in ~/.hyve/temp/
    # These are cleaned up automatically

    # Prefer `eval $(hyve use cluster)` over saving files
    eval $(hyve use my-cluster)
    ```
  </Accordion>

  <Accordion title="4. Secure ~/.hyve Directory">
    ```bash
    # Ensure proper permissions
    chmod 700 ~/.hyve
    chmod 600 ~/.hyve/kubeconfigs.db

    # Database contains encrypted kubeconfigs
    # Protect against unauthorized access
    ```
  </Accordion>

  <Accordion title="5. Use RBAC in Clusters">
    ```bash
    # Configure cluster RBAC to limit access
    kubectl create rolebinding developer \
      --clusterrole=developer \
      --user=user@example.com \
      --namespace=development
    ```
  </Accordion>
</AccordionGroup>

## Migration and Portability

### Migrate Encryption Keys

If moving databases between machines:

```bash
# On new machine, migrate from old hostname-based encryption
hyve kubeconfig migrate "old-macbook.local"
```

**Output:**
```
🔄 Starting migration for repository: production
🔑 Old hostname: old-macbook.local

✅ Migration completed successfully!

📝 All kubeconfigs have been re-encrypted with the new portable key format.
💡 Your kubeconfigs will now work across different machines without hostname dependencies.
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="No Kubeconfigs Found">
    **Problem:** `hyve cluster list` shows no clusters

    **Solutions:**
    ```bash
    # 1. Sync kubeconfigs
    hyve kubeconfig sync

    # 2. Verify clusters exist
    ls ~/.hyve/repositories/production/clusters/

    # 3. Check current repository
    hyve git status

    # 4. Verify API credentials
    hyve config list-tokens
    ```
  </Accordion>

  <Accordion title="Kubeconfig Not Found for Cluster">
    **Problem:** Error retrieving kubeconfig

    **Solutions:**
    ```bash
    # 1. Sync kubeconfigs
    hyve kubeconfig sync

    # 2. Verify cluster exists in cloud
    # Log into Civo dashboard to check

    # 3. Check cluster is ready
    hyve reconcile

    # 4. Verify cluster name
    hyve cluster list
    ```
  </Accordion>

  <Accordion title="Connection Refused">
    **Problem:** kubectl commands fail with connection error

    **Solutions:**
    ```bash
    # 1. Sync latest kubeconfig
    hyve kubeconfig sync
    eval $(hyve use my-cluster)

    # 2. Verify cluster is running
    # Check cloud provider dashboard

    # 3. Check API server endpoint
    kubectl cluster-info

    # 4. Verify network connectivity
    ping my-cluster.k8s.civo.com
    ```
  </Accordion>

  <Accordion title="Permission Denied">
    **Problem:** kubectl commands fail with permission errors

    **Solutions:**
    ```bash
    # 1. Sync to get updated credentials
    hyve kubeconfig sync

    # 2. Verify RBAC permissions in cluster
    kubectl auth can-i get pods

    # 3. Check user/token in kubeconfig
    hyve kubeconfig get my-cluster | grep token

    # 4. Contact cluster administrator
    # May need role/rolebinding created
    ```
  </Accordion>

  <Accordion title="Wrong Cluster Context">
    **Problem:** Commands execute on wrong cluster

    **Solutions:**
    ```bash
    # 1. Check current context
    kubectl config current-context

    # 2. Set correct cluster
    eval $(hyve use correct-cluster)

    # 3. Verify cluster
    kubectl get nodes
    kubectl cluster-info

    # 4. Use full context name if needed
    kubectl config use-context my-cluster
    ```
  </Accordion>

  <Accordion title="Merge Conflicts">
    **Problem:** Error merging into ~/.kube/config

    **Solutions:**
    ```bash
    # 1. Backup current config
    cp ~/.kube/config ~/.kube/config.backup

    # 2. Try merge again
    hyve kubeconfig merge my-cluster

    # 3. Manual merge if needed
    hyve kubeconfig get my-cluster > /tmp/cluster.yaml
    # Use kubectl or yq to merge manually

    # 4. Restore from backup if issues
    cp ~/.kube/config.backup ~/.kube/config
    ```
  </Accordion>

  <Accordion title="Encryption/Decryption Errors">
    **Problem:** Failed to decrypt kubeconfig

    **Solutions:**
    ```bash
    # 1. Try migration (if moved from another machine)
    hyve kubeconfig migrate "old-hostname"

    # 2. Re-sync kubeconfigs
    hyve kubeconfig sync

    # 3. Check database integrity
    ls -la ~/.hyve/kubeconfigs.db

    # 4. Last resort: Remove and re-sync
    rm ~/.hyve/kubeconfigs.db
    hyve kubeconfig sync
    ```
  </Accordion>
</AccordionGroup>

## Advanced Usage

### Shell Aliases

Add to `~/.bashrc` or `~/.zshrc`:

```bash
# Quick cluster switching
alias kprod='eval $(hyve use prod-app)'
alias kstaging='eval $(hyve use staging-app)'
alias kdev='eval $(hyve use dev-cluster)'

# Combined with kubectl
alias kgp='kubectl get pods'
alias kgs='kubectl get svc'
alias kgn='kubectl get nodes'

# Quick context reset
alias kreset='unset KUBECONFIG'
```

**Usage:**
```bash
# Switch to production and list pods
kprod && kgp

# Switch to staging and list services
kstaging && kgs

# Reset to default kubeconfig
kreset
```

### Multi-Cluster Scripts

```bash
#!/bin/bash
# check-all-clusters.sh

clusters=("prod-app" "staging-app" "dev-cluster")

for cluster in "${clusters[@]}"; do
  echo "=== Checking $cluster ==="
  eval $(hyve use $cluster)
  kubectl get nodes
  kubectl get pods -A | grep -v Running | grep -v Completed
  echo ""
done

# Reset
unset KUBECONFIG
```

### Backup Strategy

```bash
#!/bin/bash
# backup-kubeconfigs.sh

# Backup directory
BACKUP_DIR="$HOME/kubeconfig-backups/$(date +%Y%m%d)"
mkdir -p "$BACKUP_DIR"

# Get list of clusters
clusters=$(hyve cluster list | grep "^\s\s" | awk '{print $1}')

# Save each kubeconfig
for cluster in $clusters; do
  echo "Backing up $cluster..."
  hyve kubeconfig get "$cluster" > "$BACKUP_DIR/$cluster.yaml"
done

echo "Backup completed to $BACKUP_DIR"
```

### Context Switching with Namespace

```bash
#!/bin/bash
# use-cluster-ns.sh
# Usage: ./use-cluster-ns.sh prod-app production

CLUSTER=$1
NAMESPACE=$2

eval $(hyve use $CLUSTER)
kubectl config set-context --current --namespace=$NAMESPACE

echo "✅ Using cluster: $CLUSTER, namespace: $NAMESPACE"
kubectl get pods
```

## Overview

Hyve supports four cloud providers: **AWS**, **GCP**, **Azure**, and **Civo**. Authentication works differently depending on the context:

- **Local development** — use native CLI tools or stored tokens. AWS, GCP, and Azure rely on their official CLIs so credentials are managed in one place.
- **CI/CD pipelines** — use environment variables. Hyve supports a **named account pattern** so a single pipeline can reconcile clusters across multiple accounts, projects, or subscriptions.

## Local Development Authentication

### AWS

Hyve uses the standard AWS credential chain — no credentials are stored in Hyve.

```bash
# Configure default profile
aws configure

# Or configure a named profile
aws configure --profile production
export AWS_PROFILE=production
```

Alternative — environment variables:
```bash
export AWS_ACCESS_KEY_ID=your-access-key
export AWS_SECRET_ACCESS_KEY=your-secret-key
export AWS_REGION=us-east-1
```

```bash
# Verify
aws sts get-caller-identity
```

### GCP

Hyve uses Application Default Credentials (ADC) — no credentials are stored in Hyve.

```bash
# User credentials (recommended for local dev)
gcloud auth application-default login

# Service account (for automation)
export GOOGLE_APPLICATION_CREDENTIALS=/path/to/service-account.json
```

Hyve also needs to know which GCP project to use. Set one of these environment variables:

```bash
export GCP_PROJECT_ID=my-gcp-project-123
# or the standard Google Cloud variable
export GOOGLE_CLOUD_PROJECT=my-gcp-project-123
```

<Warning>
  If neither variable is set when running `hyve cluster` commands directly, Hyve will return:
  `GCP project ID not found. Please set GCP_PROJECT_ID or GOOGLE_CLOUD_PROJECT environment variable`

  If ADC is not configured, Hyve will return:
  `GCP authentication failed. Please run 'gcloud auth application-default login'`
</Warning>

**Windows:** `gcloud auth application-default login` stores credentials at `%APPDATA%\gcloud\application_default_credentials.json`. The Google SDK picks this up automatically — no extra configuration needed.

### Azure

Hyve uses Azure CLI authentication — no credentials are stored in Hyve.

```bash
az login

# If you have multiple subscriptions, set the active one
az account set --subscription "Your Subscription Name"
```

### Civo

<Tabs>
  <Tab title="Encrypted storage (recommended)">
    ```bash
    # Interactive — token input is hidden
    hyve config set-token civo --org my-org

    # Non-interactive
    hyve config set-token civo --org my-org --token YOUR_CIVO_TOKEN

    # Multiple organizations
    hyve config set-token civo --org production
    hyve config set-token civo --org development
    ```

    Tokens are encrypted with AES-256-GCM and stored in `~/.hyve/credentials.db`. The `--org` flag is required and identifies which organization the token belongs to.
  </Tab>

  <Tab title="Environment variable">
    ```bash
    export CIVO_TOKEN=your-civo-token
    ```

    If no stored token is found, Hyve falls back to `CIVO_TOKEN`.
  </Tab>
</Tabs>

```bash
# Verify
hyve config get-token civo --org my-org
hyve config list-tokens
```

## CI/CD Authentication

For pipelines, supply credentials via environment variables. Hyve resolves them in the following order for each provider:

1. **Named account env var** — `{ACCOUNT_NAME}_{PROVIDER}_{CREDENTIAL}` (for multi-account setups)
2. **Standard env var** — `CIVO_TOKEN`, `AWS_ACCESS_KEY_ID`, `GOOGLE_APPLICATION_CREDENTIALS`, etc.
3. **Provider SDK default chain** — AWS credential chain, GCP ADC, `az login` session

### Standard Environment Variables

Use these when managing a single account per provider:

| Variable | Provider | Description |
|----------|----------|-------------|
| `AWS_ACCESS_KEY_ID` | AWS | Access key ID |
| `AWS_SECRET_ACCESS_KEY` | AWS | Secret access key |
| `AWS_SESSION_TOKEN` | AWS | Session token (for temporary credentials) |
| `GOOGLE_APPLICATION_CREDENTIALS` | GCP | Path to service account JSON file |
| `AZURE_TENANT_ID` | Azure | Azure AD tenant ID |
| `AZURE_CLIENT_ID` | Azure | Service principal app ID |
| `AZURE_CLIENT_SECRET` | Azure | Service principal secret |
| `CIVO_TOKEN` | Civo | API token |
| `HYVE_GIT_TOKEN` | Git | Personal access token for state repository |

### Named Account Environment Variables

When a single pipeline manages **multiple accounts, GCP projects, or Azure subscriptions**, use named env vars so Hyve can route the correct credential to each cluster. The naming pattern is:

```
{ACCOUNT_NAME}_{PROVIDER}_{CREDENTIAL}
```

The account name comes from the alias set in the cluster definition (e.g., `awsAccount: prod-us`). Hyphens, dots, and spaces become underscores; everything is uppercased.

<Info>
  **GCP project IDs and Azure subscription IDs are never required as secrets.** Hyve resolves them automatically from `provider-configs/gcp.yaml` and `provider-configs/azure.yaml` using the name alias.
</Info>

#### AWS

| Cluster spec field | Example value | Named env vars |
|--------------------|---------------|----------------|
| `awsAccount` | `prod-us` | `PROD_US_AWS_ACCESS_KEY_ID` |
| | | `PROD_US_AWS_SECRET_ACCESS_KEY` |
| | | `PROD_US_AWS_SESSION_TOKEN` (optional) |

#### GCP

| Cluster spec field | Example value | Named env var |
|--------------------|---------------|---------------|
| `gcpProject` | `my-project` | `MY_PROJECT_GCP_CREDENTIALS_JSON` |

The GCP project ID is resolved from `provider-configs/gcp.yaml` — no project ID secret needed.

#### Azure

| Cluster spec field | Example value | Named env vars |
|--------------------|---------------|----------------|
| `azureSubscription` | `prod-sub` | `PROD_SUB_AZURE_TENANT_ID` |
| | | `PROD_SUB_AZURE_CLIENT_ID` |
| | | `PROD_SUB_AZURE_CLIENT_SECRET` |
| | | `PROD_SUB_AZURE_RESOURCE_GROUP` |

The Azure subscription ID is resolved from `provider-configs/azure.yaml` — no subscription ID secret needed.

#### Civo

| Cluster spec field | Example value | Named env var |
|--------------------|---------------|---------------|
| `civoOrganization` | `my-org` | `MY_ORG_CIVO_TOKEN` |

### Multi-Account Example

**`provider-configs/aws.yaml` (committed to state repository):**

```yaml
accounts:
  - name: prod-us
    account_id: "111122223333"
  - name: prod-eu
    account_id: "444455556666"
```

**Cluster definitions:**

```yaml
# clusters/prod-us.yaml
spec:
  provider: aws
  awsAccount: prod-us   # → PROD_US_AWS_ACCESS_KEY_ID, etc.
  region: us-east-1
```

```yaml
# clusters/prod-eu.yaml
spec:
  provider: aws
  awsAccount: prod-eu   # → PROD_EU_AWS_ACCESS_KEY_ID, etc.
  region: eu-west-1
```

**GitHub Actions secrets (credentials only — no account IDs needed):**

```
PROD_US_AWS_ACCESS_KEY_ID     = AKIA...
PROD_US_AWS_SECRET_ACCESS_KEY = ...
PROD_EU_AWS_ACCESS_KEY_ID     = AKIA...
PROD_EU_AWS_SECRET_ACCESS_KEY = ...
```

See the [CI/CD Workflow Guide](/guides/cicd) for complete GitHub Actions examples.

## Git Credentials

Configure credentials for accessing your Hyve state repository.

### Storing Credentials

Hyve delegates all git operations to the system `git` binary. Authentication is handled by git itself — configure it the same way you would for any other git workflow.

<Tabs>
  <Tab title="Environment variable (recommended for CI/CD)">
    ```bash
    export HYVE_GIT_TOKEN=ghp_xxxxxxxxxxxx
    ```

    Hyve reads `HYVE_GIT_TOKEN` and passes it to git when cloning and pushing.
  </Tab>

  <Tab title="Git Credential Manager">
    ```bash
    # macOS — Keychain (usually preconfigured)
    git config --global credential.helper osxkeychain

    # Windows — Git Credential Manager (included with Git for Windows)
    git config --global credential.helper manager

    # Linux
    git config --global credential.helper store
    ```

    Once configured, git will prompt for credentials on first use and cache them automatically.
  </Tab>

  <Tab title="SSH Keys">
    Use SSH URLs instead of HTTPS:

    ```bash
    hyve git add production --repo-url git@github.com:company/hyve-prod.git
    ```

    Ensure your SSH key is added to your git host and loaded into `ssh-agent`:

    ```bash
    ssh-add ~/.ssh/id_ed25519
    ```
  </Tab>
</Tabs>

<Info>
  For GitHub, use a Personal Access Token (PAT) with `repo` scope when using HTTPS authentication.
</Info>

## Required Permissions for Cluster Creation

Each cloud provider has its own IAM model. The sections below cover what is needed to create and manage clusters, with both a lean testing setup and a fine-grained production configuration.

---

### GCP GKE

GKE IAM operates at two layers: the **identity creating the cluster** (service account or human user) and the **node service account** (the GCP identity running on each VM). These are separate and should use separate accounts.

#### Predefined Role Reference

| Role | What it grants |
|------|---------------|
| `roles/container.admin` | Full cluster management + Kubernetes API object access |
| `roles/container.clusterAdmin` | Cluster lifecycle only — no Kubernetes API object access |
| `roles/iam.serviceAccountUser` | Required to assign a service account to node VMs |
| `roles/compute.networkViewer` | Read-only access to VPC/subnet resources during cluster creation |
| `roles/compute.networkUser` | Required when using a shared VPC (host project network) |

#### Simple Setup

Assign these three roles to the service account that creates clusters:

```bash
# Create the service account
gcloud iam service-accounts create hyve-reconciler \
  --display-name "Hyve Reconciler"

SA="hyve-reconciler@PROJECT_ID.iam.gserviceaccount.com"

# Cluster lifecycle management
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:${SA}" \
  --role="roles/container.clusterAdmin"

# Required to assign service accounts to node VMs
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:${SA}" \
  --role="roles/iam.serviceAccountUser"

# Required to read VPC/subnet info during cluster creation
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:${SA}" \
  --role="roles/compute.networkViewer"

# Create and download key for CI/CD
gcloud iam service-accounts keys create key.json \
  --iam-account="${SA}"
```

Store the JSON content as `MY_PROJECT_GCP_CREDENTIALS_JSON` in your CI/CD secrets.

<Warning>
  `roles/container.admin` grants both cluster management AND Kubernetes API object access (pods, deployments, secrets). Use `roles/container.clusterAdmin` for operators who only provision clusters.
</Warning>

#### Production: Fine-Grained Custom Role

Instead of predefined roles, create a custom IAM role with only the permissions Hyve needs:

```bash
# Save as gke-cluster-creator-role.yaml
cat > gke-cluster-creator-role.yaml << 'EOF'
title: "GKE Cluster Creator"
description: "Minimum permissions to create and manage GKE clusters"
stage: "GA"
includedPermissions:
  # Core cluster CRUD
  - container.clusters.create
  - container.clusters.delete
  - container.clusters.get
  - container.clusters.list
  - container.clusters.update
  # GKE cluster operations are async — polling is required
  - container.operations.get
  - container.operations.list
  # VPC/subnet reads during node placement
  - compute.networks.get
  - compute.networks.list
  - compute.subnetworks.get
  - compute.subnetworks.list
  - compute.subnetworks.use
  # Validate node pool configuration
  - compute.zones.list
  - compute.regions.list
  - compute.machineTypes.list
  - compute.diskTypes.list
  # Required to assign service account to node VMs
  - iam.serviceAccounts.actAs
  - iam.serviceAccounts.get
  - iam.serviceAccounts.list
  # Project metadata needed during provisioning
  - resourcemanager.projects.get
EOF

gcloud iam roles create GKEClusterCreator \
  --project=PROJECT_ID \
  --file=gke-cluster-creator-role.yaml

# Assign the custom role
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:hyve-reconciler@PROJECT_ID.iam.gserviceaccount.com" \
  --role="projects/PROJECT_ID/roles/GKEClusterCreator"
```

#### Node Service Account (Separate from the Cluster Creator)

The node service account runs on each VM. **Do not use the Compute Engine default service account** — it has far more permissions than nodes need.

```bash
gcloud iam service-accounts create gke-nodes \
  --display-name "GKE Node Service Account"

NODE_SA="gke-nodes@PROJECT_ID.iam.gserviceaccount.com"

# Minimum roles for nodes
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:${NODE_SA}" \
  --role="roles/logging.logWriter"

gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:${NODE_SA}" \
  --role="roles/monitoring.metricWriter"

gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:${NODE_SA}" \
  --role="roles/monitoring.viewer"

gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:${NODE_SA}" \
  --role="roles/stackdriver.resourceMetadata.writer"

# Only needed if pulling images from GCR/Artifact Registry in the same project
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:${NODE_SA}" \
  --role="roles/storage.objectViewer"
```

For Workload Identity (recommended — no long-lived keys):

```yaml
# GitHub Actions
- uses: google-github-actions/auth@v2
  with:
    workload_identity_provider: projects/123/locations/global/workloadIdentityPools/hyve/providers/github
    service_account: hyve-reconciler@my-project.iam.gserviceaccount.com
```

### AWS EKS

EKS requires **three separate IAM roles**, each with a distinct trust policy. The cluster role is assumed by the EKS service itself, not by humans or CI/CD.

#### Role 1: EKS Cluster Role (assumed by the EKS control plane)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "eks.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Attach the AWS managed policy — this is the minimum required for the EKS service to function:

```bash
aws iam create-role \
  --role-name hyve-eks-cluster-role \
  --assume-role-policy-document file://eks-trust-policy.json

aws iam attach-role-policy \
  --role-name hyve-eks-cluster-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonEKSClusterPolicy
```

#### Role 2: Node Group Role (assumed by EC2 instances)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

```bash
aws iam create-role \
  --role-name hyve-eks-node-role \
  --assume-role-policy-document file://ec2-trust-policy.json

# Required: worker node permissions + image pulls
aws iam attach-role-policy \
  --role-name hyve-eks-node-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonEKSWorkerNodePolicy

aws iam attach-role-policy \
  --role-name hyve-eks-node-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryPullOnly

# Required if not using IRSA/Pod Identity for the VPC CNI plugin
aws iam attach-role-policy \
  --role-name hyve-eks-node-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonEKS_CNI_Policy
```

<Info>
  The cluster role and node role must be different IAM roles. AWS enforces this. The node role is assumed by `ec2.amazonaws.com`, not `eks.amazonaws.com`.
</Info>

#### Role 3: Operator/CI-CD Role (assumed by Hyve or your pipeline)

Register these roles with Hyve after creation:

```bash
hyve config aws eks-role add --account prod-us --name cluster-role \
  --role-arn arn:aws:iam::111122223333:role/hyve-eks-cluster-role

hyve config aws node-role add --account prod-us --name node-role \
  --role-arn arn:aws:iam::111122223333:role/hyve-eks-node-role
```

##### Simple Operator Policy

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EKSCreate",
      "Effect": "Allow",
      "Action": "eks:CreateCluster",
      "Resource": "arn:aws:eks:*:111122223333:cluster/*"
    },
    {
      "Sid": "PassClusterRole",
      "Effect": "Allow",
      "Action": "iam:PassRole",
      "Resource": "arn:aws:iam::111122223333:role/hyve-eks-cluster-role",
      "Condition": {
        "StringEquals": { "iam:PassedToService": "eks.amazonaws.com" }
      }
    },
    {
      "Sid": "EKSServiceLinkedRole",
      "Effect": "Allow",
      "Action": "iam:CreateServiceLinkedRole",
      "Resource": "arn:aws:iam::111122223333:role/aws-service-role/eks.amazonaws.com/*",
      "Condition": {
        "ForAnyValue:StringEquals": { "iam:AWSServiceName": "eks" }
      }
    }
  ]
}
```

##### Production: Fine-Grained Operator Policy

```json
{
```

### Azure AKS

Azure RBAC operates at two levels: the **ARM (Azure Resource Manager) level** for managing the cluster resource itself, and the **Kubernetes API level** for workloads inside the cluster. Cluster creation only requires ARM-level permissions.

#### Why Not `Contributor` on the Subscription?

The built-in `Contributor` role grants `*` on all resource types in scope — it would allow the identity to create VMs, databases, storage accounts, and more. Even scoped to a resource group, it is broader than necessary.

#### Simple Setup

The built-in **Azure Kubernetes Service Contributor Role** covers core AKS CRUD operations scoped to the resource group:

```bash
# Create a service principal
az ad sp create-for-rbac --name hyve-reconciler --skip-assignment

SP_APP_ID="<appId from output>"
SUBSCRIPTION_ID="<your-subscription-id>"
RESOURCE_GROUP="<your-resource-group>"

# Assign the AKS-scoped built-in role (not subscription Contributor)
az role assignment create \
  --assignee "${SP_APP_ID}" \
  --role "Azure Kubernetes Service Contributor Role" \
  --scope "/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/${RESOURCE_GROUP}"
```

Store the output values as `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET` in your CI/CD secrets.

For custom VNet setups, also assign on the subnet:

```bash
SUBNET_ID="/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/networking-rg/providers/Microsoft.Network/virtualNetworks/my-vnet/subnets/aks-subnet"

az role assignment create \
  --assignee "${SP_APP_ID}" \
  --role "Network Contributor" \
  --scope "${SUBNET_ID}"
```

#### Production: Fine-Grained Custom Role

```bash
cat > aks-cluster-creator-role.json << 'EOF'
{
  "Name": "AKS Cluster Creator",
  "Description": "Minimum permissions to create and manage AKS clusters",
  "Actions": [
    "Microsoft.ContainerService/managedClusters/read",
    "Microsoft.ContainerService/managedClusters/write",
    "Microsoft.ContainerService/managedClusters/delete",
    "Microsoft.ContainerService/managedClusters/agentPools/read",
    "Microsoft.ContainerService/managedClusters/agentPools/write",
    "Microsoft.ContainerService/managedClusters/agentPools/delete",
    "Microsoft.ContainerService/managedClusters/listClusterUserCredential/action",
    "Microsoft.ContainerService/managedClusters/listClusterAdminCredential/action",
    "Microsoft.Resources/subscriptions/resourceGroups/read",
    "Microsoft.Resources/deployments/read",
    "Microsoft.Resources/deployments/write",
    "Microsoft.Network/virtualNetworks/subnets/read",
    "Microsoft.Network/virtualNetworks/subnets/join/action",
    "Microsoft.Network/publicIPAddresses/read",
    "Microsoft.Network/publicIPAddresses/write",
    "Microsoft.ManagedIdentity/userAssignedIdentities/assign/action",
    "Microsoft.ManagedIdentity/userAssignedIdentities/read",
    "Microsoft.Authorization/roleAssignments/read",
    "Microsoft.OperationalInsights/workspaces/read"
  ],
  "NotActions": [],
  "DataActions": [],
  "NotDataActions": [],
  "AssignableScopes": [
    "/subscriptions/<SUBSCRIPTION_ID>/resourceGroups/<RESOURCE_GROUP>"
  ]
}
EOF

az role definition create --role-definition aks-cluster-creator-role.json

az role assignment create \
  --assignee "${SP_APP_ID}" \
  --role "AKS Cluster Creator" \
  --scope "/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/${RESOURCE_GROUP}"
```

**What each permission group covers:**

| Permission group | Why it's needed |
|-----------------|-----------------|
| `Microsoft.ContainerService/managedClusters/*` | Core cluster and node pool CRUD |
| `listCluster*Credential/action` | Retrieve kubeconfig after creation |
| `Microsoft.Resources/*` | Resource group and ARM deployment operations |
| `Microsoft.Network/virtualNetworks/subnets/join/action` | Required when AKS attaches nodes to your subnet |
| `Microsoft.Network/publicIPAddresses/*` | Required for outbound load balancer IPs |
| `Microsoft.ManagedIdentity/userAssignedIdentities/assign/action` | Required when using a pre-created user-assigned identity |
| `Microsoft.OperationalInsights/workspaces/read` | Required only when enabling Container Insights |

<Info>
  AKS automatically assigns `Contributor` on the auto-created `MC_*` node resource group to the cluster's managed identity. You do not need to configure this manually.
</Info>

#### Built-in Role Clarification

| Built-in Role | What it's for |
|--------------|---------------|
| `Azure Kubernetes Service Contributor Role` | Creating/managing AKS clusters (ARM level) |
| `Azure Kubernetes Service Cluster User Role` | Retrieving user kubeconfig — not for cluster creation |
| `Azure Kubernetes Service Cluster Admin Role` | Retrieving admin kubeconfig — not for cluster creation |
| `Contributor` | Too broad — avoid unless strictly necessary |

### Civo

Civo uses team-based RBAC for organizations. The permission model is much simpler than the major cloud providers — there are no separate "cluster roles" or "node roles" because Civo manages the underlying infrastructure entirely.

#### Available Kubernetes Permissions

| Permission | What it allows |
|-----------|---------------|
| `kubernetes:create` | Create new clusters |
| `kubernetes:read` | View cluster details and kubeconfig |
| `kubernetes:update` | Modify cluster configuration |
| `kubernetes:delete` | Delete clusters |

#### Minimum Permissions for Hyve

For a service account or user whose only responsibility is provisioning clusters:

- `kubernetes:create`
- `kubernetes:read`
- `kubernetes:update`

Configure custom roles through the Civo dashboard: **Organisation → Team → Roles → Create Role**.

No networking, compute, or separate IAM role configuration is required — Civo handles all underlying infrastructure.

## Creating Cloud Provider Credentials

### AWS IAM User / Role

For CI/CD, create a dedicated IAM user or role with the minimum permissions required for EKS operations, then store the access key as a secret.

For OIDC (recommended — no long-lived secrets):

```yaml
# GitHub Actions
permissions:
  id-token: write
  contents: read

steps:
  - uses: aws-actions/configure-aws-credentials@v4
    with:
      role-to-assume: arn:aws:iam::123456789012:role/hyve-reconciler
      aws-region: us-east-1
```

### GCP Service Account

```bash
# Create service account
gcloud iam service-accounts create hyve-reconciler \
  --display-name "Hyve Reconciler"

# Grant GKE permissions (see "Required Permissions" above for fine-grained options)
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:hyve-reconciler@PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/container.clusterAdmin"

gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:hyve-reconciler@PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/iam.serviceAccountUser"

gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:hyve-reconciler@PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/compute.networkViewer"

# Create and download key
gcloud iam service-accounts keys create key.json \
  --iam-account=hyve-reconciler@PROJECT_ID.iam.gserviceaccount.com
```

Store the JSON content as `MY_PROJECT_GCP_CREDENTIALS_JSON` in your CI/CD secrets.

For Workload Identity (recommended — no long-lived secrets):

```yaml
# GitHub Actions
- uses: google-github-actions/auth@v2
  with:
    workload_identity_provider: projects/123/locations/global/workloadIdentityPools/hyve/providers/github
    service_account: hyve-reconciler@my-project.iam.gserviceaccount.com
```

### Azure Service Principal

```bash
# Create service principal — assign the AKS-scoped role, not subscription Contributor
az ad sp create-for-rbac \
  --name hyve-reconciler \
  --role "Azure Kubernetes Service Contributor Role" \
  --scopes /subscriptions/SUBSCRIPTION_ID/resourceGroups/RESOURCE_GROUP
```

Store the output values as `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET` in your CI/CD secrets.

## Storage Locations

```
~/.hyve/
├── credentials.db      # Civo tokens and Git credentials (AES-256-GCM encrypted)
├── kubeconfigs.db     # Cluster kubeconfigs (encrypted)
├── repositories.db    # Repository configurations (not encrypted)
└── temp/              # Temporary files (auto-cleaned)
```

AWS, GCP, and Azure credentials are **not** stored in Hyve — they use their native credential systems.

<Warning>
  Never commit files from `~/.hyve/` to version control. The `.gitignore` should exclude this directory.
</Warning>

## Security Best Practices

<AccordionGroup>
  <Accordion title="Use OIDC for AWS and GCP">
    Instead of storing long-lived access keys as secrets, use GitHub Actions' OIDC provider with AWS IAM roles or GCP Workload Identity Federation. This eliminates secrets rotation entirely.
  </Accordion>

  <Accordion title="Use interactive token entry for Civo">
    ```bash
    # Good — token not visible in shell history
    hyve config set-token civo --org my-org

    # Avoid — token visible in history
    hyve config set-token civo --org my-org --token abc123...
    ```
  </Accordion>

  <Accordion title="Scope credentials to minimum permissions">
    Follow the least-privilege configs in the [Required Permissions for Cluster Creation](#required-permissions-for-cluster-creation) section above. In summary:

    - **Civo**: Assign a custom role with only `kubernetes:create`, `kubernetes:read`, `kubernetes:update`
    - **AWS**: Use three separate IAM roles — one for EKS control plane (`eks.amazonaws.com`), one for EC2 nodes, one for your operator/CI-CD identity. Always scope `iam:PassRole` with a `Condition` on `iam:PassedToService`.
    - **GCP**: Use `roles/container.clusterAdmin` (not `roles/container.admin`) on the specific project. Use a separate minimal node service account rather than the Compute Engine default.
    - **Azure**: Assign `Azure Kubernetes Service Contributor Role` scoped to the resource group — not `Contributor` on the subscription.
  </Accordion>

  <Accordion title="Rotate tokens regularly">
    ```bash
    # Update Civo token
    hyve config set-token civo --org my-org   # enter new token

    # Update Git credentials
    hyve git credentials --password new_token
    ```
  </Accordion>

  <Accordion title="Protect shell history">
    ```bash
    # Clear a specific history entry if a token was exposed
    history -d <line_number>

    # Or clear all history
    history -c
    ```
  </Accordion>

  <Accordion title="Never commit credentials">
    ```bash
    # .gitignore — add to your state repository
    .env
    .env.*
    !.env.example
    *.key
    *.pem
    service-account.json
    ```
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Civo: authentication failed">
    ```bash
    # 1. Verify token is stored
    hyve config list-tokens
    hyve config get-token civo --org my-org

    # 2. Re-enter token
    hyve config set-token civo --org my-org

    # 3. Test with environment variable directly
    export CIVO_TOKEN=your_token
    hyve cluster list
    ```
  </Accordion>

  <Accordion title="AWS: credentials not found">
    ```bash
    # Verify CLI is configured
    aws sts get-caller-identity

    # Check environment variables
    echo $AWS_ACCESS_KEY_ID

    # Verify named account env var (if using multi-account)
    echo $PROD_US_AWS_ACCESS_KEY_ID
    ```
  </Accordion>

  <Accordion title="GCP: application default credentials not found">
    ```bash
    # Re-authenticate
    gcloud auth application-default login

    # Or set service account path
    export GOOGLE_APPLICATION_CREDENTIALS=/path/to/key.json

    # Verify
    gcloud auth application-default print-access-token
    ```
  </Accordion>

  <Accordion title="Azure: not logged in">
    ```bash
    az login

    # Verify
    az account show

    # For service principal
    export AZURE_TENANT_ID=...
    export AZURE_CLIENT_ID=...
    export AZURE_CLIENT_SECRET=...
    ```
  </Accordion>

  <Accordion title="Git operations fail">
    ```bash
    # Check stored credentials
    hyve git credentials

    # Update credentials
    hyve git credentials --username user --password new_token

    # Test with environment variable
    export HYVE_GIT_TOKEN=your_token
    hyve git pull
    ```
  </Accordion>

  <Accordion title="Named account env var not picked up">
    Verify the variable name matches the normalized alias exactly. Hyphens and dots in the account name become underscores, and everything is uppercased.

    ```bash
    # Cluster spec has: awsAccount: prod-us
    # Correct env var:
    export PROD_US_AWS_ACCESS_KEY_ID=AKIA...

    # Common mistakes:
    # PROD-US_AWS_ACCESS_KEY_ID  (hyphen — wrong)
    # prod_us_aws_access_key_id  (lowercase — wrong)
    ```
  </Accordion>

  <Accordion title="Credentials database corrupted">
    ```bash
    # Backup current database
    cp ~/.hyve/credentials.db ~/.hyve/credentials.db.backup

    # Remove corrupted database
    rm ~/.hyve/credentials.db

    # Re-enter credentials
    hyve config set-token civo --org my-org
    hyve git credentials --username user --password token
    ```
  </Accordion>
</AccordionGroup>

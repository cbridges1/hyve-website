## Overview

Hyve requires configuration for cloud provider authentication, Git credentials, and provider-specific settings. The authentication model varies by provider:

- **Civo**: API tokens stored encrypted in Hyve (AES-GCM)
- **AWS, GCP, Azure**: Native CLI authentication (no credentials stored in Hyve)

## Storage Location

All configuration data is stored in the `~/.hyve/` directory:

```
~/.hyve/
├── config.yaml         # Global configuration (git backend preference)
├── credentials.db      # Encrypted Civo tokens and Git credentials (AES-GCM)
├── kubeconfigs.db     # Encrypted cluster kubeconfigs
├── repositories.db    # Repository configurations (SQLite)
├── temp/              # Temporary kubeconfig files
└── repositories/      # Cloned repository storage
    └── <repo-name>/
        └── provider-configs/  # Provider-specific configuration
            ├── gcp.yaml       # GCP projects (name/ID aliases)
            ├── aws.yaml       # AWS accounts, EKS roles, VPCs
            └── azure.yaml     # Azure subscription IDs
```

<Note>
  Civo credentials are encrypted using AES-GCM. AWS, GCP, and Azure use their native CLI authentication instead of storing credentials in Hyve.
</Note>

## Git Backend Selection

Hyve supports two Git backends for repository operations. The preference is persisted in the database and can be changed at any time.

<CardGroup cols={2}>
  <Card title="System Git (Default)" icon="terminal" color="#10b981">
    Uses your system's installed git command

    **Advantages:**
    - No additional setup required
    - Works with existing git configuration
    - Better authentication compatibility
    - Familiar behavior

    **Requirement:** Git must be in PATH
  </Card>

  <Card title="Built-in Git" icon="package" color="#f97316">
    Uses embedded go-git library

    **Advantages:**
    - Works without system git
    - Portable across environments
  </Card>
</CardGroup>

### Setting Git Backend

```bash
# Set to built-in git (persisted in database)
hyve config set-git-backend builtin

# Set to system git (default)
hyve config set-git-backend system

# Check current backend
hyve config get-git-backend
```

<Info>
  **Recommended:** Keep the default system git backend for easier onboarding and better compatibility. The built-in backend is available for specialized environments where git cannot be installed.
</Info>

### Configuration Priority

The git backend is determined by this priority order:

1. **Config file** (`~/.hyve/config.yaml`) - Set via `hyve config set-git-backend`
2. **Environment variable** - `GIT_BACKEND=builtin` (temporary override)
3. **Default** - `system` (uses system git)

### Why Native CLI Authentication?

For AWS, GCP, and Azure, Hyve uses native cloud CLI authentication instead of storing credentials. This provides:

- **Security**: Credentials managed by official cloud CLIs with their security features
- **Consistency**: Same credentials used by other tools (Terraform, kubectl, etc.)
- **SSO Support**: Works with corporate SSO, MFA, and identity federation
- **No Duplication**: No need to manage credentials in multiple places

Civo is the exception because it doesn't have a widely-used CLI, so Hyve stores Civo API tokens securely using AES-GCM encryption.

### Civo API Token

<Steps>
  <Step title="Get your API token">
    1. Log in to [Civo Dashboard](https://dashboard.civo.com/)
    2. Navigate to Settings → Security → API Keys
    3. Create or copy your API token
  </Step>

  <Step title="Store the token">
    ```bash
    # Interactive mode (recommended)
    hyve config set-token civo
    # Enter token when prompted
    ```

    Or provide directly:

    ```bash
    hyve config set-token civo --token YOUR_CIVO_TOKEN
    ```

    For multiple Civo accounts:

    ```bash
    hyve config set-token civo --account production
    hyve config set-token civo --account development
    ```
  </Step>

  <Step title="Verify the token">
    ```bash
    # View stored token
    hyve config get-token civo

    # View token for specific account
    hyve config get-token civo --account production

    # List all Civo accounts
    hyve config list-tokens
    ```
  </Step>
</Steps>

### AWS Credentials

AWS uses native CLI authentication. Hyve does not store AWS credentials.

<Steps>
  <Step title="Configure AWS CLI">
    ```bash
    # Configure default profile
    aws configure

    # Or configure named profile
    aws configure --profile production
    ```
  </Step>

  <Step title="Alternative: Environment Variables">
    ```bash
    export AWS_ACCESS_KEY_ID=your-access-key
    export AWS_SECRET_ACCESS_KEY=your-secret-key
    export AWS_REGION=us-east-1
    ```
  </Step>

  <Step title="Verify authentication">
    ```bash
    aws sts get-caller-identity
    ```
  </Step>
</Steps>

### GCP Credentials

GCP uses Application Default Credentials. Hyve does not store GCP credentials.

<Steps>
  <Step title="Authenticate with gcloud">
    ```bash
    # For user authentication
    gcloud auth application-default login

    # For service account (CI/CD)
    export GOOGLE_APPLICATION_CREDENTIALS=/path/to/service-account.json
    ```
  </Step>

  <Step title="Configure GCP projects in Hyve">
    GCP projects are configured with aliases in your repository:

    ```bash
    # Add project with alias
    hyve config gcp add-project --name dev --id my-dev-project-123
    hyve config gcp add-project --name prod --id my-prod-project-456

    # Use alias when creating clusters
    hyve cluster add my-cluster --provider gcp --gcp-project dev --region us-central1
    ```
  </Step>
</Steps>

### Azure Credentials

Azure uses Azure CLI authentication. Hyve does not store Azure credentials.

<Steps>
  <Step title="Authenticate with Azure CLI">
    ```bash
    az login
    ```
  </Step>

  <Step title="Set subscription (if multiple)">
    ```bash
    az account set --subscription "Your Subscription Name"
    ```
  </Step>
</Steps>

### Environment Variable Alternative

You can use environment variables for Civo tokens instead of storing them:

```bash
# Civo (alternative to hyve config set-token)
export CIVO_TOKEN=your-civo-token

# Run commands without storing tokens
hyve cluster list
```

<Warning>
  Environment variables are not encrypted and may be visible in process listings. Use the encrypted storage (`hyve config set-token civo`) for production.
</Warning>

## Provider Configuration

Provider-specific configurations (like GCP project IDs and AWS account IDs) are stored in your Git repository under `provider-configs/`. This allows team sharing and version control.

### GCP Configuration

```bash
# Add GCP projects with aliases
hyve config gcp add-project --name dev --id my-dev-project-123
hyve config gcp add-project --name prod --id my-prod-project-456

# List configured projects
hyve config gcp list-projects

# Get project ID by alias
hyve config gcp get-project dev

# Remove a project
hyve config gcp remove-project dev

# Use alias when creating clusters
hyve cluster add my-cluster --provider gcp --gcp-project dev --region us-central1
```

### AWS Configuration

```bash
# Account management (with aliases)
hyve config aws account-add --name prod --id 123456789012
hyve config aws account-list
hyve config aws account-get prod
hyve config aws account-remove prod

# EKS IAM Role management (configuration only)
hyve config aws eks-role-add --name default-role --role-arn arn:aws:iam::123456789012:role/my-eks-role
hyve config aws eks-role-list
hyve config aws eks-role-get default-role
hyve config aws eks-role-remove default-role

# VPC management (configuration only)
hyve config aws vpc-add --name default-vpc --id vpc-0123456789abcdef0
hyve config aws vpc-list
hyve config aws vpc-get default-vpc
hyve config aws vpc-remove default-vpc
```

### AWS Resource Creation

Hyve can create actual AWS resources (IAM roles and VPCs) and store their aliases:

```bash
# Create EKS IAM role in AWS
hyve config aws eks-role-create --name default-role --role-name my-eks-cluster-role --region us-east-1

# Delete EKS IAM role from AWS
hyve config aws eks-role-delete default-role --region us-east-1

# Delete from config only (keep AWS resource)
hyve config aws eks-role-delete default-role --config-only

# Create VPC in AWS
hyve config aws vpc-create --name dev-vpc --region us-east-1 --cidr 10.0.0.0/16

# Create VPC with subnets
hyve config aws vpc-create --name dev-vpc --region us-east-1 --subnets 10.0.1.0/24,10.0.2.0/24

# Delete VPC from AWS
hyve config aws vpc-delete dev-vpc --region us-east-1
```

### Azure Configuration

```bash
# Add subscription IDs
hyve config azure add-subscription-ids sub-id-1,sub-id-2
hyve config azure list-subscription-ids
hyve config azure remove-subscription-ids sub-id-1
```

### Provider Config Files

Provider configurations are stored in YAML files in your repository:

**provider-configs/gcp.yaml:**
```yaml
projects:
  - name: dev
    project_id: my-dev-project-123
  - name: prod
    project_id: my-prod-project-456
```

**provider-configs/aws.yaml:**
```yaml
accounts:
  - name: prod
    account_id: "123456789012"
  - name: dev
    account_id: "987654321098"
eks_roles:
  - name: default-role
    role_arn: arn:aws:iam::123456789012:role/my-eks-role
vpcs:
  - name: default-vpc
    vpc_id: vpc-0123456789abcdef0
```

## Git Credentials

Configure Git credentials for private repositories.

### Global Git Credentials

Set credentials that apply to all repositories:

```bash
# Interactive mode
hyve git credentials --username your-username --password your_token

# Or provide directly
hyve git credentials \
  --username myusername \
  --password ghp_xxxxxxxxxxxx
```

### Environment Variables

Use environment variables for temporary access:

```bash
# GitHub Personal Access Token
export HYVE_GIT_TOKEN=ghp_xxxxxxxxxxxx

# Or use Git credential helpers
git config --global credential.helper store
```

### SSH Keys

For SSH-based authentication:

```bash
# Ensure SSH key is loaded
eval $(ssh-agent)
ssh-add ~/.ssh/id_rsa

# Add repository with SSH URL
hyve git add production --repo-url git@github.com:company/hyve-prod.git
```

## Environment Files

Create a `.env` file for environment-specific configuration:

```bash
# .env file in your working directory
CIVO_TOKEN=your-civo-token
DOCKER_TOKEN=your-docker-token
GITHUB_TOKEN=ghp_xxxxxxxxxxxx
DATADOG_API_KEY=your-datadog-key
```

Hyve will automatically load `.env` files when running workflows.

## Managing Civo Accounts

Hyve supports multiple Civo accounts. Each account has its own encrypted API token.

### List All Accounts

```bash
# List all stored Civo accounts
hyve config list-tokens
```

Output:
```
🔑 Stored Civo accounts (2):
  ✓ default
  ✓ production
```

### View Specific Token

```bash
# View token for default account
hyve config get-token civo

# View token for specific account
hyve config get-token civo --account production
```

### Update Token

```bash
# Update token for default account
hyve config set-token civo --token NEW_TOKEN

# Update token for specific account
hyve config set-token civo --account production --token NEW_TOKEN
```

### Remove Token

```bash
# Remove token for default account
hyve config clear-token civo

# Remove token for specific account
hyve config clear-token civo --account production
```

## Configuration Priority

Hyve uses the following priority order for configuration values:

<Steps>
  <Step title="1. Command-line flags">
    Highest priority - overrides all other sources

    ```bash
    hyve cluster add --region PHX1
    ```
  </Step>

  <Step title="2. Environment variables">
    Second priority - from shell or .env files

    ```bash
    export CIVO_TOKEN=token
    ```
  </Step>

  <Step title="3. Encrypted database">
    Third priority - from credentials.db

    ```bash
    hyve config set-token civo
    ```
  </Step>

  <Step title="4. Default values">
    Lowest priority - built-in defaults
  </Step>
</Steps>

## Security Best Practices

<AccordionGroup>
  <Accordion title="1. Use Encrypted Storage">
    Always prefer encrypted storage over environment variables:

    ```bash
    # Good - encrypted storage
    hyve config set-token civo

    # Avoid - visible in environment
    export CIVO_TOKEN=token
    ```
  </Accordion>

  <Accordion title="2. Rotate Tokens Regularly">
    Update tokens periodically for security:

    ```bash
    # Generate new token in Civo Dashboard
    # Update in Hyve
    hyve config set-token civo --token NEW_TOKEN
    ```
  </Accordion>

  <Accordion title="3. Use Repository-Specific Credentials">
    For different environments, use separate repositories with different credentials:

    ```bash
    # Production repository
    hyve git add production --repo-url https://github.com/company/hyve-prod.git

    # Development repository
    hyve git add development --repo-url https://github.com/company/hyve-dev.git
    ```
  </Accordion>

  <Accordion title="4. Never Commit Secrets">
    Add `.env` files to `.gitignore`:

    ```bash
    # .gitignore
    .env
    .env.*
    !.env.example
    ```
  </Accordion>

  <Accordion title="5. Backup Encryption Keys">
    The encryption key is stored separately from credentials. Back it up securely:

    ```bash
    # Backup encryption key
    cp ~/.hyve/encryption.key ~/backups/hyve-encryption.key
    ```
  </Accordion>
</AccordionGroup>

## Configuration Examples

### Complete Initial Setup

```bash
# 1. Configure cloud provider authentication

# For Civo (stored encrypted in Hyve)
hyve config set-token civo
# Enter token when prompted

# For AWS (uses native CLI)
aws configure

# For GCP (uses Application Default Credentials)
gcloud auth application-default login

# For Azure (uses Azure CLI)
az login

# 2. Configure Git credentials
hyve git credentials --username myuser --password ghp_xxxxxxxxxxxx

# 3. Add repository
hyve git add production --repo-url https://github.com/company/hyve-prod.git

# 4. Configure provider settings (stored in repository)
hyve config gcp add-project --name dev --id my-gcp-project-id
hyve config aws account-add --name prod --id 123456789012
```

### Multi-Account Civo Setup

```bash
# Production Civo account
hyve config set-token civo --account production

# Development Civo account
hyve config set-token civo --account development

# Use specific account when creating clusters
hyve cluster add prod-cluster --provider civo --civo-account production --region PHX1
```

### AWS with Resource Creation

```bash
# Create EKS IAM role
hyve config aws eks-role-create --name default-role --role-name hyve-eks-role --region us-east-1

# Create VPC with subnets
hyve config aws vpc-create --name eks-vpc --region us-east-1 --cidr 10.0.0.0/16 --subnets 10.0.1.0/24,10.0.2.0/24

# Use in cluster creation (future)
hyve cluster add my-eks-cluster --provider aws --eks-role default-role --vpc eks-vpc --region us-east-1
```

### Using .env Files

```bash
# Create .env file
cat > .env <<EOF
CIVO_TOKEN=your-civo-token
DOCKER_TOKEN=your-docker-token
GITHUB_TOKEN=ghp_xxxxxxxxxxxx
EOF

# Hyve automatically loads .env when running workflows
hyve workflow run deploy-app
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="Token not found">
    Error: `Token for provider 'civo' not found`

    **Solution:**
    ```bash
    # Set the token
    hyve config set-token civo --token YOUR_TOKEN

    # Or use environment variable
    export CIVO_TOKEN=YOUR_TOKEN
    ```
  </Accordion>

  <Accordion title="Git authentication fails">
    Error: `Failed to authenticate with Git repository`

    **Solution:**
    ```bash
    # Update Git credentials
    hyve git credentials --username your-username --password YOUR_PAT

    # Or use SSH
    eval $(ssh-agent)
    ssh-add ~/.ssh/id_rsa
    ```
  </Accordion>

  <Accordion title="Permission denied on credentials.db">
    Error: `Permission denied: ~/.hyve/credentials.db`

    **Solution:**
    ```bash
    # Fix permissions
    chmod 600 ~/.hyve/credentials.db
    ```
  </Accordion>

  <Accordion title="Encryption key not found">
    Error: `Encryption key not found`

    **Solution:**
    ```bash
    # Restore from backup
    cp ~/backups/hyve-encryption.key ~/.hyve/encryption.key
    chmod 600 ~/.hyve/encryption.key

    # Or initialize new key (will lose existing credentials)
    rm ~/.hyve/credentials.db
    hyve config set-token civo
    ```
  </Accordion>
</AccordionGroup>

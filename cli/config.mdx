The `hyve config` command group manages API tokens and configuration settings. Hyve uses different authentication methods depending on the cloud provider:

- **Civo**: API tokens stored encrypted in Hyve (supports multiple accounts)
- **AWS**: Native AWS CLI authentication (`aws configure`)
- **GCP**: Application Default Credentials (`gcloud auth application-default login`)
- **Azure**: Azure CLI authentication (`az login`)

## Overview

Civo is the only provider that requires storing credentials in Hyve. Other cloud providers use their native CLI authentication, which provides better integration with existing tooling, credential rotation, and security practices.

## Commands

### hyve config set-token

Store an encrypted Civo API token. Multiple accounts are supported using the `--account` flag.

```bash
hyve config set-token civo [flags]
```

<ParamField path="--token, -t" type="string">
  API token. If not provided, you'll be prompted securely (recommended).
</ParamField>

<ParamField path="--account, -a" type="string" default="default">
  Civo account ID. Allows storing tokens for multiple Civo accounts.
</ParamField>

**Examples:**

```bash
# Prompt for token (recommended - input hidden)
hyve config set-token civo

# Store token for a specific account
hyve config set-token civo --account production

# Provide token via flag (less secure)
hyve config set-token civo --token YOUR_TOKEN_HERE

# Store multiple accounts
hyve config set-token civo --account production
hyve config set-token civo --account development
hyve config set-token civo --account staging
```

**Output:**

```
Enter Civo API token for account 'production' (input will be hidden):
✅ Civo API token for account 'production' stored successfully

💡 The token is encrypted and stored in ~/.hyve/credentials.db
💡 Hyve will now use this token automatically for Civo operations
💡 To use this account, specify --civo-account production when creating clusters
```

<Info>
  Only Civo requires storing tokens in Hyve. For AWS, GCP, and Azure, use their native CLI tools.
</Info>

### hyve config get-token

Retrieve and display the stored Civo API token for an account.

```bash
hyve config get-token civo [flags]
```

<ParamField path="--account, -a" type="string" default="default">
  Civo account ID to retrieve token for.
</ParamField>

**Examples:**

```bash
# Get token for default account
hyve config get-token civo

# Get token for specific account
hyve config get-token civo --account production
```

**Output:**

```
🔑 Civo API token for account 'production':
abcdef1234567890abcdef1234567890abcdef1234567890
```

<Warning>
  This command displays the token in plaintext. Use with caution and avoid sharing terminal output.
</Warning>

## Cloud Provider Authentication

### Civo (Tokens stored in Hyve)

Civo is the only provider that stores credentials in Hyve's encrypted database.

**Getting a Civo API Token:**

1. Log in to [Civo Dashboard](https://dashboard.civo.com/)
2. Navigate to Settings → Security
3. Click "Generate API Key"
4. Copy the token and store it with Hyve:

```bash
hyve config set-token civo --account default
```

**Multiple Accounts:**

You can store tokens for multiple Civo accounts:

```bash
# Store production account
hyve config set-token civo --account production

# Store development account
hyve config set-token civo --account development

# List all accounts
hyve config list-tokens
```

---

### AWS (Native CLI Authentication)

AWS uses the AWS CLI's credential chain. No credentials are stored in Hyve.

**Setup:**

```bash
# Configure AWS CLI
aws configure

# Or set environment variables
export AWS_ACCESS_KEY_ID=your_access_key
export AWS_SECRET_ACCESS_KEY=your_secret_key
export AWS_DEFAULT_REGION=us-east-1
```

**Credential Sources (in order of priority):**

1. Environment variables (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`)
2. AWS credentials file (`~/.aws/credentials`)
3. AWS config file (`~/.aws/config`)
4. IAM instance profile (EC2, ECS, Lambda)

<Info>
  Hyve uses the AWS SDK's default credential chain. Any method that works with the AWS CLI will work with Hyve.
</Info>

---

### GCP (Application Default Credentials)

GCP uses Application Default Credentials (ADC). No credentials are stored in Hyve.

**Setup:**

```bash
# Authenticate with gcloud
gcloud auth application-default login
```

**Configure GCP Projects:**

GCP projects are configured with aliases in your repository:

```bash
# Add project with alias
hyve config gcp add-project --name dev --id my-dev-project-123
hyve config gcp add-project --name prod --id my-prod-project-456

# Use alias when creating clusters
hyve cluster add my-cluster --provider gcp --gcp-project dev --region us-central1
```

**Credential Sources (in order of priority):**

1. `GOOGLE_APPLICATION_CREDENTIALS` environment variable (path to service account JSON)
2. User credentials from `gcloud auth application-default login`
3. GCE metadata service (on Google Cloud instances)

<Info>
  GCP project IDs are configured using `hyve config gcp add-project` and stored in the repository for team sharing.
</Info>

---

### Azure (Azure CLI Authentication)

Azure uses the Azure CLI's authentication. No credentials are stored in Hyve.

**Setup:**

```bash
# Authenticate with Azure CLI
az login

# Set required environment variables
export AZURE_SUBSCRIPTION_ID=your-subscription-id
export AZURE_RESOURCE_GROUP=your-resource-group
```

**Credential Sources (in order of priority):**

1. Environment variables (`AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET`, `AZURE_TENANT_ID`)
2. Managed Identity (on Azure resources)
3. Azure CLI credentials (`az login`)
4. Azure PowerShell credentials

<Info>
  Both `AZURE_SUBSCRIPTION_ID` and `AZURE_RESOURCE_GROUP` must be set via environment variables.
</Info>

#### hyve config aws vpc-create

Create a VPC in AWS and store the alias.

```bash
hyve config aws vpc-create --name <alias> [--region <region>] [--cidr <cidr>] [--subnets <cidrs>] [--enable-dns]
```

<ParamField path="--name" type="string" required>
  Friendly name/alias for the VPC.
</ParamField>

<ParamField path="--region" type="string" default="us-east-1">
  AWS region.
</ParamField>

<ParamField path="--cidr" type="string" default="10.0.0.0/16">
  CIDR block for the VPC.
</ParamField>

<ParamField path="--subnets" type="string">
  Comma-separated subnet CIDRs to create (e.g., 10.0.1.0/24,10.0.2.0/24).
</ParamField>

<ParamField path="--enable-dns" type="boolean" default="true">
  Enable DNS support and hostnames.
</ParamField>

**Examples:**

hyve config aws vpc-create --name eks-vpc --region us-east-1 --subnets 10.0.1.0/24,10.0.2.0/24,10.0.3.0/24

## Security

### Encryption (Civo Tokens Only)

Civo API tokens are encrypted using AES-GCM (Galois/Counter Mode) before being stored:

- **Algorithm**: AES-256-GCM
- **Key Derivation**: Hardware-based entropy
- **Storage**: SQLite database at `~/.hyve/credentials.db`

### Storage Location

```
~/.hyve/
└── credentials.db      # Encrypted Civo tokens and Git credentials
```

### File Permissions

The credentials database is created with restrictive permissions:

```bash
# Check permissions
ls -l ~/.hyve/credentials.db
# Output: -rw-------  1 user  staff  12288 Jan 15 10:30 credentials.db
```

Only the owner can read and write the file (mode 600).

## Environment Variable Fallback

### Civo

```bash
# Set Civo token via environment variable
export CIVO_TOKEN=your_token_here

# Hyve will use this if no stored token found for "default" account
hyve cluster list
```

### Priority Order (Civo)

1. **Stored Token**: Token for the specified account (or "default")
2. **Environment Variable**: `CIVO_TOKEN` environment variable (for "default" account only)
3. **Error**: If neither is available, commands will fail

### AWS/GCP/Azure

These providers always use their native CLI authentication. Environment variables work as documented in their respective SDKs.

## CI/CD Pipelines

### Civo

```bash
# Use environment variable (recommended for CI/CD)
export CIVO_TOKEN=${CI_CIVO_TOKEN}
hyve cluster add ci-cluster --provider civo --region PHX1
```

### AWS

```bash
# AWS credentials from CI/CD secrets
export AWS_ACCESS_KEY_ID=${CI_AWS_ACCESS_KEY}
export AWS_SECRET_ACCESS_KEY=${CI_AWS_SECRET_KEY}
hyve cluster add ci-cluster --provider aws --region us-east-1
```

### GCP

```bash
# GCP service account from CI/CD secrets
echo "${CI_GCP_SERVICE_ACCOUNT}" > /tmp/gcp-key.json
export GOOGLE_APPLICATION_CREDENTIALS=/tmp/gcp-key.json

# Use project alias configured in repository
hyve cluster add ci-cluster --provider gcp --gcp-project dev --region us-central1
```

### Azure

```bash
# Azure service principal from CI/CD secrets
export AZURE_CLIENT_ID=${CI_AZURE_CLIENT_ID}
export AZURE_CLIENT_SECRET=${CI_AZURE_CLIENT_SECRET}
export AZURE_TENANT_ID=${CI_AZURE_TENANT_ID}
export AZURE_SUBSCRIPTION_ID=${CI_AZURE_SUBSCRIPTION}
export AZURE_RESOURCE_GROUP=${CI_AZURE_RESOURCE_GROUP}
hyve cluster add ci-cluster --provider azure --region eastus
```

<Info>
  For CI/CD, using environment variables is recommended over storing tokens, as it integrates better with secret management systems.
</Info>

## Troubleshooting

### Civo Token Not Found

```bash
# Error message
Error: Civo API token not found. Please run 'hyve config set-token civo --account <account-id>' or set CIVO_TOKEN environment variable

# Solution 1: Store token
hyve config set-token civo

# Solution 2: Use environment variable
export CIVO_TOKEN=your_token_here
```

### AWS Authentication Failed

```bash
# Error message
Error: AWS authentication failed. Please run 'aws configure' or set AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY environment variables

# Solution: Configure AWS CLI
aws configure

# Or set environment variables
export AWS_ACCESS_KEY_ID=your_key
export AWS_SECRET_ACCESS_KEY=your_secret
```

### GCP Authentication Failed

```bash
# Error message
Error: GCP authentication failed. Please run 'gcloud auth application-default login'

# Solution: Authenticate with gcloud
gcloud auth application-default login

# Configure GCP project alias
hyve config gcp add-project --name dev --id your-project-id
```

### Azure Authentication Failed

```bash
# Error message
Error: Azure authentication failed. Please run 'az login'

# Solution: Authenticate with Azure CLI
az login

# Set required environment variables
export AZURE_SUBSCRIPTION_ID=your-subscription
export AZURE_RESOURCE_GROUP=your-resource-group
```

## Database Schema

The credentials database (`credentials.db`) is a SQLite database:

```sql
-- Civo Accounts Table (API tokens by account)
CREATE TABLE civo_accounts (
    account_id TEXT PRIMARY KEY,
    encrypted_token TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- Git Credentials Table (used by hyve git credentials)
CREATE TABLE credentials (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT NOT NULL,
    encrypted_password TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

<Info>
  AWS, GCP, and Azure credentials are NOT stored in Hyve. They use their native CLI authentication.
</Info>

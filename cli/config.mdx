The `hyve config` command group manages provider-specific configuration (GCP projects, AWS accounts/roles/VPCs, Azure subscriptions) and Civo API tokens. Provider configs are stored as YAML in your state repository under `provider-configs/` and are shared with your team.

Authentication by provider:

| Provider | Local dev | CI/CD |
|----------|-----------|-------|
| **Civo** | `hyve config set-token civo` or `CIVO_TOKEN` | `CIVO_TOKEN` or named account `{ORG}_CIVO_TOKEN` |
| **AWS** | `aws configure` | `AWS_*` env vars, OIDC, or named account `{ACCOUNT}_AWS_*` |
| **GCP** | `gcloud auth application-default login` | `GOOGLE_APPLICATION_CREDENTIALS` or named `{PROJECT}_GCP_CREDENTIALS_JSON` |
| **Azure** | `az login` | `AZURE_*` env vars or named `{SUB}_AZURE_*` |

See the [CI/CD guide](/guides/cicd#named-account-secrets) for the full named account env var pattern.

## Civo Token Management

Civo is the only provider whose credentials are stored encrypted in Hyve (`~/.hyve/credentials.db`, AES-256-GCM). Multiple accounts are supported via the `--account` flag.

### hyve config set-token

```bash
hyve config set-token civo [--account <id>] [--token <token>]
```

| Flag | Default | Description |
|------|---------|-------------|
| `--account, -a` | `default` | Account alias to store the token under |
| `--token, -t` | _(prompted)_ | Token value. Omit to be prompted securely (recommended) |

```bash
# Interactive (recommended — input is hidden)
hyve config set-token civo

# Named account
hyve config set-token civo --account production

# Non-interactive
hyve config set-token civo --token YOUR_TOKEN
```

### hyve config get-token

```bash
hyve config get-token civo [--account <id>]
```

Displays the stored token in plaintext. Use with caution.

### hyve config clear-token

```bash
hyve config clear-token civo [--account <id>]
```

Removes the stored token for the specified account.

### hyve config list-tokens

```bash
hyve config list-tokens
```

Lists all Civo account aliases with stored tokens.

## GCP Project Management

GCP project aliases map a friendly name to a project ID. They are stored in `provider-configs/gcp.yaml` in your state repository.

```bash
# Add a project alias
hyve config gcp add-project --name <alias> --id <project-id>

# List configured projects
hyve config gcp list-projects

# Get project ID by alias
hyve config gcp get-project <alias>

# Remove a project alias
hyve config gcp remove-project <alias>
```

**Example:**

```bash
hyve config gcp add-project --name dev --id my-dev-project-123
hyve config gcp add-project --name prod --id my-prod-project-456

# Use alias when creating clusters
hyve cluster add gcp-cluster --provider gcp --gcp-project dev --region us-central1
```

**`provider-configs/gcp.yaml`:**

```yaml
projects:
  - name: dev
    project_id: my-dev-project-123
  - name: prod
    project_id: my-prod-project-456
```

## AWS Configuration

AWS resources (accounts, EKS roles, VPCs) are stored as aliases in `provider-configs/aws.yaml`.

### Account Aliases

```bash
hyve config aws account-add --name <alias> --id <account-id>
hyve config aws account-list
hyve config aws account-get <alias>
hyve config aws account-remove <alias>
```

### EKS Role Aliases

Register an existing role or create one in AWS:

```bash
# Register existing role
hyve config aws eks-role-add --name <alias> --role-arn <arn>

# Create role in AWS and register alias
hyve config aws eks-role-create --name <alias> --role-name <aws-name> [--region <region>]

# Delete role from AWS (omit --region to config-only)
hyve config aws eks-role-delete <alias> [--region <region>] [--config-only]

# List / get / remove aliases
hyve config aws eks-role-list
hyve config aws eks-role-get <alias>
hyve config aws eks-role-remove <alias>
```

`eks-role-create` creates an IAM role with the EKS assume-role policy and attaches `AmazonEKSClusterPolicy`.

### VPC Aliases

Register an existing VPC or create one in AWS:

```bash
# Register existing VPC
hyve config aws vpc-add --name <alias> --id <vpc-id>

# Create VPC in AWS and register alias
hyve config aws vpc-create --name <alias> [--region <region>] [--cidr <cidr>] [--subnets <cidrs>]

# Delete VPC from AWS (--config-only to keep the AWS resource)
hyve config aws vpc-delete <alias> [--region <region>] [--config-only]

# List / get / remove aliases
hyve config aws vpc-list
hyve config aws vpc-get <alias>
hyve config aws vpc-remove <alias>
```

<Warning>
  `vpc-delete` also removes all associated subnets and detaches internet gateways.
</Warning>

**Example setup:**

```bash
hyve config aws account-add --name prod --id 123456789012
hyve config aws eks-role-create --name eks-role --role-name hyve-eks-role --region us-east-1
hyve config aws vpc-create --name eks-vpc --region us-east-1 --subnets 10.0.1.0/24,10.0.2.0/24

hyve cluster add aws-prod \
  --provider aws \
  --aws-account prod \
  --eks-role-name eks-role \
  --vpc-name eks-vpc \
  --region us-east-1
```

**`provider-configs/aws.yaml`:**

```yaml
accounts:
  - name: prod
    account_id: "123456789012"
eks_roles:
  - name: eks-role
    role_arn: arn:aws:iam::123456789012:role/hyve-eks-role
vpcs:
  - name: eks-vpc
    vpc_id: vpc-0123456789abcdef0
```

## Azure Subscription Management

Azure subscriptions are stored with aliases and resource groups in `provider-configs/azure.yaml`. The subscription ID is resolved automatically from the alias — it does not need to be supplied as an environment variable.

```bash
hyve config azure add-subscription-ids <id,...>
hyve config azure list-subscription-ids
hyve config azure remove-subscription-ids <id,...>
```

**`provider-configs/azure.yaml`:**

```yaml
subscriptions:
  - name: prod-sub
    subscription_id: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
    resource_groups:
      - name: hyve-rg
        location: eastus
```

For CI/CD, credentials are supplied via named env vars (`PROD_SUB_AZURE_TENANT_ID`, etc.) — no subscription ID secret needed. See the [CI/CD guide](/guides/cicd#named-account-secrets).

## Environment Variables

Hyve falls back to environment variables when no stored/named credentials are found:

| Variable | Provider | Purpose |
|----------|----------|---------|
| `CIVO_TOKEN` | Civo | Default account API token |
| `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` | AWS | Default account credentials |
| `AWS_SESSION_TOKEN` | AWS | Temporary session token |
| `GOOGLE_APPLICATION_CREDENTIALS` | GCP | Path to service account JSON |
| `AZURE_TENANT_ID` / `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET` | Azure | Service principal credentials |

**Resolution order for each credential:**

1. Named account env var — `{ACCOUNT_NAME}_{PROVIDER}_{CREDENTIAL}`
2. Standard env var — `CIVO_TOKEN`, `AWS_ACCESS_KEY_ID`, etc.
3. Provider SDK default chain — AWS credential chain, GCP ADC, `az login` session

## Troubleshooting

<AccordionGroup>
  <Accordion title="Civo: token not found">
    ```bash
    hyve config set-token civo
    # or
    export CIVO_TOKEN=your_token
    ```
  </Accordion>

  <Accordion title="AWS: authentication failed">
    ```bash
    aws configure
    # or
    export AWS_ACCESS_KEY_ID=... AWS_SECRET_ACCESS_KEY=...
    aws sts get-caller-identity  # verify
    ```
  </Accordion>

  <Accordion title="GCP: application default credentials not found">
    ```bash
    gcloud auth application-default login
    # or
    export GOOGLE_APPLICATION_CREDENTIALS=/path/to/key.json
    ```
  </Accordion>

  <Accordion title="Azure: not logged in">
    ```bash
    az login
    # or set service principal env vars:
    export AZURE_TENANT_ID=... AZURE_CLIENT_ID=... AZURE_CLIENT_SECRET=...
    ```
  </Accordion>

  <Accordion title="Named account env var not resolved">
    Account names are normalized to uppercase with hyphens/dots replaced by underscores.

    ```bash
    # Cluster spec: awsAccount: prod-us
    # Expected env var:
    export PROD_US_AWS_ACCESS_KEY_ID=AKIA...
    ```
  </Accordion>
</AccordionGroup>

---

The `hyve config` command group manages provider-specific configuration (GCP projects, AWS accounts/roles/VPCs, Azure subscriptions and resource groups) and Civo API tokens. Provider configs are stored as YAML in your state repository under `provider-configs/` and are shared with your team.

Authentication by provider:

| Provider | Local dev | CI/CD |
|----------|-----------|-------|
| **Civo** | `hyve config civo token set --org <name>` or `CIVO_TOKEN` | `CIVO_TOKEN` or named account `{ORG}_CIVO_TOKEN` |
| **AWS** | `aws configure` | `AWS_*` env vars, OIDC, or named account `{ACCOUNT}_AWS_*` |
| **GCP** | `gcloud auth application-default login` | `GOOGLE_APPLICATION_CREDENTIALS` or named `{PROJECT}_GCP_CREDENTIALS_JSON` |
| **Azure** | `az login` | `AZURE_*` env vars or named `{SUB}_AZURE_*` |

See the [CI/CD guide](/guides/cicd#named-account-secrets) for the full named account env var pattern.

## Civo Token Management

Civo is the only provider whose credentials are stored encrypted in Hyve (`~/.hyve/credentials.db`, AES-256-GCM). Multiple organizations are supported via the required `--org` flag.

### hyve config civo token set

```bash
hyve config civo token set --org/-o <name> [--token/-t <token>]
```

| Flag | Short | Description |
|------|-------|-------------|
| `--org` | `-o` | **Required.** Organization alias to store the token under |
| `--token` | `-t` | Token value. Omit to be prompted securely (recommended) |

```bash
# Interactive (recommended — input is hidden)
hyve config civo token set --org my-org

# Non-interactive
hyve config civo token set --org my-org --token YOUR_TOKEN

# Multiple organizations
hyve config civo token set --org production
hyve config civo token set --org development
```

### hyve config civo token get

```bash
hyve config civo token get --org <name>
```

Displays the stored token in plaintext. Use with caution.

### hyve config civo token clear

```bash
hyve config civo token clear --org <name>
```

Removes the stored token for the specified organization.

## Civo Organization Management

Civo organization aliases map a friendly name to a Civo organization ID. They are stored in `provider-configs/civo.yaml` in your state repository and are referenced by the `--org-name` flag when creating clusters.

```bash
# Add an organization alias
hyve config civo org add --name/-n <alias> --id/-i <org-id>

# List configured organizations
hyve config civo org list

# Get organization ID by alias
hyve config civo org get <alias>

# Remove an organization alias
hyve config civo org remove <alias>
```

**Example:**

```bash
hyve config civo org add --name my-org --id abc123def456
hyve config civo org add --name staging-org --id xyz789uvw012

# Use alias when creating clusters
hyve cluster add civo-cluster --provider civo --org-name my-org --region PHX1 --nodes g4s.kube.medium
```

## GCP Project Management

GCP project aliases map a friendly name to a project ID. They are stored in `provider-configs/gcp.yaml` in your state repository.

```bash
# Add a project alias
hyve config gcp project add --name/-n <alias> --id/-i <project-id>

# List configured projects
hyve config gcp project list

# Get project ID by alias
hyve config gcp project get <alias>

# Remove a project alias
hyve config gcp project remove <alias>
```

**Example:**

```bash
hyve config gcp project add --name dev --id my-dev-project-123
hyve config gcp project add --name prod --id my-prod-project-456

# Use alias when creating clusters
hyve cluster add gcp-cluster --provider gcp --project-name dev --region us-central1
```

**`provider-configs/gcp.yaml`:**

```yaml
projects:
  - name: dev
    project_id: my-dev-project-123
  - name: prod
    project_id: my-prod-project-456
```

## AWS Configuration

AWS resources (accounts, EKS roles, VPCs) are stored as aliases in `provider-configs/aws.yaml`. All sub-commands require `--account <name>` to scope operations to a specific account.

### Account Aliases

```bash
hyve config aws account add --name/-n <alias> --id/-i <account-id>
hyve config aws account list
hyve config aws account get <alias>
hyve config aws account remove <alias>
```

### EKS Role Aliases

Register an existing role or create one in AWS:

```bash
# Register existing role
hyve config aws eks-role add --account/-a <name> --name/-n <alias> --role-arn/-r <arn>

# Create role in AWS and register alias
hyve config aws eks-role create --account/-a <name> --name/-n <alias> --role-name <aws-name> [--region/-r <region>]

# Delete role from AWS
hyve config aws eks-role delete --account/-a <name> --name/-n <alias> [--region/-r <region>] [--config-only]

# List / get / remove aliases
hyve config aws eks-role list          --account/-a <name>
hyve config aws eks-role get    <alias> --account/-a <name>
hyve config aws eks-role remove <alias> --account/-a <name>
```

`eks-role create` creates an IAM role with the EKS assume-role policy and attaches `AmazonEKSClusterPolicy`.

### Node Role Aliases

Register an existing IAM node role or create one in AWS:

```bash
# Register existing node role
hyve config aws node-role add --account/-a <name> --name/-n <alias> --role-arn/-r <arn>

# Create node role in AWS and register alias
hyve config aws node-role create --account/-a <name> --name/-n <alias> --role-name <aws-name> [--region/-r <region>]

# Delete node role from AWS
hyve config aws node-role delete <alias> --account/-a <name> [--region/-r <region>] [--config-only]

# List / get / remove aliases
hyve config aws node-role list   --account/-a <name>
hyve config aws node-role get    <alias> --account/-a <name>
hyve config aws node-role remove <alias> --account/-a <name>
```

`node-role create` creates an IAM role with the EC2 assume-role policy and attaches the policies required for EKS worker nodes (`AmazonEKSWorkerNodePolicy`, `AmazonEKS_CNI_Policy`, `AmazonEC2ContainerRegistryReadOnly`).

### VPC Aliases

Register an existing VPC or create one in AWS:

```bash
# Register existing VPC
hyve config aws vpc add --account/-a <name> --name/-n <alias> --id/-i <vpc-id>

# Create VPC in AWS and register alias
hyve config aws vpc create --account/-a <name> --name/-n <alias> [--region/-r <region>] [--cidr/-c <cidr>] [--subnets/-s <cidrs>]

# Delete VPC from AWS (--config-only to keep the AWS resource)
hyve config aws vpc delete --account/-a <name> --name/-n <alias> [--region/-r <region>] [--config-only]

# List / get / remove aliases
hyve config aws vpc list          --account/-a <name>
hyve config aws vpc get    <alias> --account/-a <name>
hyve config aws vpc remove <alias> --account/-a <name>
```

<Warning>
  `vpc delete` also removes all associated subnets and detaches internet gateways.
</Warning>

**Example setup:**

```bash
hyve config aws account add --name prod --id 123456789012
hyve config aws eks-role create --account prod --name eks-role --role-name hyve-eks-role --region us-east-1
hyve config aws vpc create --account prod --name eks-vpc --region us-east-1 --subnets 10.0.1.0/24,10.0.2.0/24

hyve cluster add aws-prod \
  --provider aws \
  --account-name prod \
  --eks-role-name eks-role \
  --vpc-name eks-vpc \
  --region us-east-1
```

**`provider-configs/aws.yaml`:**

```yaml
accounts:
  - name: prod
    account_id: "123456789012"
eks_roles:
  - name: eks-role
    role_arn: arn:aws:iam::123456789012:role/hyve-eks-role
vpcs:
  - name: eks-vpc
    vpc_id: vpc-0123456789abcdef0
```

## Azure Subscription Management

Azure subscriptions are stored with aliases and resource groups in `provider-configs/azure.yaml`. The subscription ID is resolved automatically from the alias — it does not need to be supplied as an environment variable.

### Subscriptions

```bash
# Add a subscription alias
hyve config azure subscription add --name/-n <alias> --id/-i <subscription-id>

# List configured subscriptions
hyve config azure subscription list

# Remove a subscription alias
hyve config azure subscription remove <alias>
```

**Example:**

```bash
hyve config azure subscription add --name prod-sub --id xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
hyve config azure subscription add --name dev-sub  --id yyyyyyyy-yyyy-yyyy-yyyy-yyyyyyyyyyyy
```

### Resource Groups

Resource groups are registered under a subscription alias. `resource-group add` creates the resource group in Azure **and** saves the entry to `provider-configs/azure.yaml`.

```bash
# Create resource group in Azure and register locally
hyve config azure resource-group add --subscription/-s <alias> --name/-n <rg-name> --location/-l <region>

# Delete resource group from Azure and remove from local config
hyve config azure resource-group delete --subscription/-s <alias> --name/-n <rg-name>

# List resource groups for a subscription
hyve config azure resource-group list --subscription/-s <alias>
```

**Example:**

```bash
hyve config azure resource-group add --subscription prod-sub --name hyve-rg --location eastus
hyve config azure resource-group list --subscription prod-sub
```

**`provider-configs/azure.yaml`:**

```yaml
subscriptions:
  - name: prod-sub
    subscription_id: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
    resource_groups:
      - name: hyve-rg
        location: eastus
```

For CI/CD, credentials are supplied via named env vars (`PROD_SUB_AZURE_TENANT_ID`, etc.) — no subscription ID secret needed. See the [CI/CD guide](/guides/cicd#named-account-secrets).

## Environment Variables

Hyve falls back to environment variables when no stored/named credentials are found:

| Variable | Provider | Purpose |
|----------|----------|---------|
| `CIVO_TOKEN` | Civo | Default organization API token |
| `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` | AWS | Default account credentials |
| `AWS_SESSION_TOKEN` | AWS | Temporary session token |
| `GOOGLE_APPLICATION_CREDENTIALS` | GCP | Path to service account JSON |
| `AZURE_TENANT_ID` / `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET` | Azure | Service principal credentials |

**Resolution order for each credential:**

1. Named account env var — `{ACCOUNT_NAME}_{PROVIDER}_{CREDENTIAL}`
2. Standard env var — `CIVO_TOKEN`, `AWS_ACCESS_KEY_ID`, etc.
3. Provider SDK default chain — AWS credential chain, GCP ADC, `az login` session

## Troubleshooting

<AccordionGroup>
  <Accordion title="Civo: token not found">
    ```bash
    hyve config civo token set --org my-org
    # or
    export CIVO_TOKEN=your_token
    ```
  </Accordion>

  <Accordion title="AWS: authentication failed">
    ```bash
    aws configure
    # or
    export AWS_ACCESS_KEY_ID=... AWS_SECRET_ACCESS_KEY=...
    aws sts get-caller-identity  # verify
    ```
  </Accordion>

  <Accordion title="GCP: application default credentials not found">
    ```bash
    gcloud auth application-default login
    # or
    export GOOGLE_APPLICATION_CREDENTIALS=/path/to/key.json
    ```
  </Accordion>

  <Accordion title="Azure: not logged in">
    ```bash
    az login
    # or set service principal env vars:
    export AZURE_TENANT_ID=... AZURE_CLIENT_ID=... AZURE_CLIENT_SECRET=...
    ```
  </Accordion>

  <Accordion title="Named account env var not resolved">
    Account names are normalized to uppercase with hyphens/dots replaced by underscores.

    ```bash
    # Cluster spec: awsAccount: prod-us
    # Expected env var:
    export PROD_US_AWS_ACCESS_KEY_ID=AKIA...
    ```
  </Accordion>
</AccordionGroup>

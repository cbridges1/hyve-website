---
title: 'hyve sync'
description: 'Discover unmanaged clusters and reconcile provider config resources'
---

`hyve sync` scans all configured cloud accounts and performs two operations:

1. **Cluster discovery** — compares running cloud clusters against repo definitions. Unmanaged clusters (present in the cloud, absent from the repo) are listed and you select which to import.
2. **Provider config sync** — queries each account for VPCs, IAM roles, resource groups, and networks and reconciles the provider config YAML files immediately (adding missing entries, removing stale ones).

Both steps commit and push if any changes were made.

`hyve sync` replaces `hyve cluster import`.

---

## Usage

```bash
hyve sync [flags]
```

## Flags

| Flag | Short | Default | Description |
|------|-------|---------|-------------|
| `--provider` | `-p` | _(all)_ | Limit to a specific provider: `civo`, `aws`, `gcp`, `azure` |
| `--account` | `-a` | _(all)_ | Limit to a specific account/subscription/project alias |
| `--dry-run` | | `false` | Print what would be imported without writing anything |

---

## What gets synced

### Cluster discovery

For each configured cloud account, `hyve sync` lists all running clusters and compares them against the cluster YAML files in your repo. Clusters that exist in the cloud but have no matching definition in your repo are listed as **unmanaged**.

You select which unmanaged clusters to import. For each selected cluster, Hyve writes a `ClusterDefinition` YAML to your repo's `clusters/` directory and commits the result.

### Provider config resources

After cluster discovery, `hyve sync` queries each account for cloud resources and reconciles the provider config YAML files under `provider-configs/`:

| Provider | Resources synced |
|----------|-----------------|
| **AWS** | VPCs (non-default, by Name tag and ID) |
| **AWS** | EKS and node IAM roles already in config — verified against IAM, stale entries removed |
| **Azure** | Resource groups (all groups in each subscription) |
| **Civo** | Networks (all networks in each org) |
| **GCP** | No sub-resources synced (projects are registered manually) |

New resources found in the cloud are added to the config. Resources referenced in the config that no longer exist in the cloud are removed.

## Examples

```bash
# Full sync across all providers
hyve sync

# Preview without writing anything
hyve sync --dry-run

# Limit to AWS only
hyve sync --provider aws

# Limit to a specific account
hyve sync --provider aws --account prod
```

---

## Passive sync on every reconcile

Provider config resources are also reconciled automatically on **every `hyve reconcile` run** — not just when you run `hyve sync`. This means:

- VPCs created by a `beforeCreate` hook and exported via `HYVE_VPC_NAME` / `HYVE_VPC_ID` are written to the config immediately, then confirmed on the next reconcile.
- Stale entries (roles deleted externally, resource groups removed) are cleaned up without manual intervention.

```bash
# Reconcile also syncs provider config resources automatically
hyve reconcile
```

## Provider config file structure

Each cloud account's resources are stored in a separate YAML file under a per-provider subdirectory:

```
provider-configs/
  aws/
    prod.yaml
    dev.yaml
  azure/
    prod-sub.yaml
  gcp/
    analytics.yaml
  civo/
    my-org.yaml
```

**`provider-configs/aws/prod.yaml`** (after a sync):

```yaml
name: prod
account_id: "123456789012"
access_key_id: ${AWS_ACCESS_KEY_ID}
secret_access_key: ${AWS_SECRET_ACCESS_KEY}
vpcs:
  - name: prod-vpc
    vpc_id: vpc-0abc123456789
  - name: dev-vpc
    vpc_id: vpc-0def987654321
eks_roles:
  - name: eks-cluster-role
    role_arn: arn:aws:iam::123456789012:role/eks-cluster-role
node_roles:
  - name: eks-node-role
    role_arn: arn:aws:iam::123456789012:role/eks-node-role
```

**`provider-configs/azure/prod-sub.yaml`** (after a sync):

```yaml
name: prod-sub
subscription_id: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
tenant_id: ${AZURE_TENANT_ID}
client_id: ${AZURE_CLIENT_ID}
client_secret: ${AZURE_CLIENT_SECRET}
resource_groups:
  - name: my-rg
    location: eastus
  - name: shared-rg
    location: westus2
```

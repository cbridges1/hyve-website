The `hyve cluster` command group manages Kubernetes cluster definitions stored in Git repositories. Clusters are defined as YAML files and can be created, modified, listed, and deleted through these commands.

## Overview

Hyve manages clusters declaratively through YAML files stored in your Git repository's `clusters/` directory. Each cluster definition includes provider details, region, node specifications, and optional ingress configuration.

## Commands

### hyve cluster add

Create a new cluster configuration YAML file.

```bash
hyve cluster add [cluster-name] [flags]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster to create. This will be used as the filename and cluster identifier.
</ParamField>

<ParamField path="--region" type="string" default="PHX1">
  Cloud provider region where the cluster will be created
</ParamField>

<ParamField path="--provider" type="string" required>
  Cloud provider. Must be one of the supported providers:
  - `civo` - Civo Cloud (K3s/Talos clusters)
  - `aws` - Amazon Web Services (EKS)
  - `gcp` - Google Cloud Platform (GKE)
  - `azure` - Microsoft Azure (AKS)
</ParamField>

<ParamField path="--nodes" type="string[]" default='["g4s.kube.small"]'>
  Node sizes for the cluster (comma-separated). Each entry creates one node. For simple single-pool clusters. Use `--node-group` instead for named multi-pool configurations with autoscaling.
</ParamField>

<ParamField path="--cluster-type" type="string" default="k3s">
  Type of Kubernetes cluster (`k3s`, `talos`). **Civo only** — ignored for AWS, GCP, and Azure providers.
</ParamField>

<ParamField path="--account-name, -a" type="string">
  AWS account alias (required for AWS provider). Configure aliases with `hyve config aws account add`.
</ParamField>

<ParamField path="--project-name" type="string">
  GCP project alias (required for GCP provider). Configure aliases with `hyve config gcp project add`.
</ParamField>

<ParamField path="--subscription-name, -s" type="string">
  Azure subscription alias (required for Azure provider). Configure aliases with `hyve config azure subscription add`.
</ParamField>

<ParamField path="--org-name, -o" type="string">
  Civo organization alias (required for Civo provider). Configure aliases with `hyve config civo org add`.
</ParamField>

<ParamField path="--vpc-name, -v" type="string">
  AWS VPC alias (required for AWS provider). The alias must be present in `provider-configs/aws/<account>.yaml` — run `hyve sync` to auto-populate from the cloud, or add it manually.
</ParamField>

<ParamField path="--eks-role-name, -e" type="string">
  AWS EKS IAM role alias (required for AWS provider). The alias must be present in `provider-configs/aws/<account>.yaml` — run `hyve sync` to auto-populate from the cloud, or add it manually.
</ParamField>

<ParamField path="--node-group, -g" type="string (repeatable)">
  Named node group spec. Can be specified multiple times for multiple pools. Mutually exclusive with `--nodes` — when node groups are provided, they take precedence.

  Format: `name=<name>,type=<instanceType>,count=<n>[,min=<n>,max=<n>,disk=<gb>,spot=true,mode=System|User]`

  | Key | Required | Description |
  |-----|----------|-------------|
  | `name` | Yes | Node group name |
  | `type` | Yes | Instance type (e.g. `t3.large`, `e2-standard-4`) |
  | `count` | No | Desired node count (default: 1) |
  | `min` | No | Minimum node count for autoscaling |
  | `max` | No | Maximum node count for autoscaling |
  | `disk` | No | OS disk size in GB |
  | `spot` | No | Use spot/preemptible instances (`true`/`false`) |
  | `mode` | No | Azure only: `System` (default) or `User` |

  <Info>
    **AWS EKS only:** avoid `nano` and `micro` instance types. These expose only 4 pods per node — insufficient for EKS system daemonsets plus workloads. Minimum: `t3.small`. Recommended: `t3.medium`.
  </Info>
</ParamField>

<ParamField path="--before-create" type="string (repeatable)">
  Name of a workflow to run before the cluster is created (pre-provision hook). Can be specified multiple times. Workflow names must match an existing workflow defined with `hyve workflow add`.

  ```bash
  hyve cluster add my-cluster --provider civo --region PHX1 \
    --nodes g4s.kube.medium --org-name my-org \
    --before-create pre-flight-checks
  ```
</ParamField>

<ParamField path="--on-created" type="string (repeatable)">
  Name of a workflow to run after the cluster is successfully created. Can be specified multiple times to attach several workflows. Workflow names must match an existing workflow defined with `hyve workflow add`.

  ```bash
  hyve cluster add my-cluster --provider civo --region PHX1 \
    --nodes g4s.kube.medium --org-name my-org \
    --on-created setup-monitoring \
    --on-created notify-slack
  ```
</ParamField>

<ParamField path="--on-destroy" type="string (repeatable)">
  Name of a workflow to run before the cluster is destroyed. Can be specified multiple times to attach several workflows. Workflow names must match an existing workflow defined with `hyve workflow add`.

  ```bash
  hyve cluster add my-cluster --provider civo --region PHX1 \
    --nodes g4s.kube.medium --org-name my-org \
    --on-destroy backup-data \
    --on-destroy notify-slack
  ```
</ParamField>

<ParamField path="--after-delete" type="string (repeatable)">
  Name of a workflow to run after the cluster has been deleted (post-teardown hook). Can be specified multiple times. Workflow names must match an existing workflow defined with `hyve workflow add`.

  ```bash
  hyve cluster add my-cluster --provider civo --region PHX1 \
    --nodes g4s.kube.medium --org-name my-org \
    --after-delete cleanup-dns
  ```
</ParamField>

<ParamField path="--pause" type="boolean" default="false">
  Create the cluster in a paused state. The YAML file is written and committed but the reconciler skips this cluster until the pause is lifted with `hyve cluster modify <name> --unpause`.
</ParamField>

<ParamField path="--expires-at" type="string">
  RFC 3339 timestamp after which the reconciler automatically deletes the cluster (e.g. `2026-05-01T00:00:00Z`). When the deadline passes the reconciler runs any `onDestroy` workflows, deletes the cluster, and removes the YAML file. Clusters created from templates with a `schedule` field have this value computed automatically at execute time.
</ParamField>

### hyve cluster modify

Update an existing cluster configuration YAML file.

```bash
hyve cluster modify [cluster-name] [flags]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster to modify
</ParamField>

<ParamField path="--region" type="string">
  Update the region
</ParamField>

<ParamField path="--provider" type="string">
  Update the cloud provider. Must be one of: `civo`, `aws`, `gcp`, `azure`
</ParamField>

<ParamField path="--nodes" type="string[]">
  Update node sizes (comma-separated). For simple single-pool clusters. Use `--node-group` instead for named multi-pool configurations with autoscaling.
</ParamField>

<ParamField path="--node-group" type="string (repeatable)">
  Named node group spec. Can be specified multiple times for multiple pools. Mutually exclusive with `--nodes` — when node groups are provided, they take precedence.

  Format: `name=<name>,type=<instanceType>,count=<n>[,min=<n>,max=<n>,disk=<gb>,spot=true,mode=System|User]`

  | Key | Required | Description |
  |-----|----------|-------------|
  | `name` | Yes | Node group name |
  | `type` | Yes | Instance type (e.g. `t3.large`, `e2-standard-4`) |
  | `count` | No | Desired node count (default: 1) |
  | `min` | No | Minimum node count for autoscaling |
  | `max` | No | Maximum node count for autoscaling |
  | `disk` | No | OS disk size in GB |
  | `spot` | No | Use spot/preemptible instances (`true`/`false`) |
  | `mode` | No | Azure only: `System` (default) or `User` |
</ParamField>

<ParamField path="--cluster-type" type="string">
  Update the cluster type (`k3s`, `talos`). **Civo only** — ignored for AWS, GCP, and Azure providers.
</ParamField>

<ParamField path="--before-create" type="string (repeatable)">
  Replace the `beforeCreate` workflow list. Pass an empty value to clear: `--before-create ""`.
</ParamField>

<ParamField path="--on-created" type="string (repeatable)">
  Replace the `onCreated` workflow list.
</ParamField>

<ParamField path="--on-destroy" type="string (repeatable)">
  Replace the `onDestroy` workflow list.
</ParamField>

<ParamField path="--after-delete" type="string (repeatable)">
  Replace the `afterDelete` workflow list.
</ParamField>

<ParamField path="--pause" type="boolean">
  Pause reconciliation for this cluster. The cluster keeps running in the cloud; Hyve stops comparing or modifying it until `--unpause` is set.
</ParamField>

<ParamField path="--unpause" type="boolean">
  Resume reconciliation for a previously paused cluster. Clears the `spec.pause` flag.
</ParamField>

<ParamField path="--expires-at" type="string">
  Update the expiry timestamp (RFC 3339, e.g. `2026-05-01T00:00:00Z`). Set to `none` to clear an existing expiry.
</ParamField>

**Example:**

```bash
# Scale up by adding more nodes
hyve cluster modify my-cluster \
  --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large

# Change region (requires recreation)
hyve cluster modify staging-cluster --region FRA1

# Pause reconciliation during a maintenance window
hyve cluster modify production --pause

# Resume reconciliation after maintenance
hyve cluster modify production --unpause

# Set an expiry on a temporary cluster
hyve cluster modify dev-feature-x --expires-at 2026-05-01T00:00:00Z

# Clear an existing expiry
hyve cluster modify dev-feature-x --expires-at none
```

### hyve cluster delete

Delete a cluster by removing its YAML definition and reconciling.

```bash
hyve cluster delete [cluster-name] [flags]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster to delete
</ParamField>

<ParamField path="--force" type="boolean" default="false">
  Delete the cluster from the cloud provider immediately before removing the configuration file. Bypasses CI/CD — use this when you want to destroy the cluster right now rather than letting the pipeline handle it.
</ParamField>

<ParamField path="--force-cloud" type="boolean" default="false">
  Used together with `--force`: delete from cloud even if no configuration file exists.
</ParamField>

**Examples:**

```bash
# Default: remove YAML, commit, reconcile (CI/CD pipeline handles cloud deletion)
hyve cluster delete my-cluster

# Force: delete from cloud immediately, then remove configuration
hyve cluster delete my-cluster --force

# Force delete from cloud when no config file exists
hyve cluster delete orphaned-cluster --force --force-cloud
```

**Default process:**

1. Removes the cluster configuration YAML file
2. Commits and pushes the change to the state repository
3. Runs reconciliation

   - In **CI/CD mode** with `strictDelete: true`: the push triggers the pipeline, which deletes the cloud cluster
   - In **local mode** with `strictDelete: true`: the local reconcile deletes the cloud cluster immediately

**Force process (`--force`):**

1. **Deletes the cluster from the cloud provider first** (before touching Git)
2. Removes any ingress controllers
3. Removes the cluster configuration YAML file
4. Commits changes to Git
5. Cleans up the stored kubeconfig
6. Runs reconciliation

<Note>
  Cloud deletion always happens before YAML removal. If the cloud delete fails the configuration file is preserved, so you can safely retry.
</Note>

<Info>
  The default (non-force) delete is the recommended approach in CI/CD workflows. Enable `strictDelete: true` in `hyve.yaml` so that removing a cluster YAML automatically triggers deletion in the pipeline. See the [CI/CD guide](/guides/cicd) for details.
</Info>

### hyve cluster import

Import an already-running cluster into Hyve without reprovisioning it. Hyve records the cluster definition in Git and auto-detects node group configuration from the cloud provider.

```bash
hyve cluster import <name> [flags]
```

<ParamField path="name" type="string" required>
  Name of the existing cluster (must match the name in the cloud provider)
</ParamField>

<ParamField path="--provider, -p" type="string" required>
  Cloud provider: `civo`, `aws`, `gcp`, or `azure`
</ParamField>

<ParamField path="--region, -r" type="string">
  Region where the cluster is running
</ParamField>

<ParamField path="--account-name, -a" type="string">
  AWS account alias (required for AWS)
</ParamField>

<ParamField path="--project-name" type="string">
  GCP project alias (required for GCP)
</ParamField>

<ParamField path="--subscription-name, -s" type="string">
  Azure subscription alias (required for Azure)
</ParamField>

<ParamField path="--org-name, -o" type="string">
  Civo organization alias (required for Civo)
</ParamField>

<ParamField path="--vpc-name, -v" type="string">
  AWS VPC alias (required for AWS)
</ParamField>

<ParamField path="--eks-role-name, -e" type="string">
  AWS EKS IAM role alias (required for AWS)
</ParamField>

<ParamField path="--nodes, -n" type="string[]">
  Explicit node sizes. If omitted, Hyve queries the cloud provider and auto-detects the running node pool configuration.
</ParamField>

**Examples:**

```bash
# Import an Azure AKS cluster (node groups auto-detected from cloud)
hyve cluster import my-aks-cluster \
  --provider azure \
  --region eastus \
  --subscription-name prod-sub

# Import a GKE cluster with explicit nodes
hyve cluster import my-gke-cluster \
  --provider gcp \
  --region us-central1 \
  --project-name my-project \
  --nodes e2-standard-4

# Import an EKS cluster (node groups auto-detected)
hyve cluster import my-eks-cluster \
  --provider aws \
  --region us-east-1 \
  --account-name production \
  --vpc-name prod-vpc \
  --eks-role-name eks-cluster-role
```

<Info>
  When `--nodes` is not specified, Hyve queries the live cluster and records its current node pool configuration. The cloud cluster is **not** reprovisioned — only a YAML definition is written to Git.
</Info>

<Warning>
  Import is disabled when `strictDelete: true` is set in `hyve.yaml`. In strict-delete mode, Hyve owns the full desired state — importing an unmanaged cluster would cause it to be deleted on the next reconciliation.
</Warning>

```bash
hyve cluster release <name>
```

<ParamField path="name" type="string" required>

</ParamField>

```bash
hyve cluster release my-cluster
```

## Cluster Definition Format

Clusters are defined in YAML format and stored in the `clusters/` directory of your Git repository. Provider-specific fields are required for each cloud provider.

Cluster specs support two mutually exclusive ways to define nodes: `spec.nodes` (simple single-pool, primarily for Civo) and `spec.nodeGroups` (named multi-pool with autoscaling, preferred for AWS/GCP/Azure). When `nodeGroups` are present, they take precedence over `nodes`.

### Civo Cluster Structure

```yaml
apiVersion: v1
kind: Cluster
metadata:
  name: my-cluster
  region: PHX1               # PHX1 | NYC1 | LON1 | FRA1
spec:
  provider: civo
  civoOrganization: my-org   # org alias — registered with hyve config civo org add
  nodes:
    - g4s.kube.medium        # one entry per node
    - g4s.kube.medium
  clusterType: k3s            # k3s | talos
  ingress:
    enabled: true
    loadBalancer: true
  workflows:
    beforeCreate:
      - pre-flight-checks
    onCreated:
      - setup-monitoring
    onDestroy:
      - backup-data
    afterDelete:
      - cleanup-dns
```

<Info>
  Civo supports only a single node pool. For Civo clusters, use `spec.nodes` rather than `spec.nodeGroups`.
</Info>

### Azure Cluster Structure

Azure AKS clusters require a `nodeGroups` definition. The first pool must use `mode: System`.

```yaml
apiVersion: v1
kind: Cluster
metadata:
  name: my-aks-cluster
  region: eastus
spec:
  provider: azure
  azureSubscription: prod-sub      # alias from hyve config azure subscription add
  nodeGroups:
    - name: system                 # first pool must be System mode
      instanceType: Standard_DS2_v2
      count: 2
      mode: System
    - name: workers
      instanceType: Standard_DS3_v2
      count: 3
      minCount: 1
      maxCount: 8
      mode: User
      spot: true
  clusterType: k3s
  ingress:
    enabled: true
    loadBalancer: true
  workflows:
    beforeCreate:
      - pre-flight-checks
    onCreated:
      - setup-monitoring
    onDestroy:
      - backup-data
    afterDelete:
      - cleanup-dns
```

<Info>
  The `azureSubscription` alias must be registered with `hyve config azure subscription add --name <alias> --id <subscription-id>` before cluster creation.
</Info>

### GCP Cluster Structure

GCP GKE clusters require a project alias and `nodeGroups` definition:

```yaml
apiVersion: v1
kind: Cluster
metadata:
  name: my-gke-cluster
  region: us-central1
spec:
  provider: gcp
  gcpProject: prod              # alias from hyve config gcp project add
  gcpProjectId: my-project-123  # resolved project ID (auto-populated)
  nodeGroups:
    - name: workers
      instanceType: e2-standard-4
      count: 3
      minCount: 2
      maxCount: 8
  clusterType: k3s
  ingress:
    enabled: true
    loadBalancer: true
```

### AWS Cluster Structure

AWS EKS clusters require VPC and IAM role aliases and a `nodeGroups` definition:

```yaml
apiVersion: v1
kind: Cluster
metadata:
  name: my-eks-cluster
  region: us-east-1
spec:
  provider: aws
  awsAccount: prod                 # alias from hyve config aws account add
  awsAccountId: "123456789012"     # resolved account ID (auto-populated)
  awsVpcName: prod-vpc             # alias from provider-configs/aws/prod.yaml (required)
  awsVpcId: vpc-12345678           # resolved VPC ID (auto-populated)
  awsEksRole: eks-cluster-role     # alias from provider-configs/aws/prod.yaml (required)
  awsEksRoleArn: arn:aws:iam::123456789012:role/EKSClusterRole  # resolved (auto-populated)
  nodeGroups:
    - name: system
      instanceType: t3.medium
      count: 2
    - name: workers
      instanceType: m5.xlarge
      count: 4
      minCount: 2
      maxCount: 10
      spot: true
  clusterType: k3s
  ingress:
    enabled: true
    loadBalancer: true
```

<Note>
  When creating an AWS EKS cluster, Hyve automatically:
  - Discovers subnets from the specified VPC
  - Creates a security group for the cluster with appropriate rules
  - Uses the specified IAM role for the EKS control plane
</Note>

## Reconciliation

After creating or modifying cluster configurations, run `hyve reconcile` to synchronize the desired state (Git) with the actual state (cloud provider).

### Reconciliation Modes

The reconciliation mode is controlled by a `hyve.yaml` file at the root of your state repository:

```yaml
# hyve.yaml
reconcile:
  mode: local          # "local" (default) or "cicd"
  strictDelete: false  # set to true to delete unmanaged cloud clusters
```

| Mode | Behaviour |
|------|-----------|
| `local` | Hyve provisions clusters directly from the machine running the command. Default when `hyve.yaml` is absent. |
| `cicd` | Hyve validates and commits the desired state to Git, then exits. A CI/CD pipeline handles the actual provisioning by calling `hyve reconcile --path .`. |

#### Strict Delete

When `strictDelete: true`, Hyve deletes any cloud cluster that has no matching YAML definition in your state repository. By default (`false`), only clusters whose names begin with a managed prefix (`hyve-`, `civo-deploy-`) are eligible for deletion.

<Warning>
  Enable `strictDelete` only after confirming your state repository contains definitions for every cluster you want to keep.
</Warning>

### Running Reconciliation

```bash
# Local mode — provisions clusters from this machine
hyve reconcile

# CI/CD pipeline mode — use the checked-out repo path directly
# This bypasses the hyve.yaml cicd check and always reconciles locally
hyve reconcile --path .

# Provide a relative path to a specific checkout
hyve reconcile --path /path/to/repo
```

#### `--path` Flag

| Flag | Description |
|------|-------------|
| `--path / -p` | Path to an already-checked-out local repository. Skips cloning and the `hyve.yaml` mode check; always runs full local reconciliation. Designed for use inside CI/CD pipelines. |

### Reconciliation Process

1. **Load Definitions**: Read all cluster YAML files from the Git repository
2. **Validate**: Ensure all cluster definitions are valid
3. **Check Cloud State**: Query the cloud provider for existing clusters
4. **Determine Actions**: Compare desired vs actual state
5. **Execute Changes**: Create, update, or delete clusters as needed
6. **Configure Ingress**: Set up ingress controllers if enabled
7. **Commit & Push**: Write reconciliation results back to the state repository

### Actions

- **Create**: Cluster exists in Git but not in cloud
- **Update**: Cluster exists in both but configuration differs
- **Delete**: Cluster exists in cloud but not in Git
- **None**: States match, no action needed

<Note>
  See the [CI/CD Workflow Guide](/guides/cicd) for a complete example using GitHub Actions.
</Note>

## Common Use Cases

### Create a Production Cluster

```bash
# Create cluster with 3 large nodes (--provider is required)
hyve cluster add production \
  --provider civo \
  --region NYC1 \
  --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large \
  --cluster-type k3s

# Wait for cluster to be ready (check with kubectl)
hyve use production
kubectl get nodes
```

### Scale a Cluster

```bash
# Add more nodes to existing cluster
hyve cluster modify my-cluster \
  --nodes g4s.kube.medium,g4s.kube.medium,g4s.kube.medium,g4s.kube.medium

# Reconciliation will update the cluster
```

### Multi-Region Deployment

```bash
# Create clusters in different regions (--provider is required)
hyve cluster add us-east --provider civo --region NYC1 --nodes g4s.kube.medium,g4s.kube.medium
hyve cluster add us-west --provider civo --region PHX1 --nodes g4s.kube.medium,g4s.kube.medium
hyve cluster add eu-central --provider civo --region FRA1 --nodes g4s.kube.medium,g4s.kube.medium

# All clusters managed in the same Git repository
```

### Attach Lifecycle Workflows

```bash
# All four lifecycle hooks on a Civo cluster
hyve cluster add production \
  --provider civo \
  --region NYC1 \
  --nodes g4s.kube.large,g4s.kube.large \
  --org-name my-org \
  --before-create pre-flight-checks \
  --on-created setup-monitoring \
  --on-destroy backup-data \
  --after-delete cleanup-dns

# Multiple workflows per lifecycle event on AWS
hyve cluster add production \
  --provider aws \
  --region us-east-1 \
  --account-name prod \
  --vpc-name prod-vpc \
  --eks-role-name eks-role \
  --node-group "name=workers,type=m5.large,count=3" \
  --before-create pre-flight-checks \
  --on-created setup-monitoring \
  --on-created notify-slack \
  --on-destroy backup-data \
  --on-destroy notify-slack \
  --after-delete cleanup-dns
```

### Cleanup Orphaned Clusters

```bash
# List all clusters
hyve cluster list

# Force delete cluster not in Git
hyve cluster force-delete orphaned-cluster --region PHX1

# Or delete with configuration
hyve cluster delete old-cluster
```

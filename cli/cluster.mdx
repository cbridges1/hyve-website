The `hyve cluster` command group manages Kubernetes cluster definitions stored in Git repositories. Clusters are defined as YAML files and can be created, modified, listed, and deleted through these commands.

## Overview

Hyve manages clusters declaratively through YAML files stored in your Git repository's `clusters/` directory. Each cluster definition includes provider details, region, node specifications, and optional ingress configuration.

## Commands

### hyve cluster add

Create a new cluster configuration YAML file.

```bash
hyve cluster add [cluster-name] [flags]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster to create. This will be used as the filename and cluster identifier.
</ParamField>

<ParamField path="--region" type="string" default="PHX1">
  Cloud provider region where the cluster will be created
</ParamField>

<ParamField path="--provider" type="string" required>
  Cloud provider. Must be one of the supported providers:
  - `civo` - Civo Cloud (K3s/Talos clusters)
  - `aws` - Amazon Web Services (EKS)
  - `gcp` - Google Cloud Platform (GKE)
  - `azure` - Microsoft Azure (AKS)
</ParamField>

<ParamField path="--nodes" type="string[]" default='["g4s.kube.small"]'>
  Node sizes for the cluster (comma-separated). Each entry creates one node. For simple single-pool clusters. Use `--node-group` instead for named multi-pool configurations with autoscaling.
</ParamField>

<ParamField path="--node-group" type="string (repeatable)">
  Named node group spec. Can be specified multiple times for multiple pools. Mutually exclusive with `--nodes` — when node groups are provided, they take precedence.

  Format: `name=<name>,type=<instanceType>,count=<n>[,min=<n>,max=<n>,disk=<gb>,spot=true,mode=System|User]`

  | Key | Required | Description |
  |-----|----------|-------------|
  | `name` | Yes | Node group name |
  | `type` | Yes | Instance type (e.g. `t3.large`, `e2-standard-4`) |
  | `count` | No | Desired node count (default: 1) |
  | `min` | No | Minimum node count for autoscaling |
  | `max` | No | Maximum node count for autoscaling |
  | `disk` | No | OS disk size in GB |
  | `spot` | No | Use spot/preemptible instances (`true`/`false`) |
  | `mode` | No | Azure only: `System` (default) or `User` |

  <Warning>
    **AWS EKS only:** `nano` and `micro` instance types are rejected. These expose only 4 pods per node — insufficient for EKS system daemonsets plus workloads. Minimum: `t3.small`. Recommended: `t3.medium`.
  </Warning>
</ParamField>

<ParamField path="--cluster-type" type="string" default="k3s">
  Type of Kubernetes cluster (k3s, talos)
</ParamField>

<ParamField path="--gcp-project" type="string">
  GCP project alias (required for GCP provider). Configure aliases with `hyve config gcp add-project`.
</ParamField>

<ParamField path="--civo-account" type="string" default="default">
  Civo account to use (for multi-account setups).
</ParamField>

<ParamField path="--aws-account" type="string">
  AWS account alias (optional for AWS provider). Configure aliases with `hyve config aws account-add`.
</ParamField>

<ParamField path="--vpc-name" type="string">
  AWS VPC name alias (required for AWS provider). Configure aliases with `hyve config aws vpc-add` or create with `hyve config aws vpc-create`.
</ParamField>

<ParamField path="--eks-role-name" type="string">
  AWS EKS IAM role name alias (required for AWS provider). Configure aliases with `hyve config aws eks-role-add` or create with `hyve config aws eks-role-create`.
</ParamField>

**Examples by Provider:**

<Tabs>
  <Tab title="AWS">

    ```bash
    hyve config aws vpc-add --name dev-vpc --id vpc-12345678
    hyve config aws eks-role-add --name eks-cluster-role --role-arn arn:aws:iam::123456789012:role/EKSClusterRole

      --provider aws \

      --provider aws \
      --region us-east-1 \
      --account-name production \
      --vpc-name prod-vpc \
      --eks-role-name eks-cluster-role \
      --node-role-name node-role \
    ```

  </Tab>

  The `--provider` flag only accepts valid providers: `civo`, `aws`, `gcp`, or `azure`. Using an invalid provider will result in an error.
</Warning>

  - EKS Role: `hyve config aws eks-role-add --name <alias> --role-arn <arn>` or `hyve config aws eks-role-create --name <alias> --role-name <name>`
</Info>

### hyve cluster modify

Update an existing cluster configuration YAML file.

```bash
hyve cluster modify [cluster-name] [flags]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster to modify
</ParamField>

<ParamField path="--region" type="string">
  Update the region
</ParamField>

<ParamField path="--provider" type="string">
  Update the cloud provider. Must be one of: `civo`, `aws`, `gcp`, `azure`
</ParamField>

<ParamField path="--nodes" type="string[]">
  Update node sizes (comma-separated). For simple single-pool clusters. Use `--node-group` instead for named multi-pool configurations with autoscaling.
</ParamField>

<ParamField path="--node-group" type="string (repeatable)">
  Named node group spec. Can be specified multiple times for multiple pools. Mutually exclusive with `--nodes` — when node groups are provided, they take precedence.

  Format: `name=<name>,type=<instanceType>,count=<n>[,min=<n>,max=<n>,disk=<gb>,spot=true,mode=System|User]`

  | Key | Required | Description |
  |-----|----------|-------------|
  | `name` | Yes | Node group name |
  | `type` | Yes | Instance type (e.g. `t3.large`, `e2-standard-4`) |
  | `count` | No | Desired node count (default: 1) |
  | `min` | No | Minimum node count for autoscaling |
  | `max` | No | Maximum node count for autoscaling |
  | `disk` | No | OS disk size in GB |
  | `spot` | No | Use spot/preemptible instances (`true`/`false`) |
  | `mode` | No | Azure only: `System` (default) or `User` |
</ParamField>

<ParamField path="--cluster-type" type="string">
  Update the cluster type
</ParamField>

**Example:**

```bash
# Scale up by adding more nodes
hyve cluster modify my-cluster \
  --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large

# Change region (requires recreation)
hyve cluster modify staging-cluster --region FRA1
```

**Output:**

```
Using Git repository 'production': https://github.com/company/hyve-prod.git
Git repository synchronized
Updated cluster definition file: /Users/username/.hyve/repositories/production/clusters/my-cluster.yaml
Cluster my-cluster updated configuration:
  Region: PHX1
  Provider: civo
  Nodes: [g4s.kube.large g4s.kube.large g4s.kube.large]
  Cluster Type: k3s
Changes committed and pushed to Git repository
Exporting cluster information...
```

<Warning>
  Modifying certain fields (like region or provider) may require cluster recreation. Hyve's reconciliation process will handle this automatically.
</Warning>

### hyve cluster delete

Delete a cluster by removing its YAML definition and reconciling.

```bash
hyve cluster delete [cluster-name] [flags]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster to delete
</ParamField>

<ParamField path="--force" type="boolean" default="false">
  Delete the cluster from the cloud provider immediately before removing the configuration file. Bypasses CI/CD — use this when you want to destroy the cluster right now rather than letting the pipeline handle it.
</ParamField>

<ParamField path="--force-cloud" type="boolean" default="false">
  Used together with `--force`: delete from cloud even if no configuration file exists.
</ParamField>

**Examples:**

```bash
# Default: remove YAML, commit, reconcile (CI/CD pipeline handles cloud deletion)
hyve cluster delete my-cluster

# Force: delete from cloud immediately, then remove configuration
hyve cluster delete my-cluster --force

# Force delete from cloud when no config file exists
hyve cluster delete orphaned-cluster --force --force-cloud
```

**Default process:**

1. Removes the cluster configuration YAML file
2. Commits and pushes the change to the state repository
3. Runs reconciliation

   - In **CI/CD mode** with `strictDelete: true`: the push triggers the pipeline, which deletes the cloud cluster
   - In **local mode** with `strictDelete: true`: the local reconcile deletes the cloud cluster immediately

**Force process (`--force`):**

1. Explicitly searches for and deletes the cluster from the cloud provider
2. Removes any ingress controllers
3. Removes the cluster configuration YAML file
4. Commits changes to Git
5. Runs reconciliation to clean up any remaining resources

<Info>
  The default (non-force) delete is the recommended approach in CI/CD workflows. Enable `strictDelete: true` in `hyve.yaml` so that removing a cluster YAML automatically triggers deletion in the pipeline. See the [CI/CD guide](/guides/cicd) for details.
</Info>

### hyve cluster force-delete

Force delete a cluster by name directly from the cloud provider without requiring a configuration file.

```bash
hyve cluster force-delete [cluster-name] [flags]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster to force delete
</ParamField>

<ParamField path="--region" type="string">
  Specific region to search. If not provided, searches common regions (PHX1, NYC1, FRA1, LON1).
</ParamField>

**Example:**

```bash
# Force delete with region hint
hyve cluster force-delete orphaned-cluster --region NYC1

# Search all common regions
hyve cluster force-delete lost-cluster
```

**Output:**

```

🔍 Searching for cluster 'orphaned-cluster' in region PHX1...
❌ Cluster 'orphaned-cluster' not found in region PHX1
🔍 Searching for cluster 'orphaned-cluster' in region NYC1...
✅ Found cluster 'orphaned-cluster' in region NYC1 with ID xyz789
🗑️ Force deleting cluster 'orphaned-cluster'...
🔌 Attempting to remove any ingress controllers...
✅ Successfully force deleted cluster 'orphaned-cluster' from region NYC1
```

<Warning>
  This command does not remove configuration files or run reconciliation. It only deletes the cloud resources. Useful when configuration files are lost or corrupted.
</Warning>

## Cluster Definition Format

Clusters are defined in YAML format and stored in the `clusters/` directory of your Git repository.

### Basic Structure

Cluster specs support two mutually exclusive ways to define nodes: `spec.nodes` (simple single-pool) and `spec.nodeGroups` (named multi-pool with autoscaling). When `nodeGroups` are present, they take precedence over `nodes`.

**Using `spec.nodes` (simple/Civo):**

```yaml
apiVersion: v1
kind: Cluster
metadata:
  name: my-cluster
  region: PHX1
spec:
  provider: civo
  nodes:
    - g4s.kube.small
    - g4s.kube.small
  clusterType: k3s
  ingress:
    enabled: true
    loadBalancer: true
```

**Using `spec.nodeGroups` (AWS/GCP/Azure with named pools and autoscaling):**

```yaml
apiVersion: v1
kind: Cluster
metadata:
  name: production
  region: us-east-1
spec:
  provider: aws
  nodeGroups:
    - name: system
      instanceType: t3.medium
      count: 2
    - name: workers
      instanceType: m5.xlarge
      count: 4
      minCount: 2
      maxCount: 10
      spot: true
  clusterType: k3s
  ingress:
    enabled: true
    loadBalancer: true
```

<Info>
  **Civo limitation**: Civo supports only a single node pool. If `nodeGroups` is used with a Civo cluster, only the first group is applied and a warning is logged. For Civo clusters, prefer `spec.nodes`.
</Info>

### GCP Cluster Structure

GCP clusters include additional fields for project configuration:

```yaml
apiVersion: v1
kind: Cluster
metadata:
  name: my-gke-cluster
  region: us-central1
spec:
  provider: gcp
  gcpProject: dev           # Project alias (configured via hyve config gcp add-project)
  gcpProjectId: my-project  # Resolved project ID (auto-populated)
  nodes:
    - e2-standard-4
  clusterType: k3s
  ingress:
    enabled: true
    loadBalancer: true
```

### AWS Cluster Structure

AWS EKS clusters include additional fields for VPC and IAM role configuration:

```yaml
apiVersion: v1
kind: Cluster
metadata:
  name: my-eks-cluster
  region: us-east-1
spec:
  provider: aws
  awsAccount: production           # Account alias (optional, configured via hyve config aws account-add)
  awsAccountId: "123456789012"     # Resolved account ID (auto-populated)
  awsVpcName: dev-vpc              # VPC alias (required, configured via hyve config aws vpc-add)
  awsVpcId: vpc-12345678           # Resolved VPC ID (auto-populated)
  awsEksRole: eks-cluster-role     # EKS role alias (required, configured via hyve config aws eks-role-add)
  awsEksRoleArn: arn:aws:iam::123456789012:role/EKSClusterRole  # Resolved role ARN (auto-populated)
  nodes:
    - m5.large
  clusterType: k3s
  ingress:
    enabled: true
    loadBalancer: true
```

<Note>
  When creating an AWS EKS cluster, Hyve automatically:
  - Discovers subnets from the specified VPC
  - Creates a security group for the cluster with appropriate rules
  - Uses the specified IAM role for the EKS control plane
</Note>

### Field Reference

<ParamField path="apiVersion" type="string" required>

</ParamField>

<ParamField path="kind" type="string" required>

</ParamField>

<ParamField path="metadata.name" type="string" required>

  Cloud provider region (e.g., PHX1, NYC1, FRA1, LON1 for Civo)
</ParamField>

<ParamField path="spec.provider" type="string" required>
  Cloud provider name. Must be one of: `civo`, `aws`, `gcp`, `azure`
</ParamField>

<ParamField path="spec.nodes" type="string[]">
  Array of node sizes for simple single-pool clusters. Each entry represents one node. Use `spec.nodeGroups` instead for named multi-pool configurations with autoscaling. One of `nodes` or `nodeGroups` should be specified.
</ParamField>

<ParamField path="spec.nodeGroups" type="NodeGroup[]">
  Array of named node group definitions. Preferred over `spec.nodes` for AWS (EKS), GCP (GKE), and Azure (AKS) clusters — providers that natively support multiple named node pools with independent scaling.

  **NodeGroup fields:**
  - `name` (string, required) — Node group name
  - `instanceType` (string, required) — Instance/VM size
  - `count` (int) — Desired node count
  - `minCount` (int) — Minimum count for autoscaling
  - `maxCount` (int) — Maximum count for autoscaling
  - `diskSize` (int) — OS disk size in GB
  - `labels` (map) — Kubernetes node labels
  - `taints` (array) — Kubernetes node taints (`key`, `value`, `effect`)
  - `mode` (string) — Azure only: `System` or `User`
  - `spot` (bool) — Use spot/preemptible instances
</ParamField>

<ParamField path="spec.clusterType" type="string" default="k3s">
  Kubernetes distribution type (k3s, talos)
</ParamField>

<ParamField path="spec.ingress.enabled" type="boolean" default="true">

</ParamField>

<ParamField path="spec.ingress.loadBalancer" type="boolean" default="true">

</ParamField>

<ParamField path="spec.gcpProject" type="string">

</ParamField>

<ParamField path="spec.gcpProjectId" type="string">
  GCP project ID (auto-populated from alias). Do not set manually.
</ParamField>

<ParamField path="spec.awsAccount" type="string">
  AWS account alias (for AWS provider only). Set via `--aws-account` flag.
</ParamField>

<ParamField path="spec.awsAccountId" type="string">
  AWS account ID (auto-populated from alias). Do not set manually.
</ParamField>

<ParamField path="spec.awsVpcName" type="string">
  AWS VPC alias (required for AWS provider). Set via `--vpc-name` flag.
</ParamField>

<ParamField path="spec.awsVpcId" type="string">
  AWS VPC ID (auto-populated from alias). Do not set manually.
</ParamField>

<ParamField path="spec.awsEksRole" type="string">
  AWS EKS IAM role alias (required for AWS provider). Set via `--eks-role-name` flag.
</ParamField>

    ```bash
    hyve config gcp add-project --name dev --id my-dev-project
    hyve config gcp add-project --name prod --id my-prod-project

    # Production cluster with named node group (preferred for GKE)
    hyve cluster add prod --provider gcp --gcp-project prod --region us-east1 \
      --node-group "name=workers,type=e2-standard-4,count=3,min=2,max=6"

    # Multi-pool cluster with system and high-memory node pools
    hyve cluster add memory --provider gcp --gcp-project prod --region europe-west1 \
      --node-group "name=system,type=e2-standard-2,count=2" \
      --node-group "name=workers,type=n1-highmem-8,count=2,min=1,max=6,spot=true"

    # Development cluster using --nodes (backward-compatible, --gcp-project is required)
    hyve cluster add dev --provider gcp --gcp-project dev --region us-central1 --nodes e2-medium
    ```
  </Accordion>

  <Accordion title="Azure Examples">
    ```bash
    # Production cluster with named node groups (preferred for AKS)
    hyve cluster add prod --provider azure --region westus2 \
      --node-group "name=system,type=Standard_DS2_v2,count=2,mode=System"
    ```

## Reconciliation

After creating or modifying cluster configurations, run `hyve reconcile` to synchronize the desired state (Git) with the actual state (cloud provider).

### Reconciliation Modes

The reconciliation mode is controlled by a `hyve.yaml` file at the root of your state repository:

```yaml
# hyve.yaml
reconcile:
  mode: local          # "local" (default) or "cicd"
  strictDelete: false  # set to true to delete unmanaged cloud clusters
```

| Mode | Behaviour |
|------|-----------|
| `local` | Hyve provisions clusters directly from the machine running the command. Default when `hyve.yaml` is absent. |
| `cicd` | Hyve validates and commits the desired state to Git, then exits. A CI/CD pipeline handles the actual provisioning by calling `hyve reconcile --path .`. |

#### Strict Delete

When `strictDelete: true`, Hyve deletes any cloud cluster that has no matching YAML definition in your state repository. By default (`false`), only clusters whose names begin with a managed prefix (`hyve-`, `civo-deploy-`) are eligible for deletion.

<Warning>
  Enable `strictDelete` only after confirming your state repository contains definitions for every cluster you want to keep.
</Warning>

### Running Reconciliation

```bash
# Local mode — provisions clusters from this machine
hyve reconcile

# CI/CD pipeline mode — use the checked-out repo path directly
# This bypasses the hyve.yaml cicd check and always reconciles locally
hyve reconcile --path .

# Provide a relative path to a specific checkout
hyve reconcile --path /path/to/repo
```

#### `--path` Flag

| Flag | Description |
|------|-------------|
| `--path / -p` | Path to an already-checked-out local repository. Skips cloning and the `hyve.yaml` mode check; always runs full local reconciliation. Designed for use inside CI/CD pipelines. |

### Reconciliation Process

1. **Load Definitions**: Read all cluster YAML files from the Git repository
2. **Validate**: Ensure all cluster definitions are valid
3. **Check Cloud State**: Query the cloud provider for existing clusters
4. **Determine Actions**: Compare desired vs actual state
5. **Execute Changes**: Create, update, or delete clusters as needed
6. **Configure Ingress**: Set up ingress controllers if enabled
7. **Commit & Push**: Write reconciliation results back to the state repository

### Actions

- **Create**: Cluster exists in Git but not in cloud
- **Update**: Cluster exists in both but configuration differs
- **Delete**: Cluster exists in cloud but not in Git
- **None**: States match, no action needed

<Note>
  See the [CI/CD Workflow Guide](/guides/cicd) for a complete example using GitHub Actions.
</Note>

## Common Use Cases

### Create a Production Cluster

```bash
# Create cluster with 3 large nodes (--provider is required)
hyve cluster add production \
  --provider civo \
  --region NYC1 \
  --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large \
  --cluster-type k3s

# Wait for cluster to be ready (check with kubectl)
eval $(hyve use production)
kubectl get nodes
```

### Scale a Cluster

```bash
# Add more nodes to existing cluster
hyve cluster modify my-cluster \
  --nodes g4s.kube.medium,g4s.kube.medium,g4s.kube.medium,g4s.kube.medium

# Reconciliation will update the cluster
```

### Multi-Region Deployment

```bash
# Create clusters in different regions (--provider is required)
hyve cluster add us-east --provider civo --region NYC1 --nodes g4s.kube.medium,g4s.kube.medium
hyve cluster add us-west --provider civo --region PHX1 --nodes g4s.kube.medium,g4s.kube.medium
hyve cluster add eu-central --provider civo --region FRA1 --nodes g4s.kube.medium,g4s.kube.medium

# All clusters managed in the same Git repository
```

### Cleanup Orphaned Clusters

```bash
# List all clusters
hyve cluster list

# Force delete cluster not in Git
hyve cluster force-delete orphaned-cluster --region PHX1

# Or delete with configuration
hyve cluster delete old-cluster
```

```
~/.hyve/repositories/<repository-name>/

    └── development.yaml
```

These files are version-controlled in your Git repository.

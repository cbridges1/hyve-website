The `hyve template` command group manages reusable cluster templates that combine cluster specifications with automated workflows. Templates enable you to quickly create standardized clusters with pre-configured deployments.

## Overview

Templates are declarative definitions that specify cluster configuration and associated workflows to run after creation. They enable consistent, repeatable cluster deployments across environments.

## Commands

### hyve template create

Create a new cluster template with cluster specifications and workflows.

```bash
hyve template create [template-name] [flags]
```

<ParamField path="template-name" type="string" required>
  Name of the template to create
</ParamField>

<ParamField path="--description" type="string">
  Template description
</ParamField>

<ParamField path="--provider" type="string" default="civo">
  Cloud provider (civo, aws, gcp, azure)
</ParamField>

<ParamField path="--region" type="string" default="PHX1">
  Cloud provider region
</ParamField>

<ParamField path="--nodes" type="string" default="g4s.kube.small">
  Node sizes (comma-separated). Each entry creates one node.
</ParamField>

<ParamField path="--cluster-type" type="string" default="k3s">
  Kubernetes cluster type (`k3s`, `talos`). **Civo only** — ignored for AWS, GCP, and Azure providers.
</ParamField>

<ParamField path="--ingress" type="boolean" default="true">
  Enable ingress controller
</ParamField>

<ParamField path="--load-balancer" type="boolean" default="true">
  Enable load balancer for ingress
</ParamField>

<ParamField path="--on-create, -c" type="string">
  Workflows to run after cluster creation (comma-separated)
</ParamField>

<ParamField path="--on-delete" type="string">
  Workflows to run before cluster destruction (comma-separated)
</ParamField>

<ParamField path="--schedule" type="string">
  A 5-field cron expression (e.g. `0 20 * * 5`) that sets an automatic expiry on every cluster created from this template. When the template is executed, the next occurrence of the schedule is evaluated and written to the cluster's `spec.expiresAt` field as an RFC 3339 timestamp. Leave blank for clusters that should not expire automatically.

  Standard cron field order: `<minute> <hour> <day-of-month> <month> <day-of-week>`

  | Field | Range | Special values |
  |-------|-------|----------------|
  | Minute | 0–59 | `*` = any |
  | Hour | 0–23 | `*` = any |
  | Day of month | 1–31 | `*` = any |
  | Month | 1–12 | `*` = any |
  | Day of week | 0–6 (Sun=0) | `*` = any; ranges (`1-5`) and lists (`1,3,5`) supported |

  Examples: `0 20 * * 5` (every Friday at 20:00), `0 0 * * *` (daily at midnight)
</ParamField>

**Example:**

```bash
# Create basic template with lifecycle workflows
hyve template create prod-template \
  --description "Production cluster template" \
  --region NYC1 \
  --nodes g4s.kube.large,g4s.kube.large,g4s.kube.large \
  --on-create setup-monitoring,deploy-app \
  --on-delete backup-data,cleanup

# Create a template with an automatic weekly Friday expiry
hyve template create dev-weekly \
  --region PHX1 \
  --nodes g4s.kube.small \
  --schedule "0 20 * * 5"

# Create minimal template
hyve template create dev-template \
  --region PHX1 \
  --nodes g4s.kube.small \
  --ingress=false
```

**Output:**

```
✅ Template 'prod-template' created successfully
Template path: /Users/username/.hyve/repositories/production/templates/prod-template.yaml

📋 Template Details:
  Provider: civo
  Region: NYC1
  Nodes: g4s.kube.large, g4s.kube.large, g4s.kube.large
  Cluster Type: k3s
  Ingress: true
  OnCreate Workflows: setup-monitoring, deploy-app
  OnDestroy Workflows: backup-data, cleanup

💡 Create a cluster from this template with:
  hyve cluster create <cluster-name> --template prod-template
```

### hyve cluster create

Create a cluster from a template and run associated workflows. This command lives under `hyve cluster` (not `hyve template`) — see the [Cluster CLI reference](/cli/cluster) for the full command.

```bash
hyve cluster create [cluster-name] --template [template-name]
```

<ParamField path="cluster-name" type="string" required>
  Name for the new cluster
</ParamField>

<ParamField path="--template" type="string" required>
  Name of the template to create the cluster from
</ParamField>

**Flags:**

| Flag | Short | Description |
|------|-------|-------------|
| `--template` | `-t` | Template to create the cluster from (required) |
| `--region` | `-r` | Override the template's default region |
| `--set` | | Override driver params (repeatable): `KEY=VALUE` |

**Example:**

```bash
hyve cluster create prod-cluster-01 --template prod-template

# With a region override and driver param overrides
hyve cluster create prod-cluster-01 --template prod-template \
  --region NYC1 \
  --set node_count=5
```

<Info>
  Cluster creation is idempotent. If the cluster already exists, it will be reused and workflows will run on the existing cluster.
</Info>

## Template Definition Format

Templates are defined in YAML format and stored in the `templates/` directory.

### Basic Structure

```yaml
apiVersion: v1
kind: Template
metadata:
  name: prod-template
  description: Production cluster template
spec:
  provider: civo
  region: NYC1
  nodes:
    - g4s.kube.large
    - g4s.kube.large
    - g4s.kube.large
  clusterType: k3s
  schedule: "0 20 * * 5"    # optional: auto-expire clusters at next Friday 20:00
  ingress:
    enabled: true
    loadBalancer: true
  workflows:
    onCreate:
      - setup-monitoring
      - deploy-app
    onDelete:
      - backup-data
```

### Field Reference

<ParamField path="apiVersion" type="string" required>
  API version of the template definition (currently `v1`)
</ParamField>

<ParamField path="kind" type="string" required>
  Resource kind (must be `Template`)
</ParamField>

<ParamField path="metadata.name" type="string" required>
  Unique name for the template
</ParamField>

<ParamField path="metadata.description" type="string">
  Human-readable description
</ParamField>

<ParamField path="spec.provider" type="string" required>
  Cloud provider (civo, aws, gcp, azure)
</ParamField>

<ParamField path="spec.region" type="string" required>
  Cloud provider region
</ParamField>

<ParamField path="spec.nodes" type="string[]">
  Array of node sizes for simple single-pool clusters (Civo). Each entry creates one node. Either `nodes` or `nodeGroups` must be specified.
</ParamField>

<ParamField path="spec.nodeGroups" type="object[]">
  Named node pools for AWS EKS, GCP GKE, and Azure AKS. Use instead of `nodes` when you need multiple pools, autoscaling, spot instances, or per-pool labels and taints. Each entry supports:

  | Field | Type | Description |
  |-------|------|-------------|
  | `name` | string | Pool name (required) |
  | `instanceType` | string | Node instance/machine type (required) |
  | `count` | integer | Desired node count |
  | `minCount` | integer | Minimum nodes for autoscaling |
  | `maxCount` | integer | Maximum nodes for autoscaling |
  | `diskSize` | integer | Root disk size in GB |
  | `spot` | boolean | Use spot/preemptible instances |
  | `mode` | string | Azure only: `System` or `User` pool |
  | `labels` | map | Key-value node labels |
  | `taints` | object[] | Node taints (`key`, `value`, `effect`) |
</ParamField>

<ParamField path="spec.clusterType" type="string" default="k3s">
  Kubernetes distribution type (`k3s`, `talos`). **Civo only** — this field is ignored for AWS (EKS), GCP (GKE), and Azure (AKS) providers.
</ParamField>

<ParamField path="spec.ingress.enabled" type="boolean" default="true">
  Enable ingress controller installation
</ParamField>

<ParamField path="spec.ingress.loadBalancer" type="boolean" default="true">
  Enable load balancer for ingress
</ParamField>

<ParamField path="spec.workflows.beforeCreate" type="string[]">
  Names of workflows to run before the cluster is provisioned. **No kubeconfig is injected** — the cluster does not exist yet. Set via `--before-create` flag or the interactive TUI.
</ParamField>

<ParamField path="spec.workflows.onCreate" type="string[]">
  Names of workflows to execute after cluster creation. Kubeconfig is injected automatically. Set via `--on-create` flag or the interactive TUI.
</ParamField>

<ParamField path="spec.workflows.onDelete" type="string[]">
  Names of workflows to execute before cluster destruction. Kubeconfig is injected automatically. Set via `--on-delete` flag or the interactive TUI.
</ParamField>

<ParamField path="spec.workflows.afterDelete" type="string[]">
  Names of workflows to run after the cluster is deleted. **No kubeconfig is injected** — the cluster no longer exists. Set via `--after-delete` flag or the interactive TUI.
</ParamField>

<ParamField path="spec.schedule" type="string">
  A 5-field cron expression (e.g. `0 20 * * 5`). When the template is executed, the next occurrence after the current time is computed and written to the generated cluster's `spec.expiresAt` as an RFC 3339 timestamp. The cluster is then automatically deleted when that deadline passes. Omit this field for clusters that should run indefinitely.
</ParamField>

<ParamField path="spec.dynamicFields" type="string[]">
  A list of optional provider-specific field names whose values will be supplied at runtime by a `beforeCreate` workflow via `HYVE_*` output variables. When a field is listed here, the interactive TUI skips the prompt for it and the cluster definition is written with that field empty so the workflow can populate it.

  Supported field names:

  | Field name | Provider | Populated by |
  |---|---|---|
  | `awsVpcId` | AWS | `HYVE_VPC_ID` |
  | `awsEksRoleName` | AWS | `HYVE_EKS_ROLE_NAME` |
  | `awsNodeRoleName` | AWS | `HYVE_NODE_ROLE_NAME` |
  | `awsClusterSgId` | AWS | `HYVE_CLUSTER_SG_ID` |
  | `awsWorkerSgId` | AWS | `HYVE_WORKER_SG_ID` |
  | `azureResourceGroup` | Azure | `HYVE_RESOURCE_GROUP_NAME` |

  Example — VPC and IAM roles created by a `beforeCreate` workflow:

  ```yaml
  spec:
    provider: aws
    awsAccount: my-account
    dynamicFields:
      - awsVpcId
      - awsEksRoleName
      - awsNodeRoleName
    workflows:
      beforeCreate:
        - provision-eks-prereqs   # outputs HYVE_VPC_ID, HYVE_EKS_ROLE_ARN, HYVE_NODE_ROLE_ARN
      onCreate:
        - deploy-app
      afterDelete:
        - teardown-eks-prereqs
  ```
</ParamField>

### AWS-Specific Fields

These fields apply when `spec.provider` is `aws`.

<ParamField path="spec.awsAccount" type="string">
  AWS account alias name as registered via `hyve config aws account add`.
</ParamField>

<ParamField path="spec.awsVpcId" type="string">
  AWS VPC ID. Set directly in the template YAML or override at execute time with `--vpc-id/-v`. The interactive TUI offers a live cloud lookup to select from available VPCs. Omit this field (or list it in `dynamicFields`) if a `beforeCreate` workflow creates the VPC and outputs `HYVE_VPC_ID`.
</ParamField>

<ParamField path="spec.awsEksRoleName" type="string">
  IAM role name for the EKS control plane. Set in the template YAML or override with `--eks-role-name/-e`. Omit (or list in `dynamicFields`) if a `beforeCreate` workflow creates the role and outputs `HYVE_EKS_ROLE_NAME` / `HYVE_EKS_ROLE_ARN`.
</ParamField>

<ParamField path="spec.awsNodeRoleName" type="string">
  IAM role name for EKS node groups. Set in the template YAML or override with `--node-role-name/-n`. Omit (or list in `dynamicFields`) if a `beforeCreate` workflow outputs `HYVE_NODE_ROLE_NAME` / `HYVE_NODE_ROLE_ARN`.
</ParamField>

<ParamField path="spec.awsProfile" type="string">
  Named AWS CLI profile to use for authentication (e.g. `"prod-admin"`). When set, Hyve uses `~/.aws/config` profile credentials instead of static key/secret.
</ParamField>

<ParamField path="spec.awsKmsKeyAlias" type="string">
  AWS KMS key alias for encrypting Kubernetes secrets at rest (e.g. `"alias/my-eks-key"`). When set, EKS envelope encryption is enabled on cluster creation.
</ParamField>

<ParamField path="spec.awsClusterSgId" type="string">
  Pre-existing EC2 security group ID to use as the EKS cluster security group. Can be set by a `beforeCreate` workflow via the `HYVE_CLUSTER_SG_ID` output variable. Omit to have EKS create one automatically.
</ParamField>

<ParamField path="spec.awsWorkerSgId" type="string">
  Pre-existing EC2 security group ID to attach to worker nodes via an EC2 launch template. Can be set by a `beforeCreate` workflow via the `HYVE_WORKER_SG_ID` output variable.
</ParamField>

<ParamField path="spec.kubernetesVersion" type="string">
  Kubernetes version to pin for clusters created from this template (e.g. `"1.30"`). Supported by AWS (EKS), GCP (GKE), and Azure (AKS). Omit to use the cloud provider's current default.
</ParamField>

<Info>
  AWS VPC and role fields can be set in the template YAML, overridden at execute time via flags, or left to a `beforeCreate` workflow by listing them in `spec.dynamicFields`. When listed in `dynamicFields`, the TUI skips the prompt and the reconciler reads the values from workflow output variables instead.
</Info>

<Info>
  **EKS instance type:** avoid `nano` and `micro` instance types (e.g. `t3.micro`, `t2.micro`) for EKS node groups. These instances support only 4 pods maximum (2 ENIs × 2 IPs), which is not enough to run EKS system daemonsets (`aws-node`, `kube-proxy`) alongside workloads. Use **`t3.small`** (11 pods) at minimum, or **`t3.medium`** (17 pods) as the recommended baseline.
</Info>

## Common Use Cases

### Production Template with Monitoring

```yaml
apiVersion: v1
kind: Template
metadata:
  name: production-ha
  description: High-availability production cluster with full monitoring stack
spec:
  provider: civo
  region: NYC1
  nodes:
    - g4s.kube.xlarge
    - g4s.kube.xlarge
    - g4s.kube.xlarge
  clusterType: k3s
  ingress:
    enabled: true
    loadBalancer: true
  workflows:
    onCreate:
      - setup-monitoring
      - install-cert-manager
      - configure-backups
      - deploy-app
    onDelete:
      - final-backup
      - notify-team
```

**Execute:**

```bash
hyve cluster create prod-us-east-01 --template production-ha
```

## Template Best Practices

### 1. Naming Conventions

Use descriptive names that indicate purpose and scale:

```bash
# Good examples
prod-ha-monitoring      # Production, high-availability with monitoring
staging-standard        # Standard staging setup
dev-minimal            # Minimal development cluster
testing-ephemeral      # Temporary testing cluster
```

### 2. Workflow Organization

Order workflows from infrastructure to application:

```yaml
workflows:
  onCreate:
    - setup-networking      # Network policies, CNI config
    - install-cert-manager  # Certificate management
    - setup-monitoring      # Prometheus, Grafana
    - configure-storage     # PV, PVC, storage classes
    - deploy-app           # Application deployment
  onDelete:
    - backup-data          # Final backup before deletion
    - cleanup-dns          # Remove DNS records
```

### 3. Resource Sizing

Match node sizes to environment purpose:

```yaml
# Development
nodes:
  - g4s.kube.small

# Staging
nodes:
  - g4s.kube.medium
  - g4s.kube.medium

# Production
nodes:
  - g4s.kube.large
  - g4s.kube.large
  - g4s.kube.large
```

### 4. Template Reuse

Create base templates and extend them:

```bash
# Base template
hyve template create base-cluster \
  --region PHX1 \
  --nodes g4s.kube.medium \
  --on-create setup-base

# Extended for specific use case
# Copy and modify base-cluster.yaml to add specific workflows
```

### AWS EKS Cluster (static VPC and roles)

For clusters where the VPC and IAM roles already exist:

```yaml
apiVersion: v1
kind: Template
metadata:
  name: aws-eks-standard
  description: Standard AWS EKS cluster with named node groups
spec:
  provider: aws
  region: us-east-1
  nodeGroups:
    - name: system
      instanceType: t3.medium
      count: 2
      minCount: 1
      maxCount: 4
    - name: workers
      instanceType: t3.large
      count: 3
      minCount: 1
      maxCount: 10
      spot: true
      labels:
        role: worker
  clusterType: eks
  awsAccount: main-account      # alias from hyve config aws account add
  awsVpcId: vpc-0abc123456789   # VPC ID (or override at execute time with --vpc-id)
  awsEksRoleName: k8srole       # EKS cluster role name
  awsNodeRoleName: k8snoderole  # EKS node role name
  ingress:
    enabled: false
    loadBalancer: false
  workflows:
    onCreate:
      - deploy-app
```

**Execute:**

```bash
hyve cluster create eks-prod-01 --template aws-eks-standard

# Override VPC and roles at execution time:
hyve cluster create eks-prod-01 --template aws-eks-standard \
  --vpc-id vpc-0def987654321 \
  --eks-role-name prod-eks-role
```

### AWS EKS Cluster (workflow-provisioned VPC and roles)

For fully automated clusters where a `beforeCreate` workflow creates the VPC and IAM roles. Listing those fields in `dynamicFields` suppresses the interactive prompt — the reconciler reads the values from the workflow's `HYVE_*` output variables instead:

```yaml
apiVersion: v1
kind: Template
metadata:
  name: aws-eks-automated
  description: EKS cluster with VPC and IAM roles created by a beforeCreate workflow
spec:
  provider: aws
  region: us-east-1
  nodeGroups:
    - name: system
      instanceType: t3.medium
      count: 2
      minCount: 1
      maxCount: 4
  clusterType: eks
  awsAccount: main-account
  dynamicFields:
    - awsVpcId          # populated from HYVE_VPC_ID
    - awsEksRoleName    # populated from HYVE_EKS_ROLE_NAME / HYVE_EKS_ROLE_ARN
    - awsNodeRoleName   # populated from HYVE_NODE_ROLE_NAME / HYVE_NODE_ROLE_ARN
  ingress:
    enabled: false
    loadBalancer: false
  workflows:
    beforeCreate:
      - provision-eks-prereqs   # creates VPC and IAM roles, outputs HYVE_* vars
    onCreate:
      - deploy-app
    afterDelete:
      - teardown-eks-prereqs    # removes VPC and IAM roles after cluster is deleted
```

**Execute:**

```bash
hyve cluster create eks-prod-01 --template aws-eks-automated
# No --vpc-id or role flags needed — the beforeCreate workflow provisions everything
```

```yaml
  name: azure-aks-standard
  description: Azure AKS cluster with System and User agent pool profiles
spec:
  provider: azure
  region: eastus
  nodeGroups:
    - name: systempool
      instanceType: Standard_D2s_v3
      count: 2
      minCount: 1
      maxCount: 3
      mode: System
    - name: workerpool
      instanceType: Standard_D4s_v3
      count: 3
      minCount: 1
      maxCount: 10
      mode: User
      labels:
        role: worker
  clusterType: aks
  ingress:
    enabled: true
    loadBalancer: true
  workflows:
    onCreate:
      - setup-monitoring
      - deploy-app
```

**Execute:**

```bash
hyve cluster create aks-prod-01 --template azure-aks-standard
```

## Storage

Template definitions are stored in:

```
~/.hyve/repositories/<repository-name>/
└── templates/
    ├── prod-template.yaml
    ├── staging-template.yaml
    └── dev-template.yaml
```

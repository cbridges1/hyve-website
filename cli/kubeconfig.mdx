The `hyve kubeconfig` command group manages kubeconfigs for clusters in the current repository. Kubeconfigs are encrypted and stored securely in a local database, enabling easy access across different machines.

## Overview

Hyve automatically syncs and stores kubeconfigs from active clusters, encrypting them with AES-GCM for security. You can retrieve kubeconfigs for use with kubectl and other Kubernetes tools.

## Commands

### hyve kubeconfig sync

Sync kubeconfigs from all active clusters in the current repository.

```bash
hyve kubeconfig sync
```

**Example:**

```bash
hyve kubeconfig sync
```

**Output:**

```
📁 Syncing kubeconfigs for repository 'production'
🔍 Found 3 cluster definitions
📥 Syncing kubeconfig for cluster 'prod-cluster-01'... ✅
📥 Syncing kubeconfig for cluster 'staging-cluster'... ✅
📥 Syncing kubeconfig for cluster 'dev-cluster'... ✅
✅ Kubeconfig sync completed
```

**Process:**

1. Reads all cluster definitions from Git repository
2. Queries each cloud provider for cluster status
3. Normalizes provider-specific status values to a canonical `ACTIVE` state — Azure AKS reports `Succeeded`, GKE reports `RUNNING`; both are treated as active
4. Retrieves the kubeconfig from each active cluster
5. Encrypts and stores kubeconfigs in the local database

<Info>
  Run this command after creating new clusters or when switching machines to ensure you have the latest kubeconfigs.
</Info>

### hyve kubeconfig get

Retrieve and display the kubeconfig for a specific cluster.

```bash
hyve kubeconfig get [cluster-name] [flags]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster
</ParamField>

<ParamField path="--save" type="boolean" default="false">
  Save kubeconfig to ~/.kube/config-[cluster-name]
</ParamField>

<ParamField path="--merge" type="boolean" default="false">
  Merge kubeconfig into ~/.kube/config
</ParamField>

<ParamField path="--output" type="string">
  Output file path for kubeconfig
</ParamField>

**Examples:**

```bash
# Display kubeconfig
hyve kubeconfig get prod-cluster-01

# Save to standard location
hyve kubeconfig get prod-cluster-01 --save

# Save to custom location
hyve kubeconfig get prod-cluster-01 --output /path/to/kubeconfig

# Merge into main kubeconfig
hyve kubeconfig get prod-cluster-01 --merge
```

**Output (display):**

```yaml
apiVersion: v1
kind: Config
clusters:
- cluster:
    certificate-authority-data: LS0tLS1CRUdJTi...
    server: https://api.prod-cluster-01.example.com:6443
  name: prod-cluster-01
contexts:
- context:
    cluster: prod-cluster-01
    user: prod-cluster-01-admin
  name: prod-cluster-01
current-context: prod-cluster-01
users:
- name: prod-cluster-01-admin
  user:
    client-certificate-data: LS0tLS1CRUdJTi...
    client-key-data: LS0tLS1CRUdJTi...
```

**Output (save):**

```
✅ Kubeconfig saved to /Users/username/.kube/config-prod-cluster-01
💡 To use: export KUBECONFIG=/Users/username/.kube/config-prod-cluster-01
```

<Info>
  If the kubeconfig is not found, run `hyve kubeconfig sync` first to retrieve it from the cloud provider.
</Info>

### hyve kubeconfig use

Merge cluster context into `~/.kube/config` and set it as the active kubectl context.

```bash
hyve kubeconfig use [cluster-name]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster
</ParamField>

**Example:**

```bash
hyve kubeconfig use prod-cluster-01
```

**Output:**

```
🔀 Merging cluster 'prod-cluster-01' into /Users/username/.kube/config
📦 Backup created at /Users/username/.kube/config.backup
✅ Merged cluster 'prod-cluster-01' into /Users/username/.kube/config
Switched to context "prod-cluster-01".
✅ Active context set to 'prod-cluster-01'

💡 Test your connection:
   kubectl get nodes
```

**Process:**

1. Reads cluster kubeconfig from Hyve storage
2. Backs up existing `~/.kube/config`
3. Merges cluster context into `~/.kube/config`
4. Runs `kubectl config use-context <cluster>` to set the active context

<Info>
  Unlike `hyve kubeconfig merge`, this command is designed for interactive use — it merges and immediately switches context in one step.
</Info>

---

### hyve kubeconfig merge

Merge cluster context into local `~/.kube/config` without changing the active context.

```bash
hyve kubeconfig merge [cluster-name]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster to merge
</ParamField>

**Example:**

```bash
hyve kubeconfig merge prod-cluster-01
```

**Output:**

```
🔀 Merging cluster 'prod-cluster-01' into /Users/username/.kube/config
📦 Backup created at /Users/username/.kube/config.backup
✅ Successfully merged cluster 'prod-cluster-01' into /Users/username/.kube/config

💡 Next steps:
   kubectl config use-context prod-cluster-01
   kubectl get nodes
```

**Process:**

1. Backs up existing `~/.kube/config`
2. Merges cluster configuration into main kubeconfig
3. Preserves existing contexts and clusters

<Info>
  Use `hyve kubeconfig merge` to add a cluster to `~/.kube/config` without switching to it. To merge and switch in one step, use `hyve kubeconfig use` instead.
</Info>

---

### hyve kubeconfig remove

Remove cluster context from local ~/.kube/config.

```bash
hyve kubeconfig remove [cluster-name]
```

<ParamField path="cluster-name" type="string" required>
  Name of the cluster to remove
</ParamField>

**Example:**

```bash
hyve kubeconfig remove prod-cluster-01
```

**Output:**

```
📦 Backup created at /Users/username/.kube/config.backup
🗑️  Removing cluster 'prod-cluster-01' from /Users/username/.kube/config
✅ Successfully removed cluster 'prod-cluster-01' from /Users/username/.kube/config

💡 View remaining contexts:
   kubectl config get-contexts
```

<Warning>
  This removes the cluster context from your main kubeconfig but does not delete it from Hyve's database.
</Warning>

---

### hyve kubeconfig migrate

Migrate kubeconfig encryption from hostname-based keys to portable keys.

```bash
hyve kubeconfig migrate [old-hostname]
```

<ParamField path="old-hostname" type="string" required>
  Hostname that was used when kubeconfigs were originally encrypted
</ParamField>

**Example:**

```bash
hyve kubeconfig migrate "old-macbook.local"
```

**Output:**

```
🔄 Starting migration for repository: production
🔑 Old hostname: old-macbook.local

✅ Migration completed successfully!

📝 All kubeconfigs have been re-encrypted with the new portable key format.
💡 Your kubeconfigs will now work across different machines without hostname dependencies.
```

<Info>
  Use this command when moving Hyve database to a new machine or after hostname changes. This ensures kubeconfigs remain accessible.
</Info>

## Kubeconfig Storage

Kubeconfigs are stored encrypted in a SQLite database:

```
~/.hyve/
└── kubeconfigs.db      # Encrypted kubeconfigs (AES-GCM)
```

### Security Features

- **AES-GCM Encryption**: All kubeconfigs encrypted at rest
- **Per-Repository Isolation**: Kubeconfigs scoped to repository
- **Portable Keys**: Encryption keys not tied to hostname (after migration)

## Common Use Cases

### Initial Setup

```bash
# After creating a cluster
hyve cluster add my-cluster --provider civo --region PHX1

# Wait for cluster to be ready, then sync kubeconfigs
hyve kubeconfig sync

# Use the cluster (merges into ~/.kube/config and sets active context)
hyve use my-cluster
kubectl get nodes
```

---

### Switching Between Clusters

```bash
# Switch to production
hyve use production
kubectl get pods -A

# Switch to staging
hyve use staging
kubectl get pods -A

# Back to production
hyve use production
```

---

### Working Across Multiple Machines

**On Machine 1:**
```bash
# Create and sync cluster
hyve cluster add shared-cluster --provider civo --region NYC1
hyve kubeconfig sync

# Push Git repo (kubeconfigs stored in database)
hyve git push "Add shared cluster"
```

**On Machine 2:**
```bash
# Pull Git repo
hyve git pull

# Sync kubeconfigs
hyve kubeconfig sync

# Use cluster
hyve use shared-cluster
kubectl get nodes
```

<Info>
  The kubeconfigs database is stored locally but cluster definitions are in Git, making it easy to share cluster configs across teams.
</Info>

---

### Merge Multiple Clusters

```bash
# Merge several clusters into main kubeconfig
hyve kubeconfig merge prod-us-east
hyve kubeconfig merge prod-us-west
hyve kubeconfig merge prod-eu-central

# View all contexts
kubectl config get-contexts

# Switch between them
kubectl config use-context prod-us-east
kubectl config use-context prod-eu-central
```

---

### Export for CI/CD

```bash
# Export kubeconfig to file for CI/CD pipeline
hyve kubeconfig get prod-cluster-01 --output ./ci-kubeconfig.yaml

# Use in CI/CD
export KUBECONFIG=./ci-kubeconfig.yaml
kubectl apply -f manifests/
```

---

### Backup and Restore

```bash
# Backup kubeconfigs database
cp ~/.hyve/kubeconfigs.db ~/backups/kubeconfigs-$(date +%Y%m%d).db

# Restore on new machine
cp ~/backups/kubeconfigs-20240115.db ~/.hyve/kubeconfigs.db

# Migrate encryption if hostname changed
hyve kubeconfig migrate "old-hostname.local"
```

## Integration with kubectl

### Switching Clusters

```bash
# Merge and set active context (recommended)
hyve use my-cluster
kubectl get nodes

# View all contexts
kubectl config get-contexts

# Switch between already-merged clusters
kubectl config use-context other-cluster
```

### Parallel Sessions on Different Clusters

Since `hyve use` modifies the shared `~/.kube/config`, switching in one terminal affects all terminals. For true per-terminal isolation, use a saved kubeconfig with `KUBECONFIG`:

```bash
# Terminal 1 - Production (isolated)
hyve kubeconfig get production --save
export KUBECONFIG=~/.kube/config-production
kubectl get pods -n production

# Terminal 2 - Staging (isolated)
hyve kubeconfig get staging --save
export KUBECONFIG=~/.kube/config-staging
kubectl get pods -n staging

# Each terminal maintains its own context
```

## Troubleshooting

### Kubeconfig Not Found

```bash
# Error: Kubeconfig not found for cluster my-cluster
# Solution: Sync kubeconfigs
hyve kubeconfig sync
```

### Connection Refused

```bash
# Error: Unable to connect to the server
# Check if cluster is running
hyve cluster list

# Re-sync kubeconfig
hyve kubeconfig sync
```

### Encryption Migration Issues

```bash
# If you moved database to new machine
# Find old hostname
hostname  # On old machine

# Migrate on new machine
hyve kubeconfig migrate "old-machine.local"
```

### Wrong Context After Switching

```bash
# Check current context
kubectl config current-context

# Switch to correct cluster
hyve use correct-cluster
kubectl get nodes
```

## Best Practices

### 1. Use `hyve use` for Quick Switching

```bash
# Switch cluster and set context in one command
hyve use production
hyve use staging
hyve use development
```

### 2. Regular Sync

```bash
# Sync kubeconfigs regularly
hyve kubeconfig sync

# Especially after:
# - Creating new clusters
# - Cluster upgrades
# - Certificate rotation
```

### 3. Backup Kubeconfigs

```bash
# Periodic backups
cp ~/.hyve/kubeconfigs.db ~/backups/kubeconfigs-$(date +%Y%m%d).db

# Or include in Git (encrypted)
```

### 4. Per-Environment Workflows

```bash
# Switch to production
hyve use production
kubectl get pods

# Switch to development
hyve use dev-cluster
kubectl get pods
```

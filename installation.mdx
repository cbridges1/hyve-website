<Note>
  **Cloud Provider Account:** You'll need at least one cloud provider account:
  - [AWS](https://aws.amazon.com/) with EKS permissions
  - [Google Cloud](https://console.cloud.google.com/) with GKE permissions
  - [Azure](https://portal.azure.com/) with AKS permissions
  - [Civo](https://dashboard.civo.com/)
</Note>

<Note>
  **Git Requirement:** Hyve uses your system's git command by default for all repository operations. This provides better compatibility and easier onboarding since you don't need to configure additional libraries.

  If you prefer, you can optionally switch to the built-in go-git library using `hyve config set-git-backend builtin`.
</Note>

## Install

<Steps>
  <Step title="Install with Go">
    ```bash
    go install github.com/your-org/hyve@latest
    ```

    This installs the `hyve` binary to your `$GOPATH/bin` directory (usually `~/go/bin`).
  </Step>

  <Step title="Ensure Go bin is in PATH">
    ```bash
    export PATH=$PATH:$(go env GOPATH)/bin
    ```

    Add this line to your shell profile (`~/.bashrc`, `~/.zshrc`, etc.) to make it permanent.
  </Step>

  <Step title="Verify installation">
    ```bash
    hyve --help
    ```

    You should see the Hyve help output.
  </Step>
</Steps>

## Git Backend Configuration

Hyve supports two Git backends:

<Tabs>
  <Tab title="System Git (Default)">
    Uses your system's installed git command. This is the default and recommended option.

    **Advantages:**
    - No additional setup required
    - Works with your existing git configuration
    - Better compatibility with various authentication methods
    - Familiar behavior for git users

    **Requirements:**
    - Git must be installed and available in your PATH
    - Verify with: `git --version`
  </Tab>

  <Tab title="Built-in Git">
    Uses the go-git library embedded in Hyve.

    **Advantages:**
    - Works without system git installed
    - Portable across different environments

    **Enable:**
    ```bash
    # Set to built-in git (persisted in config)
    hyve config set-git-backend builtin

    # Verify the setting
    hyve config get-git-backend
    ```

    The preference is stored in `~/.hyve/config.yaml` and persists across sessions.

    **Temporary Override:**
    You can temporarily override the config with an environment variable:
    ```bash
    export GIT_BACKEND=builtin
    ```
  </Tab>
</Tabs>

## Configuration

### API Token Setup

Configure credentials for your preferred cloud provider:

<Tabs>
  <Tab title="AWS">
    ```bash
    hyve config set-token aws-access-key --token YOUR_ACCESS_KEY_ID
    hyve config set-token aws-secret-key --token YOUR_SECRET_ACCESS_KEY
    ```

    Or via environment:
    ```bash
    export AWS_ACCESS_KEY_ID=your_access_key
    export AWS_SECRET_ACCESS_KEY=your_secret_key
    export AWS_REGION=us-east-1
    ```
  </Tab>

  <Tab title="GCP">
    ```bash
    hyve config set-token gcp-project --token YOUR_PROJECT_ID
    # Uses Application Default Credentials
    gcloud auth application-default login
    ```

    Or via environment:
    ```bash
    export GCP_PROJECT_ID=your_project_id
    export GOOGLE_APPLICATION_CREDENTIALS=/path/to/credentials.json
    ```
  </Tab>

  <Tab title="Azure">
    ```bash
    hyve config set-token azure-subscription --token YOUR_SUBSCRIPTION_ID
    hyve config set-token azure-resource-group --token YOUR_RESOURCE_GROUP
    # Uses Azure CLI credentials
    az login
    ```

    Or via environment:
    ```bash
    export AZURE_SUBSCRIPTION_ID=your_subscription_id
    export AZURE_RESOURCE_GROUP=your_resource_group
    ```
  </Tab>

  <Tab title="Civo">
    ```bash
    hyve config set-token civo
    # Enter token when prompted
    ```

    Or via environment:
    ```bash
    export CIVO_TOKEN=your_civo_api_token
    ```
  </Tab>
</Tabs>

**Benefits of database storage:**
- AES-GCM encrypted storage
- Portable across machines
- No environment variables needed

### Token Priority

Hyve checks for credentials in this order:

1. **Database** (encrypted storage via `hyve config set-token`)
2. **Environment variables**
3. **`.env` file**

### Git Credentials (Optional)

For private Git repositories:

```bash
# Store globally encrypted credentials
hyve git credentials --username your-username --password your_token

# Or use environment variable as fallback
export HYVE_GIT_TOKEN=your_personal_access_token
```

## Directory Structure

After installation, Hyve creates:

```
~/.hyve/
├── hyve.db             # Unified encrypted database (credentials, repos, kubeconfigs)
├── context.yaml        # Active provider account context
├── temp/               # Temporary kubeconfig files
└── repositories/       # Cloned repository storage
    ├── production/
    │   ├── clusters/
    │   ├── workflows/
    │   └── templates/
    └── development/
```

<Note>
  All credentials and kubeconfigs are encrypted using AES-GCM
</Note>

### Overriding the data directory

The default `~/.hyve` location can be overridden with the `--home` flag or the `HYVE_HOME` environment variable. This is useful for testing multiple isolated instances:

```bash
# Use a custom directory for a single command
hyve --home /tmp/hyve-staging reconcile

# Set it persistently for a shell session
export HYVE_HOME=~/hyve-staging
hyve git add staging --repo-url https://github.com/company/staging-state.git

# Or point to an existing directory
HYVE_HOME=/opt/hyve hyve cluster list
```

The flag and environment variable are equivalent; the `--home` flag takes precedence over `HYVE_HOME`.

## Platform-Specific Notes

<AccordionGroup>
  <Accordion title="macOS">
    On macOS, you may need to allow the binary in Security & Privacy settings:

    1. Run `hyve` once
    2. Go to System Preferences → Security & Privacy
    3. Click "Allow Anyway" for the hyve binary
  </Accordion>

  <Accordion title="Linux">
    Ensure you have the necessary dependencies:

    ```bash
    # Debian/Ubuntu
    sudo apt-get install git

    # RHEL/CentOS
    sudo yum install git
    ```
  </Accordion>

  <Accordion title="Windows">
    Variable substitution in workflows uses:
    - Windows: `cmd /C "command"`
    - Use `%VAR%` for environment variables alongside `${VAR}` for workflow variables
  </Accordion>
</AccordionGroup>

## Verifying Installation

Run these commands to verify everything is set up correctly:

```bash
# Check Hyve installation
hyve --help

# Verify system git is available (required for default backend)
git --version

# Verify API token
hyve config get-token civo

# List stored tokens
hyve config list-tokens

# Check Git credentials (if configured)
hyve git credentials
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="Command not found">
    Ensure the `hyve` binary is in your PATH:

    ```bash
    # Check if hyve is in PATH
    which hyve

    # If not, add directory to PATH
    export PATH=$PATH:/path/to/hyve/directory
    ```
  </Accordion>

  <Accordion title="Permission denied">
    Make the binary executable:

    ```bash
    chmod +x hyve
    ```
  </Accordion>

  <Accordion title="Database connection error">
    Ensure the `.hyve` directory has proper permissions:

    ```bash
    chmod 755 ~/.hyve
    chmod 644 ~/.hyve/*.db
    ```
  </Accordion>
</AccordionGroup>
